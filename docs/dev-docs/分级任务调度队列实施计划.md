# 分级任务调度队列实施计划

> 目标：在已有的 scanner worker + media worker 双 utilityProcess 架构之上，引入主进程侧的分级任务调度队列，实现任务优先级管控——保证前台用户操作流畅，后台任务在空闲时执行。

## 1. 任务优先级划分

将所有耗时任务按「触发来源」+「用户体验影响」分两级。判定原则：**用户在前台等待结果的任务为高优先级；用户没有直接等待的任务为低优先级**。

### 1.1 高优先级（HIGH）—— 直接执行，可抢占低优先级

| 任务 | 触发来源 | 入口位置 | 说明 |
|---|---|---|---|

| 完整 sha256 重复检测 | 用户点击「查找重复」 | `ipc/handlers/media.ts: media:findDuplicates` | 主进程 CPU 密集 |

## 2. 调度逻辑

### 2.1 核心数据结构

```typescript
// src/main/scheduler/task-scheduler.ts

type TaskPriority = 'high' | 'low'

interface ScheduledTask {
  id: string
  priority: TaskPriority
  run: () => Promise<void>
  // 用于低优先级任务的取消（已入队但未开始执行）
  cancelToken?: { cancelled: boolean }
}

class TaskScheduler extends EventEmitter {
  // 低优先级等待队列（FIFO）
  private lowPriorityQueue: ScheduledTask[] = []
  // 当前正在执行的低优先级任务（最多 maxLowConcurrency 个）
  private runningLow: Set<ScheduledTask> = new Set()
  // 当前正在执行的高优先级任务计数
  private highPriorityRunning = 0
  // 全局暂停标志（cleanup 时置 true，阻止新任务执行）
  private isPaused = false

  // 后台任务最大并发数（保守值，避免与高优先级争抢 CPU/IO）
  private readonly maxLowConcurrency = 1
}
```

### 2.2 入队与执行

```
用户主动触发任务（高优先级）:
  scheduler.runHighPriority(async () => {
    await ctx.scannerManager.startScan(options)
  })
  → highPriorityRunning++
  → 若有低优先级任务正在执行，调用其 worker 的 stop 接口（postMessage STOP）
  → 等待低优先级任务发送 COMPLETE（最多等待 2s，超时强制继续）
  → 执行任务本体
  → highPriorityRunning--
  → 若 highPriorityRunning === 0，调度器尝试 pump 队列

后台自动触发任务（低优先级）:
  scheduler.enqueueLow(async () => {
    await ctx.mediaWorkerManager.startPhashBatch()
  })
  → 加入 lowPriorityQueue
  → 若 highPriorityRunning === 0 && runningLow.size < maxLowConcurrency && !isPaused，立即 pump
```

### 2.3 高优先级抢占低优先级的流程

1. 高优先级任务调用 `scheduler.runHighPriority(fn)` 入口
2. 调度器 `highPriorityRunning++`
3. 检查 `runningLow`：
   - 若为空，直接执行高优先级任务
   - 若非空，对每个 running low task 调用其 `cancel()` 回调（由任务注册时提供，例如 `() => ctx.mediaWorkerBridge.stopPhashBatch()`）
4. 等待 `runningLow` 清空（监听低优先级任务的 COMPLETE 事件，最多等 2s）
5. 超时仍未清空：记录 warn 日志，强制执行高优先级任务（WAL 模式下并发安全）
6. 高优先级任务执行完成，`highPriorityRunning--`
7. 若 `highPriorityRunning === 0`，调用 `pump()` 从队列取出下一个低优先级任务执行

### 2.4 低优先级任务恢复

- 高优先级任务结束时（计数归零）自动调用 `pump()`
- `pump()` 检查：`!isPaused && highPriorityRunning === 0 && runningLow.size < maxLowConcurrency && lowPriorityQueue.length > 0`
- 满足条件则从队首取出任务，加入 `runningLow`，调用 `task.run()`
- 任务完成（无论成功失败）后从 `runningLow` 移除，再次 `pump()`

### 2.5 任务取消

- **队列中等待的低优先级任务**：`cancelToken.cancelled = true`，`pump()` 时跳过
- **正在执行的低优先级任务**：通过注册的 `cancel()` 回调向 worker 发送 STOP，worker 在当前批次完成后退出循环并发送 COMPLETE
- **应用退出**：`isPaused = true`，所有队列任务标记 cancelled，正在执行的等待 2s 超时后强制退出

## 3. 并发数控制

### 3.1 配置

| 参数 | 默认值 | 说明 |
|---|---|---|
| `maxLowConcurrency` | `1` | 低优先级任务最大并发数。保守值，避免与高优先级任务争抢 CPU/IO |
| `highPriorityMaxConcurrency` | `Infinity` | 高优先级任务不限制并发（用户主动触发应立即响应） |
| `lowPriorityStartDelay` | `0` | 低优先级任务启动延迟（用于启动期等主窗口 ready） |

### 3.2 为什么低优先级并发设为 1

1. **避免与高优先级争抢**：低优先级任务通常是 sharp/ffmpeg 密集型，并发 2 个就会让用户操作的 UI 卡顿
2. **避免 DB 写入冲突**：虽然 WAL 模式支持并发，但 `busy_timeout=5000` 仍可能在并发写入时触发重试，串行更稳
3. **避免缓存抖动**：LRU 校准、缩略图批量生成都会读写 `cache-metadata.json`，串行避免覆盖

### 3.3 启动期特殊处理

- `app.whenReady()` 后到 `createMainWindow()` 完成前，`isPaused = true`
- `mainWindow.on('ready-to-show')` 后 `isPaused = false`，开始 pump 队列
- 这样所有启动期注册的低优先级任务（启动扫描、LRU 校准、崩溃清理、启动备份）都会等到主窗口显示后才执行

- 所有 `ipcMain.handle` 的 channel 名、参数、返回值结构完全不变
- 渲染层无需任何修改

### 5.2 用户主动触发的任务自动按高优先级处理

- `scanner:start` / `media:generatePhash` / `media:markDuplicates` 等 IPC 入口默认走 `scheduler.runHighPriority(...)` 包裹
- 行为与之前一致：立即执行，返回 Promise

### 5.3 启动期自动任务自动入队

- `performStartupScan` 等启动期 fire-and-forget 任务改为 `scheduler.enqueueLow(...)`
- 调用方代码结构不变（仍是 `void ... .catch(...)`），只是被调度器包裹
- 执行时机可能延后（等主窗口 ready + 高优先级任务完成），但语义不变

### 5.4 worker 进程接口不变

- scanner worker / media worker 的消息协议完全不变
- 暂停通过现有的 STOP 消息实现，worker 侧 `shouldStop` 标志已支持
- worker 收到 STOP 后完成当前批次 → 发送 COMPLETE(success=true, message='用户取消') → 主进程 bridge 转发 → 调度器从 runningLow 移除

### 5.5 退出清理顺序不变

- `performCleanup` 中先 `taskScheduler.pause()` + `taskScheduler.cancelAllLow()`
- 再走现有的 `scannerWorkerBridge.dispose()` / `mediaWorkerBridge.dispose()` / `dbManager.close()` 等步骤
- 不破坏现有退出超时保护（2s/5s）

## 6. 边界情况处理

### 6.1 低优先级任务被抢占后的恢复策略

- **不自动恢复**：被 STOP 的低优先级任务视为「完成」（worker 已发送 COMPLETE）
- **原因**：自动恢复需要保存任务的中间进度（已处理多少条），复杂度高且易出错
- **替代方案**：下次启动或下次用户主动触发同类任务时会自然补全（增量扫描、增量缩略图生成都是幂等的）
- **可选优化**：调度器在抢占时记录被取消的任务，高优先级任务完成后**重新入队尾部**（不是恢复执行，而是重新执行一次，依赖任务的幂等性）

→ 本计划采用「不自动恢复」策略，保持简单。如后续需要，可在 `task.cancel()` 时返回 `{ reschedule: true }` 触发重新入队。

### 6.2 高优先级任务嵌套

- 高优先级任务内部触发的低优先级任务（如 `scanner:start` 完成后链式 `generateThumbnailsForUnprocessed`）
- 处理：链式触发的调用点显式传 `priority: 'low'`，自动入队
- 高优先级任务本身先 resolve，低优先级任务等高优先级计数归零后执行

### 6.3 同类任务互斥锁保留

- `MediaWorkerManager.isThumbnailRunning` 等互斥锁**保留**
- 调度器只是优先级队列，不替代 worker 内部的互斥控制
- 若调度器 pump 出一个 phash 任务但 worker 内部 `isPhashRunning=true`（理论不会发生，但防御），manager 返回 `{ success: false, message: '已在进行中' }`，调度器视为完成

### 6.4 调度器自身异常

- `task.run()` 抛错：catch 后记录 logger.error，从 runningLow 移除，继续 pump
- 调度器内部状态损坏：暴露 `scheduler.reset()` 方法，clear 队列 + clear runningLow（仅在 performCleanup 调用）

### 8.1 静态验证

- `npm run typecheck` 通过
- `npm run build:main` 通过

### 8.2 运行时验证（`npm run dev`）

1. **启动期任务延迟**：
   - 启动应用，主窗口显示前任务管理器不应出现 `media-worker` 进程
   - 主窗口 `ready-to-show` 后，启动扫描开始（`scanner-worker` 进程出现）
   - 扫描完成后，缩略图批量生成等任务串行执行（观察日志：不应有并发 sharp/ffmpeg）

   - 启动完成后，观察日志：LRU 校准、启动备份、崩溃清理不应同时执行

4. **退出清理**：
   - 任务执行中关闭应用
   - 退出应在 2s 内完成（调度器立即 pause + cancelAllLow，不等任务完成）
   - 无僵尸 worker 进程残留

5. **UI 流畅度**：
   - 后台任务执行期间，切换页面/拖动窗口 FPS ≥ 55
   - 用户主动触发任务时，UI 立即响应（无延迟）

### 8.3 打包验证（`npm run dist:win`）

- 安装后首次启动行为与 dev 一致
- worker 进程能正确加载 asar unpacked 的 sharp/ffmpeg/better-sqlite3

### 8.4 异常路径验证

1. **worker 异常退出**：任务管理器强杀 `media-worker` 进程
   - 调度器收到 bridge 合成的失败 COMPLETE，从 runningLow 移除
   - 继续执行队列下一个任务

2. **高优先级任务抛错**：模拟 IPC handler 抛错
   - 调度器 catch 错误，highPriorityRunning-- 
   - 继续执行队列

3. **调度器队列堆积**：连续触发 10 个低优先级任务
   - 应串行执行，不并发
   - 队列长度日志可见

## 9. 风险与回滚

### 9.1 风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 调度器单点故障 | 所有任务无法执行 | 调度器代码保持简单（< 200 行），充分 catch |
| 启动期任务延后过久 | 首次扫描推迟到主窗口 ready | `ready-to-show` 后立即 resume，延迟通常 < 100ms |
| 抢占后任务不恢复 | 增量任务下次启动才补全 | 任务幂等性保证；用户可手动触发 |
| 退出时任务中断 | 数据库写入可能不完整 | worker 内部事务保证；WAL 模式 crash-safe |

### 9.2 回滚方案

- 调度器是纯增量改动，回滚只需：
  1. 删除 `src/main/scheduler/task-scheduler.ts`
  2. 还原 `thumbnail-phash-service.ts` / `backup-service.ts` / `editor-service.ts` / `index.ts` / `misc.ts` 的调用方式（去掉 scheduler 包裹）
  3. 还原 `handler-context.ts` 的字段
- worker 进程接口零改动，回滚不影响 worker
