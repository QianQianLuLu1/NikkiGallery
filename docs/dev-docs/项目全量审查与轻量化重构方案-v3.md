## 2. 分维度问题与优化/删改方案

### 2.1 底层架构

以「深度模块」视角审查：模块应"小接口 + 深实现"，反过来即为浅模块。**接缝**是模块边界的明确位置，**适配器**是满足接口的具体实现。本维度共发现 1 项致命 + 9 项严重 + 7 项一般。

#### 2.1.1 致命问题

**【致命-A1】scanner 全盘扫描阻塞主进程事件循环**

- **位置**：`src/main/scanner/index.ts:359-469`（startScan）+ `561-607`（findMediaDirectories）+ `1085-1153`（saveBatchToDatabase）
- **根因分析**：scanner 虽包裹在 async 函数中，但有三处主线程阻塞点：
  1. `findMediaDirectories` 的 shallow search（深度 4）+ deep search（深度 15）串行，遍历大量目录时 `fsp.readdir` 异步但 better-sqlite3 写入是同步事务，每个 batch 提交阻塞事件循环 50-200ms
  2. `saveBatchToDatabase` 的 prepared statement + ON CONFLICT 在主线程执行，1000 条记录事务约阻塞 50-200ms
  3. fullScan 模式扫描所有盘符，单次完整扫描可能持续 30 秒以上，期间 IPC 响应延迟显著增加
- **删改建议**：把 scanner 整体迁移到 `worker_threads`。better-sqlite3 在 worker 中创建独立连接，主进程通过 MessagePort 接收进度。或更轻量——至少把 `saveBatchToDatabase` 改为 `worker_threads` 跑，扫描 IO 留在主进程（IO 已异步）。
- **预期收益**：扫描期间 UI 60fps 不掉帧；IPC 响应延迟 < 16ms；用户可继续操作而扫描后台进行。
- **架构词汇**：当前 scanner 是「无接缝的深实现」——直接耦合 db 句柄与 fsp；深化方向是引入 ScannerWorker seam，主进程只持进度订阅接口。

#### 2.1.2 严重问题

**【严重-A2】media.ts 1367 行内联 786 行 SQL，handler 既是适配器又是实现**

- **位置**：`src/main/ipc/handlers/media.ts`（全文件 1367 行）
- **根因分析**：handler 不只做 IPC 边界适配，还内联了大量 SQL 业务逻辑——`media:list` 一条 SELECT 就 786 行（含 7 表 LEFT JOIN + 动态过滤 + 排序 + 分页）、`findDuplicates`、`findSimilar`、`profile:*` 全部在 handler 里直接写 SQL。这是典型的「浅模块」反模式：handler 看似只是路由层，实际承担了 MediaRepository 的职责。**深度模块**的标志是"简单接口 + 复杂实现"，此处反过来变成了"复杂接口 + 内联实现"。
- **删改建议**：新建 `src/main/repositories/media-repository.ts`，把 `listMedia`、`findDuplicates`、`findSimilar`、`profile:*` 的 SQL 全部搬进去，handler 只做参数校验 + 调用 repo + 错误包装。预计可拆出 4-5 个 repository 文件（media/profile/duplicate/similar）。
- **预期收益**：media.ts 减至 ~400 行；SQL 集中后可独立单元测试（mock dbManager）；新功能（如视频列表）只需扩 repository，不再膨胀 handler。
- **架构词汇**：当前是 handler = adapter + implementation 混合；深化为 handler（adapter）+ MediaRepository（deep module）。**deletion test**：删掉 handler 后 SQL 复杂度消失，但会在 N 个调用点重新出现——确实在赚其用。

**【严重-A3】misc.ts 454 行 8+ 域混杂，模块边界被「杂项」伪概念抹平**

- **位置**：`src/main/ipc/handlers/misc.ts`
- **根因分析**：单文件混合了 scanner/decrypt/thumbnail/ui-theme/shell/settings/dialog/data/import/app/operation-history 等 8+ 个互不相关的域。每个域只有 30-60 行，但都被塞进同一文件。**模块**的定义是"有接口有实现的东西"，misc 没有 interface，只是个杂物间。
- **删改建议**：按域拆分为 `scanner-handler.ts`、`decrypt-handler.ts`、`settings-handler.ts`、`shell-handler.ts`、`import-handler.ts`、`app-handler.ts`、`operation-history-handler.ts`。
- **预期收益**：单文件不超过 100 行；定位 bug 时无需通读 454 行；新增域时不必再往"杂项"里塞。

**【严重-A4】ScannerManager 同时持有 db + dbManager 双引用，绕过接缝**

- **位置**：`src/main/scanner/index.ts:213-244`
- **根因分析**：ScannerManager 有两个私有字段：`db: Database.Database | null` 和 `dbManager: DatabaseManager | null`。scanner 内部 SQL（saveBatchToDatabase L1085-1153、checkMissingFiles 等）直接走 `this.db.prepare()`，完全绕过 DatabaseManager 的 query/execute 抽象，导致 DatabaseManager 这个**接缝**形同虚设。
- **删改建议**：移除 `this.db` 字段，scanner 所有 SQL 调用统一走 `this.dbManager`。如果 dbManager 的 query/execute 不够用（如需要 prepared statement 缓存），就在 DatabaseManager 上加 `prepare(sql)` 接口，而不是让 scanner 直接拿底层句柄。
- **预期收益**：数据访问层有唯一入口；测试 scanner 时只需 mock dbManager；未来换 SQL 驱动只改 DatabaseManager 一处。

**【严重-A5】缺失 MediaRepository / ProfileRepository 接缝**

- **位置**：`src/main/ipc/handlers/media.ts` + `src/main/scanner/index.ts:1085-1153`
- **根因分析**：项目没有 repository 层。media.ts 和 scanner 都直接拼 SQL 字符串 + 调 dbManager.execute。SQL 散落在 3+ 个文件（media.ts、scanner、editor.ts），同一张 media 表的访问逻辑无法收敛。这正是「缺失接缝」——本应有一个 MediaRepository 作为业务层与数据层的接缝。
- **删改建议**：新建 `src/main/repositories/` 目录，至少拆出 `media-repository.ts`、`profile-repository.ts`、`filter-preset-repository.ts`、`operation-history-repository.ts`。DatabaseManager 暴露 `getRepository<T>()` 或直接由构造函数注入。
- **预期收益**：SQL 集中；media 表 schema 变更只改 repository 一处；可对 repository 做独立单元测试。
- **架构词汇**：**One adapter = hypothetical seam, two = real one**——media.ts 与 scanner 都访问 media 表（两个 adapter），是真实接缝，必须抽出 Repository。

**【严重-A6】DatabaseManager 暴露原始 db 句柄，封装泄漏**

- **位置**：`src/main/database/connection.ts`（596 行）
- **根因分析**：虽然封装了 query/execute，但 scanner 仍能通过 `dbManager` 的方法或字段拿到底层 `Database` 句柄（否则 scanner 的 `this.db` 从何而来）。这是封装泄漏——上层应该只看到意图接口，不应看到驱动句柄。
- **删改建议**：DatabaseManager 只暴露意图明确的接口（query/execute/prepare/transaction），不暴露 `getDb()` 或公开 `db` 字段。scanner 的 prepared statement 缓存改为通过 `dbManager.prepare(sql)` 拿到。
- **预期收益**：数据访问层可整体替换；测试可注入 in-memory db；驱动升级（better-sqlite3 → node:sqlite）只改一处。

**【严重-A7】media.ts 内联 786 行 7 表 JOIN SQL，无法被单元测试覆盖**

- **位置**：`src/main/ipc/handlers/media.ts` 中 `media:list` 的 SELECT（约 786 行）
- **根因分析**：单条 SQL 涉及 7 表 LEFT JOIN + 动态 WHERE + 排序 + 分页，全部内联在 handler 字符串拼接里。SQL 字符串无法被 IDE 类型检查、无法被单元测试覆盖、无法被静态分析。这是 v2 审查未充分覆盖的高耦合热点。
- **删改建议**：把 SQL 拆成 `media-repository.ts` 中的 `listMedia(filters)` 方法，配合 query builder（如自写轻量 builder）或至少把 SQL 模板字符串集中到 repository 常量。考虑用 TypeScript 类型推导 `as const` 让结果有类型。
- **预期收益**：SQL 可独立测试；media 表 schema 变更只改一处；handler 减至 ~50 行。
- **注**：本项与 A2 同源，合并实施。

**【严重-A8】缩略图生成无 worker 线程，sharp 同步阻塞主进程**

- **位置**：`src/main/thumbnail/generator.ts`（505 行）
- **根因分析**：ThumbnailGenerator 在主进程内调用 sharp。sharp 是基于 N-API 的原生模块，单次 resize 虽有 libuv 线程池支持，但 composite + metadata 等链式调用仍会占用主线程的 JS 编排时间。批量生成（如首次扫描后补算 1000 张缩略图）期间，主进程 IPC 队列会显著积压。
- **删改建议**：用 `utilityProcess`（Electron 22+）或 `worker_threads` 跑 sharp。主进程只负责调度和缓存查询，实际图像处理在 worker。LRU 缓存留在主进程（查询热路径），生成逻辑移到 worker。
- **预期收益**：批量缩略图生成时 UI 不卡顿；主进程 IPC 响应稳定；可并行多 worker 提升吞吐。

**【严重-A9】better-sqlite3 同步调用遍布所有 IPC handler**

- **位置**：`src/main/ipc/handlers/media.ts`（786 行 SELECT）+ `file.ts`（batchRename 同步事务）+ `editor.ts`（filter_presets）+ `watermark.ts` + `misc.ts`（operation_history）
- **根因分析**：better-sqlite3 是同步驱动，所有 `dbManager.query/execute` 在主线程执行。media:list 这种 786 行 SQL + 7 表 JOIN 的查询，单次可能耗时 100-500ms，期间主进程完全阻塞。用户在列表滚动时触发翻页查询，会感受到明显卡顿。
- **删改建议**：长查询（list/duplicates/similar）迁移到 worker_threads 中的只读副本（better-sqlite3 支持多连接读同一文件）；短查询（getById、setting）保留主线程。或彻底迁移到 `node:sqlite`（Node 22+ 内置）的异步 API。
- **预期收益**：列表翻页、相似查询不再卡 UI；主进程保持 60fps；并发查询能力提升。

**【严重-A10】performStartupScan 阻塞应用首屏 5-15 秒**

- **位置**：`src/main/index.ts:1234-1279`（performStartupScan）+ `1282-1311`（autoAnalyzeSceneTime）
- **根因分析**：应用启动时执行 `incremental:false, fullScan:true` 的全盘扫描（首次启动）+ 缩略图/pHash 补算 + 场景时段分析，全部在主进程串行。用户点击应用图标到首屏可交互可能延迟 5-15 秒（取决于相册大小）。
- **删改建议**：拆分为三阶段——
  1. 快速索引（启动后 500ms 内完成）：仅读 db 已有记录，渲染列表
  2. 增量扫描（后台）：检测新增文件，逐步插入 db
  3. 缩略图/pHash 补算（空闲时）：requestIdleCallback 或 worker
- **预期收益**：首屏可交互时间从 5-15s 降至 < 1s；用户感知"应用秒开"。

**【严重-F4】share installPathCache 永久缓存无 TTL/失效策略**

- **位置**：`src/main/services/share-clipboard-service.ts:489`；命中检查 `L501-503`
- **根因分析**：`const installPathCache: Record<string, string | null> = {}` 为模块级永久缓存，应用启动后第一次检测后整个生命周期内不再重新检测。命中逻辑 `if (channelId in installPathCache) return installPathCache[channelId]` 仅检查 key 存在性，无 TTL、无失效机制。问题场景：
  - 用户首次分享时 WeChat 未安装，缓存 `installPathCache['wechat'] = null`
  - 用户后续安装 WeChat，再次分享仍命中 null 缓存，UI 永远显示"未安装"
  - 反向场景：用户卸载已检测到的应用，缓存仍返回 installPath，UI 显示"已安装"但 `share:launchApp` 失败
- **删改建议**：组合方案——
  - (A) 加 TTL：`installPathCache[channelId] = { path, expireAt: Date.now() + 5 * 60 * 1000 }`，5 分钟过期重检
  - (B) 显式失效：在 `share:detectApp` handler 增加 `force?: boolean` 参数，UI 提供"重试检测"按钮
- **预期收益**：消除"安装/卸载后检测不准"的用户反馈，分享引导窗状态准确。

**【严重-F5】ensureDesktopShortcut 未显式读 Windows 注册表自定义桌面路径**

- **位置**：`src/main/index.ts:686`
- **根因分析**：项目硬约束明确要求「应用必须处理 Windows 注册表自定义桌面路径」，但实装直接 `app.getPath('desktop')` 依赖 Electron 内置行为。已知问题：
  - 当注册表值为 `REG_EXPAND_SZ` 类型且包含 `%USERPROFILE%` 等环境变量时，Electron 在某些版本下展开不一致
  - OneDrive 重定向桌面场景下（Windows 10/11 常见），路径可能为 `C:\Users\<user>\OneDrive\桌面`
  - 当桌面目录不可写时（企业策略限制），`shell.writeShortcutLink` 静默失败，无 fallback 到用户主目录
- **删改建议**：
  1. 显式读取注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\Shell Folders\Desktop`，与 `app.getPath('desktop')` 比对，取存在且可写者
  2. 增加 `fs.accessSync(desktopPath, fs.constants.W_OK)` 写权限校验，失败时 fallback 到 `app.getPath('home')`
  3. `shell.writeShortcutLink` 返回 false 时记录 `logger.error` 并尝试 fallback 路径
- **预期收益**：企业环境/OneDrive 重定向/自定义桌面路径场景下快捷方式创建成功率提升，符合硬约束。

### 4.3 风险与权衡

1. **P1-A5 scanner worker 化**（XL 工时）：better-sqlite3 在 worker 中需独立连接，WAL 模式下多连接读一致性强，但写连接仍需串行。建议保留主进程一个写连接，worker 只读。风险：迁移期间可能引入新的并发 bug。
2. **P1-A7 长查询 worker 化**（L 工时）：better-sqlite3 多连接读同一文件需 WAL 模式（已开启），但读副本可能读到稍旧数据。对于"列表翻页"场景可接受，对于"重复检测"需保证扫描完成后再读。
3. **P1-C3 i18n 批量处理**（XL 工时）：9 个组件 30+ 处文案，需在 en.json 中补 60+ key，再同步到 11 种语言文件。建议用 i18next 的 `returnObjects: true` 减少数组类配置的 key 数量。
4. **P2-2 release/ 版本处理**：需用户显式确认当前发布版本是 2.5.0 还是 2.6.0，不可擅自更改 package.json 版本号（项目记忆硬约束）。

## 附录 C：架构词汇对照表

| 词汇 | 定义 | 本项目应用 |
|---|---|---|
| **Module** | 有接口有实现的东西 | media.ts / scanner / DatabaseManager 都是 module |
| **Interface** | 调用方必须知道的一切 | handler 的 IPC 契约 + Repository 的方法签名 |
| **Depth** | 接口背后的行为量 | media.ts 是反例：接口大、实现内联，深度为零 |
| **Seam** | 改行为不改代码的位置 | Repository 是缺失的 seam；DatabaseManager 是被绕过的 seam |
| **Adapter** | 满足接口的具体实现 | handler 是 IPC 边界的 adapter；scanner 应是 Repository 的 adapter |
| **Leverage** | 调用方从深度获得的收益 | Repository 抽出后，N 个调用点共享同一 SQL 实现 |
| **Locality** | 维护者从深度获得的收益 | SQL 集中后，schema 变更只改一处 |
| **Deletion test** | 删掉模块后复杂度是否消失 | handler 删掉后 SQL 复杂度消失但在 N 调用点重现 → 在赚其用 |
