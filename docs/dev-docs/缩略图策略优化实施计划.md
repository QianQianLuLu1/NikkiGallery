# 缩略图策略优化实施计划

> 适用版本：wxnn-photo-manager v2.3.0
> 制定日期：2026-07-18
> 涉及代码：`src/main/thumbnail/`、`src/main/ipc/handlers/`、`src/main/preload.ts`、`src/renderer/components/gallery/`、`src/renderer/pages/settings/`

---

## 一、优化目标

在**不破坏现有功能**的前提下，完成两项增强：

1. **提升高分屏清晰度**
   - 现状：仅有 `low`(64px) + `standard`(320px) 两档，2x/3x 高分屏（4K 屏、Retina 屏）下图库模糊。
   - 目标：新增 `high`(512px) 高清档位，渲染层按 `window.devicePixelRatio` 自动选用，DPR < 2 时保持现状，DPR ≥ 2 时自动切换到高清档位。

2. **增加缓存容量管控**
   - 现状：已有 1/2/3/5/10 GB 五档预设按钮 + LRU 自动淘汰。但缺少自定义输入、缺少淘汰策略可视化说明。
   - 目标：在「数据管理」缓存卡片中新增「自定义上限」输入框，并补充淘汰策略说明卡片，让用户明确知道缓存按「最后访问时间」淘汰。

## 三、具体实现

### 3.1 新增 512px 高清档位（`generator.ts`）

#### 3.1.1 新增字段（类成员）

在 `ThumbnailGenerator` 类中现有 `lowMaxWidth/lowMaxHeight/lowQuality` 之后追加：

```ts
// 高清档位参数（高分屏 DPR ≥ 2 时使用）
private highMaxWidth = 512
private highMaxHeight = 512
private highQuality = 90
```

#### 3.1.2 命名规则

| 档位 | 文件名格式 | 分辨率 | 质量 |
|---|---|---|---|
| low（现有） | `${fileHash}_low.jpg` | 64×64 | q30 |
| standard（现有） | `${fileHash}.jpg` | 320×320 | q85 |
| **high（新增）** | `${fileHash}_high.jpg` | 512×512 | q90 |

#### 3.1.3 新增 `doGenerateHigh` 方法

模仿现有 `doGenerateLow` 的实现模式：

```ts
// 高清档位生成（512px, q90）
// 命名规则：${fileHash}_high.jpg
// 优先从原文件生成（保证清晰度）；若原文件不存在则从 standard 缩放
private async doGenerateHigh(filePath: string): Promise<string | null> {
  try {
    await this.ensureCacheDir()
    if (!(await pathExists(filePath))) return null

    const stats = await fsp.stat(filePath)
    const fileHash = await this.getFileHash(filePath, stats)
    const highPath = path.join(this.cacheDir, `${fileHash}_high.jpg`)

    // 缓存命中直接返回
    if (await pathExists(highPath)) return highPath

    // 优先从原文件生成（保证清晰度，512px 比原图小时 withoutEnlargement 生效）
    await sharp(filePath)
      .resize(this.highMaxWidth, this.highMaxHeight, {
        fit: 'inside',
        withoutEnlargement: true
      })
      .jpeg({ quality: this.highQuality, progressive: true })
      .toFile(highPath)

    // 累加缓存大小计数器
    try {
      const thumbStat = await fsp.stat(highPath)
      this.totalCacheSize = Math.max(0, this.totalCacheSize) + thumbStat.size
    } catch {}

    // 记录访问时间 + 触发 LRU 检查
    this.accessTimes.set(fileHash, Date.now())
    this.accessCounter++
    await this.enforceCacheLimit()
    await this.maybePersistAccessTimes()
    return highPath
  } catch (error) {
    console.error('[Thumbnail] 高清缩略图生成失败:', error)
    return null
  }
}
```

#### 3.1.4 修改 `generate` 方法签名与分发

将现有 `generate` 方法的 quality 参数类型扩展，并在分发逻辑中增加 high 分支：

```ts
async generate(
  filePath: string,
  quality?: 'low' | 'standard' | 'high'
): Promise<string | null> {
  // 锁 key 区分档位，避免并发重复生成
  const lockKey = quality === 'low' ? `${filePath}:low` :
                  quality === 'high' ? `${filePath}:high` : filePath
  const existing = this.generatingLocks.get(lockKey)
  if (existing) return existing

  const promise = (
    quality === 'low' ? this.doGenerateLow(filePath) :
    quality === 'high' ? this.doGenerateHigh(filePath) :
    this.doGenerate(filePath)
  ).finally(() => {
    this.generatingLocks.delete(lockKey)
  })

  this.generatingLocks.set(lockKey, promise)
  return promise
}
```

#### 3.1.5 视频缩略图高清档位说明

`doGenerateHigh` 仅处理图片。视频高清档位的实现策略：

- **本期不实现视频 high 档位**（视频帧提取耗时较高，且视频在图库中占比小）
- 视频文件请求 high 档位时，自动降级到 standard（在 `doGenerateHigh` 中检测 `isVideoExt`，若是视频则调用 `this.doGenerate(filePath)` 返回 standard 路径）
- 这样保证视频卡片在高分屏下不会因请求 high 失败而显示空白，依然显示 standard 缩略图

```ts
// doGenerateHigh 开头追加：
const ext = path.extname(filePath).toLowerCase()
if (isVideoExt(ext)) {
  // 视频暂不生成高清档位，降级到 standard
  return this.doGenerate(filePath)
}
```

### 3.2 高分屏自动适配（`VirtualImageGrid.tsx`）

#### 3.2.1 新增工具函数 `toHighQualityUrl`

模仿现有 `toLowQualityUrl` 的实现，在 `toLowQualityUrl` 之后追加：

```ts
const toHighQualityUrl = useCallback((thumbnail: string | undefined): string | null => {
  if (!thumbnail) return null
  // 仅对标准缩略图路径（xxx.jpg）推导高清版本（xxx_high.jpg）
  // _low.jpg 和 _high.jpg 不再推导，避免链式错误
  if (thumbnail.endsWith('.jpg') && !thumbnail.endsWith('_low.jpg') && !thumbnail.endsWith('_high.jpg')) {
    return thumbnail.slice(0, -4) + '_high.jpg'
  }
  return null
}, [])
```

#### 3.2.2 DPR 检测（模块级常量，一次检测）

在文件顶部 import 之后追加：

```ts
// 高分屏检测：DPR ≥ 2 时启用高清缩略图
// 模块加载时一次性检测，避免每次渲染都读取
const HIGH_DPI_MODE = typeof window !== 'undefined' && window.devicePixelRatio >= 2
```

> **零基础说明**：`window.devicePixelRatio` 是浏览器/Electron 提供的屏幕物理像素与 CSS 像素的比值。普通屏为 1，Retina 屏为 2，4K 屏可能为 1.5~2。≥2 时认定为高分屏。

#### 3.2.3 修改缩略图 URL 选用逻辑

在现有 `standardUrl` / `lowUrl` 旁边追加 `highUrl`，并修改 `thumbUrl` 选用优先级：

```ts
const standardUrl = toFileUrl(file.thumbnail || file.file_path)
const useHighQuality = highQualityIds.has(file.id)
const lowUrl = toLowQualityUrl(file.thumbnail)
const highUrl = HIGH_DPI_MODE ? toHighQualityUrl(file.thumbnail) : null

// 选用优先级：
// 1. 用户已切到高质量视图 → standardUrl
// 2. 高分屏且 high 缓存可用 → highUrl
// 3. 默认视图且 low 缓存可用 → lowUrl
// 4. 兜底 → standardUrl
const thumbUrl =
  (useHighQuality || !lowUrl) ? standardUrl :
  (highUrl ? toFileUrl(highUrl) : toFileUrl(lowUrl))
```

**关键说明**：
- `highUrl` 推导仅基于路径字符串，**不会触发实际文件生成**。若 high 缓存文件不存在，浏览器加载 `<img>` 会失败，此时渲染层的 `hasError` 机制会触发降级到 `standardUrl`（已有逻辑）。
- 为避免每次都失败，需在首次访问时主动触发生成。生成触发点放在 `useEffect` 中，监听 `visibleItems` 变化，对 DPR ≥ 2 且未生成 high 的文件调用 `thumbnail.generate(filePath, 'high')`。

#### 3.2.4 high 档位预生成（后台触发）

在现有 `useEffect` 旁边追加一个独立的 `useEffect`：

```ts
// 高分屏模式下，预生成可见项的高清缩略图（后台异步，不阻塞渲染）
useEffect(() => {
  if (!HIGH_DPI_MODE) return
  if (!window.electronAPI?.thumbnail?.generate) return
  // 仅对当前可见项预生成，避免一次性生成全部
  const items = visibleItems.slice(0, 20) // 仅前 20 项，控制并发
  items.forEach(({ item }) => {
    const file = item as MediaFile
    if (!file.thumbnail) return
    // 后台触发，不 await，不影响渲染
    window.electronAPI.thumbnail.generate(file.file_path, 'high').catch(() => {})
  })
}, [visibleItems])
```

> **注意**：此 `useEffect` 依赖 `visibleItems`，在用户滚动时会重复触发。由于 `generator.ts` 内部已有 `generatingLocks` 缓存命中检查，重复调用不会重复生成，只会命中缓存返回路径。但仍会带来 IPC 开销，因此限制为前 20 项。

### 3.4 缓存自动淘汰逻辑（已存在，本期补充说明）

#### 3.4.1 现状评估

经核查 `generator.ts` 第 455-502 行，**LRU 淘汰逻辑已完整实现**：

| 优化项 | 现状 | 评估 |
|---|---|---|
| 淘汰策略 | 按 `accessTimes` Map 记录的最后访问时间升序淘汰 | ✅ 已满足「按最后访问时间淘汰」 |
| 访问时间记录时机 | 缓存命中 + 新生成时均更新 | ✅ 已优化 |
| 访问时间持久化 | 每 100 次访问持久化一次到 `cache-metadata.json` | ✅ 已优化 |
| 跨重启保留 | 启动时 `loadAccessTimes` 加载 sidecar JSON | ✅ 已优化 |
| 触发时机 | 内存计数器超阈值 10% + 后台 5 分钟定时 + 手动 enforce | ✅ 已优化 |
| 高质量计数器 | `totalCacheSize` 内存维护，避免全目录扫描 | ✅ 已优化 |

#### 3.4.2 本期对 LRU 逻辑的改动

**仅补充一处**：在 `doGenerateHigh` 中正确维护 `accessTimes` 与 `totalCacheSize`（已在 3.1.3 代码中体现），确保 high 档位也纳入 LRU 管控。

**不改动** `enforceCacheLimit` 方法本身，保持现有淘汰逻辑不变。

#### 3.4.3 high 档位与 LRU 的协同

- high 缓存文件 (`${hash}_high.jpg`) 与 standard / low 共享同一个 `accessTimes` 记录（key 为 `fileHash`，不带后缀）
- 三档缓存中任一档被访问，都会更新该 `fileHash` 的访问时间
- LRU 淘汰时按文件名解析 hash（`entry.name.replace(/\.jpg$/, '').replace(/_low$/, '').replace(/_high$/, '')`），三档同步淘汰

**需要修改 `enforceCacheLimit` 中的 hash 解析逻辑**（仅 1 行）：

```ts
// 现有：
const hash = entry.name.replace(/\.jpg$/, '')

// 改为：
const hash = entry.name.replace(/\.jpg$/, '').replace(/_low$/, '').replace(/_high$/, '')
```

这样确保淘汰 standard 时同步淘汰同 hash 的 low 和 high，避免残留孤儿文件。

## 四、兼容性说明

### 4.1 旧缓存文件兼容

| 旧文件 | 处理方式 |
|---|---|
| `${hash}.jpg` (standard) | **保留不动**，继续使用 |
| `${hash}_low.jpg` (low) | **保留不动**，继续使用 |
| `cache-metadata.json` | **保留不动**，结构未变 |

### 4.2 新文件生成时机

| 新文件 | 生成时机 |
|---|---|
| `${hash}_high.jpg` (high) | 仅在 **DPR ≥ 2** 且用户滚动到该文件可见区域时，由 `VirtualImageGrid` 的 `useEffect` 后台触发 |

### 4.3 降级策略

| 场景 | 降级行为 |
|---|---|
| DPR < 2（普通屏） | 不请求 high，保持 low + standard 双档位现状 |
| DPR ≥ 2 但 high 缓存未生成 | `<img>` 加载失败 → `hasError` 机制降级到 standardUrl |
| 视频文件请求 high | `doGenerateHigh` 检测到视频扩展名，直接返回 standard 路径 |
| high 生成失败 | `doGenerateHigh` 返回 null，不影响其他档位 |

### 4.4 数据库兼容

- `media_files.thumbnail` 列**继续存储 standard 路径**，不改动
- 新增 high 路径不写入数据库，仅由渲染层按命名规则推导
- 数据库迁移、备份、恢复全部不受影响

### 4.5 IPC 兼容

- `thumbnail:generate` 的 quality 参数新增 `'high'` 可选值
- 旧调用方传入 `'low'` / `'standard'` / `undefined` 的行为**完全不变**
- 新调用方传入 `'high'` 才触发新逻辑

## 六、验证方法

### 6.1 高清档位生成验证

启动应用后，在主进程开发者工具执行：

```js
// 模拟高分屏：手动调用 high 档位生成
await window.electronAPI.thumbnail.generate('某图片路径', 'high')
// 预期：返回路径以 _high.jpg 结尾
```

文件系统检查：
- 缩略图目录下应出现 `${hash}_high.jpg` 文件
- 文件尺寸约为 standard 的 1.5-2 倍（512² vs 320²）

### 6.2 高分屏自动适配验证

1. 在 4K/Retina 屏设备上启动应用（或通过 Electron DevTools 模拟 DPR=2）
2. 进入图库页面
3. 检查 Network 面板：`<img>` 的 `src` 应包含 `_high.jpg`
4. 检查图片显示清晰度：应明显比优化前清晰

### 6.3 普通屏无影响验证

1. 在普通屏设备上启动应用（DPR=1）
2. 进入图库页面
3. 检查 Network 面板：`<img>` 的 `src` 应**不含** `_high.jpg`，仍为 `${hash}.jpg` 或 `${hash}_low.jpg`
4. 缩略图目录下**不应**出现 `_high.jpg` 文件

### 6.4 视频降级验证

1. 对视频文件调用 `thumbnail.generate(videoPath, 'high')`
2. 预期返回 standard 路径（`${hash}.jpg`，无 `_high` 后缀）
3. 缩略图目录下**不应**出现该视频的 `_high.jpg` 文件

### 6.5 缓存上限自定义验证

1. 进入「设置 → 数据管理 → 缓存管理」
2. 在「自定义上限」输入框中输入 `7`，按回车
3. 预期：
   - Toast 提示「上限已设置为 7 GB」
   - 统计卡片「上限」显示 7 GB

## 九、风险评估

| 风险点 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| high 档位生成耗时较长 | 中 | 首次访问某图时后台生成约 50-200ms | 后台异步生成，不阻塞渲染；先显示 standard 再替换 |
| high 缓存占用增长 | 中 | 缓存体积增加约 30% | 受 LRU 上限管控，超限自动淘汰 |
| DPR 检测不准确 | 低 | 某些设备 DPR 介于 1.5-2 之间 | 仅 DPR ≥ 2 启用，1.5 视为普通屏（保守策略） |
| high 预生成 IPC 频繁 | 低 | 滚动时重复触发 IPC | 限制前 20 项 + generator 内部锁缓存命中检查 |
| LRU hash 解析修改破坏现有淘汰 | 极低 | 仅影响文件名解析，不影响淘汰逻辑 | 修改前后对 low/standard 文件名解析结果一致 |
| 自定义上限输入非法 | 低 | 用户输入超范围值 | 输入校验 + Toast 提示 |

**总体风险等级：低**。所有改动均为「纯增量、可回滚、失败有降级」的优化操作。
