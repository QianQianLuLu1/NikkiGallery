# 数据库性能优化实施计划

> 适用版本：wxnn-photo-manager v2.3.0
> 制定日期：2026-07-18
> 涉及代码：`src/main/database/` 与 `src/main/scanner/`

---

## 一、优化目标

解决两类已识别问题，**不引入新功能、不改动业务逻辑**：

1. **大数据量下查询卡顿**
   - 现状：`media_files` 表仅有 17 个**单列索引**，列表查询常用「`is_deleted=0` + `account_uid=?` + `album_type=?`」组合条件，单列索引无法覆盖，SQLite 退化为全表扫描。
   - 目标：为高频组合查询添加联合索引，使列表/分组/统计类查询走索引扫描。

2. **数据库文件膨胀**
   - 现状：项目**没有任何** `VACUUM` / `PRAGMA optimize` / `auto_vacuum` 碎片清理逻辑。长期删除/更新后，数据库文件存在空闲页无法回收，文件体积持续增长。
   - 目标：启用增量自动清理 + 退出时 `PRAGMA optimize`，让查询计划器自动维护统计信息，避免文件无限膨胀。

| `idx_media_files_scene_analysis` | `(scene_time, file_type, is_deleted)` | 场景时段分析 |

#### 创建方式（幂等，安全）

'CREATE INDEX IF NOT EXISTS idx_media_files_duplicate_group ON media_files(is_duplicate, is_deleted, original_id)',
'CREATE INDEX IF NOT EXISTS idx_media_files_outfit_agg ON media_files(is_deleted, outfit)',
'CREATE INDEX IF NOT EXISTS idx_media_files_scene_analysis ON media_files(scene_time, file_type, is_deleted)',

### 3.2 自动碎片清理

#### 触发时机（2 处，均不阻塞用户操作）

| 时机 | 位置 | 执行内容 | 阻塞性 |
|---|---|---|---|
| 应用启动时 | `connection.ts` 的 `initialize()` 方法末尾 | `PRAGMA optimize` | 非阻塞，毫秒级 |
| 应用退出时 | `connection.ts` 的 `close()` 方法中，WAL checkpoint 之前 | `PRAGMA optimize` | 非阻塞，毫秒级 |

#### 执行逻辑

```ts
// 启动时：让 SQLite 自动分析索引使用统计，优化查询计划
try {
  this.db.pragma('optimize')
} catch (err) {
  console.warn('[DB] PRAGMA optimize 失败（已忽略）:', err)
}
```

```ts
// 退出时：再次执行 optimize，确保本次会话的统计信息落盘
try {
  this.db.pragma('optimize')
} catch (err) {
  console.warn('[DB] 退出时 PRAGMA optimize 失败（已忽略）:', err)
}
// 之后执行现有的 wal_checkpoint(PASSIVE)
```

#### 为什么不用 `VACUUM`？

- `VACUUM` 会**重写整个数据库文件**，需要数据库独占锁，万级数据下耗时可达数秒甚至更久，会卡住应用启动或退出。
- `PRAGMA optimize` 是 SQLite 官方推荐的轻量维护操作，仅更新查询计划器统计信息，**毫秒级完成**，不阻塞业务。
- 文件体积回收交由 SQLite 内部的 `auto_vacuum` 机制（见下）。

#### 启用增量自动清理（一次性配置）

在 `initialize()` 方法中，WAL 相关 PRAGMA 之后追加：

```ts
// 启用增量自动清理：删除数据时自动回收空闲页，避免文件无限膨胀
// 注意：auto_vacuum 只能在数据库创建时设置，旧库需通过 VACUUM 一次性转换
// 此处使用 INCREMENTAL 模式，配合 incremental_vacuum 在后台逐步回收
try {
  const autoVacuumMode = this.db.pragma('auto_vacuum', { simple: true })
  if (autoVacuumMode === 0) {
    // 旧库未启用 auto_vacuum：标记为 INCREMENTAL，下次 VACUUM 时生效
    // 不主动执行 VACUUM（会阻塞），仅设置模式，留给未来某次手动维护时生效
    this.db.pragma('auto_vacuum = INCREMENTAL')
  }
} catch (err) {
  console.warn('[DB] auto_vacuum 设置失败（已忽略）:', err)
}
```

> **零基础说明**：`auto_vacuum = INCREMENTAL` 是一种「软回收」模式，删除数据后产生的空闲页会在后续写入时被自动复用，避免文件无限增长。它不会立即缩小文件体积，但能防止文件持续膨胀。旧库（v2.3.0 之前创建的）需要手动执行一次 `VACUUM` 才能真正切换到该模式，本计划**不主动执行 VACUUM**，避免阻塞用户。

## 四、影响范围

### 4.1 数据安全

| 项目 | 影响 |
|---|---|
| 现有数据 | **零影响**。`CREATE INDEX IF NOT EXISTS` 是幂等操作，仅新增索引结构，不修改任何表数据 |
| 数据库结构 | **零影响**。不新增/删除/修改任何表、列、外键、触发器 |
| 数据库文件体积 | **轻微增加**。5 个联合索引会增加约 1-5MB 索引文件空间（取决于数据量），换取查询性能大幅提升 |
| 备份/恢复 | **零影响**。索引随数据库一起备份，恢复后自动重建 |

### 4.2 业务代码

| 项目 | 影响 |
|---|---|
| `media-repository.ts` | **零改动**。所有 SQL 字符串、placeholder、事务边界保持原样 |

### 4.3 性能预期

| 场景 | 优化前 | 优化后 |
|---|---|---|
| 图库默认列表查询（万级数据） | 全表扫描 | 联合索引扫描，预计提升 5-10 倍 |
| 角色档案统计查询 | 全表扫描 | 联合索引扫描，预计提升 3-5 倍 |
| 重复分组查询 | 全表扫描 | 联合索引扫描，预计提升 3-5 倍 |

| 数据库文件体积增长 | 持续膨胀 | 增量回收，增长趋缓 |

## 六、严格承诺

1. **不改动任何业务逻辑**：所有 SQL 查询、IPC 处理器、UI 组件、状态管理代码保持原样。
2. **不改动任何表结构**：不新增/删除/修改任何表、列、外键、触发器、视图。
3. **不删除任何现有索引**：17 个单列索引全部保留，5 个联合索引为纯新增。
4. **不主动执行 VACUUM**：避免阻塞应用启动/退出，`auto_vacuum = INCREMENTAL` 仅设置模式，不强制转换旧库。
5. **不引入新依赖**：所有优化使用 SQLite/better-sqlite3 原生能力，零新增 npm 包。
6. **不修改版本号**：`package.json` 的 `version` 字段保持 `2.3.0` 不变。
7. **不修改 HTML 预览版**：本次为纯主进程数据库层优化，与渲染层无关。
8. **所有改动可回滚**：如出现问题，删除新增的 5 个联合索引 + 移除 PRAGMA optimize 调用即可完全还原。
9. **失败不阻塞启动**：所有新增逻辑均包裹在 try/catch 中，单条失败仅记录日志，不影响应用正常启动。
10. **保持现有健壮性约束**：所有 PRAGMA 调用遵循项目既有的「错误不阻塞启动」原则，与现有 `wal_autocheckpoint` 处理方式一致。
