# 错误报告与崩溃日志改进计划

> **版本**：v1.0
> **制定日期**：2026-07-05
> **背景**：v2.1 / v2.2 版本出现"安装后前几次正常，后续无法启动"问题，初次修复未根因，二次修复才发现是 ffmpeg/ffprobe 子进程未 kill 导致进程不退出 + 单实例锁持有。本次教训暴露了错误诊断体系的多个盲区，需系统性改进。

---

## 一、本次教训总结

### 1.1 问题时间线

| 阶段 | 现象 | 根因 | 诊断盲区 |
|---|---|---|---|
| 用户报告 | 前几次正常，后续无法启动 | — | — |
| 第一次修复 | 仍第二次就失败 | 仅修了 initialize 失败兜底，未修子进程泄漏 | 未发现 ffmpeg/ffprobe 子进程未 kill |
| 第二次修复 | 首次都打不开 | 旧版本僵尸进程残留持锁 | gotLock=false 时静默退出，用户无感知 |
| 第三次修复 | 加入 gotLock=false 提示对话框 | 用户提供僵尸进程清理步骤 | — |

### 1.2 暴露的诊断盲区

1. **进程退出诊断缺失**：无法判断进程是否正常退出、退出时持有哪些资源、子进程是否被 kill
2. **单实例锁状态不可见**：gotLock=false 时静默退出，用户和开发者都不知道发生了什么
3. **渲染进程错误盲区**：无 ErrorBoundary、无 window.onerror、无 unhandledrejection，组件抛错直接白屏
4. **错误反馈链断裂**：FaultType.ipcError 和 FaultType.manual 已定义但全项目零调用
5. **错误弹窗覆盖不足**：仅启动失败弹窗，运行时错误用户不可见
6. **日志信息不完整**：缺版本号、环境信息、错误上下文（最近 N 条日志）

## 二、现状分析

### 2.1 三大系统覆盖度评估

| 维度 | 崩溃日志 | 错误日志 | 错误弹窗 |
|---|---|---|---|
| **覆盖完整性** | 中（仅 native 崩溃） | 中高（主进程完整，渲染层缺 Promise/JS 异常） | 低（仅启动失败 + 设置页 Toast） |
| **信息完整度** | 低（仅文件元信息） | 中（堆栈+上下文，缺版本/环境） | 低（仅消息字符串） |
| **用户可见性** | 中（设置页可看） | 中高（设置页可看 + 导出） | 低（除设置页外几乎不可见） |
| **反馈便利性** | 低（需手动取 dump） | 中（导出 zip + 复制单条） | 低（无反馈通道入口） |
| **容量管理** | 中（20 份硬上限） | 中（5GB 总量，无时间过期） | 不适用 |

### 2.2 关键缺陷清单

#### 崩溃日志系统
- dump 文件为二进制 minidump 格式，普通用户无法阅读
- UI 仅展示文件元信息（filename/size/mtime/processType），**不包含崩溃原因、调用栈**
- 进程类型靠文件名子串匹配，命名规则变化会失效
- 无"上报"按钮，用户需手动取 dump 反馈
- 20 份上限对频繁崩溃场景可能不够，且无时间维度过期

#### 错误报告日志系统
- **渲染进程未捕获 Promise 拒绝**：无 `window.addEventListener('unhandledrejection')`
- **渲染进程 window.onerror**：无监听，资源加载失败、脚本异常无记录
- **渲染进程 React 渲染异常**：无 ErrorBoundary，组件抛错直接白屏
- **FaultType.ipcError 从未被使用**：IPC handler 返回 `{success:false}` 但不记录到 faults
- **FaultType.manual 从未被使用**：业务代码无主动 logFault 调用
- **FaultRecord 字段不够丰富**：缺 appVersion / platform / osVersion / electronVersion / pid
- **rendererError 捕获信息丢失**：仅用 message 字符串构造 Error，原堆栈丢失
- **listFaults 性能问题**：每次全量读取所有 jsonl 文件并 JSON.parse，无索引无缓存
- **exportLogsAsZip 跨平台问题**：依赖 Windows PowerShell
- **5GB 上限过大**：可能耗尽 SSD，且无时间维度过期

#### 错误弹窗系统
- **设置页之外无 Toast**：Gallery / Detail / Editor / Categories / RecycleBin / Duplicates 页面错误对用户不可见
- **无全局 ErrorBoundary**：组件抛错直接白屏
- **无"错误反馈"入口**：需进设置→关于才能找到 QQ 群和 GitHub
- **dialog.showErrorBox 内容简陋**：不含错误类型、堆栈、版本号、环境信息
- **错误提示不一致**：主进程用 dialog，设置页用 Toast，其它页面用 console.error
- **静默吞错普遍存在**：App.tsx 多处 `catch { // 静默忽略 }`

## 三、改进目标

### 3.1 核心目标

1. **可诊断性**：任何错误都能在日志中找到完整上下文（堆栈、版本、环境、最近操作）
2. **可反馈性**：用户能一键导出诊断包，包含所有定位问题所需信息
3. **可见性**：所有错误对用户有适当反馈（Toast / Dialog / 错误页面）
4. **可定位性**：开发者拿到诊断包后能在 5 分钟内定位问题根因

### 3.2 量化指标

| 指标 | 现状 | 目标 |
|---|---|---|
| 渲染进程 JS 错误捕获率 | 0% | 100%（ErrorBoundary + window.onerror + unhandledrejection） |
| 全局 Toast 覆盖页面数 | 1/7（仅设置页） | 7/7（所有页面） |
| FaultRecord 包含版本信息 | 否 | 是（appVersion + electron + node + os） |
| IPC 错误记录到 faults | 0 处 | 关键 IPC handler 100% 覆盖 |
| 错误反馈入口可见性 | 设置→关于 | 主界面右下角常驻 + 错误弹窗内 |
| 进程退出诊断 | 无 | 完整（资源清单 + 子进程状态 + 锁状态） |

## 四、详细改进方案

### 4.1 崩溃日志系统改进

#### 改进点 1：dump 文件信息增强

**目标**：UI 展示崩溃关键信息，无需用户手动解析 dump。

**方案**：
- 集成 `minidump-stackwalk` 或调用 Electron 内置 `crashReporter.getCrashesDirectory()` + 自定义解析
- 在 `crash-service.ts` 新增 `parseCrashDump(filePath): Promise<CrashDetail>` 函数
- `CrashDetail` 包含：`crashReason` / `topFrame` / `threadCount` / `modules`

**优先级**：P2（dump 解析复杂度高，可后续迭代）

#### 改进点 2：崩溃上下文记录

**目标**：崩溃发生时记录最近 N 条主日志，辅助定位。

**方案**：
- `logger.ts` 新增环形缓冲区 `recentLogs: string[]`（容量 100 条）
- `render-process-gone` 触发时，将 `recentLogs` 作为 context 写入 fault
- `crash-service.ts` 新增 `saveCrashContext(pid, context)` 保存为 `<pid>.context.json`

**优先级**：P1

#### 改进点 3：清理策略增强

**目标**：避免崩溃文件占用过多空间。

**方案**：
- `MAX_CRASH_FILES` 从 20 调整为 10（够用且节省空间）
- 新增时间维度过期：超过 30 天的 dump 自动清理
- `enforceCrashLimit` 失败时记录到 startup-errors.log 并重试一次

**优先级**：P2

#### 改进点 4：崩溃上报通道

**目标**：用户能一键上报崩溃 dump 给开发者。

**方案**：
- UI 新增"上报此崩溃"按钮
- 上传到 GitHub Issue（附带 dump 文件）或自建接收端点
- 上报前脱敏（移除可能包含的文件路径中的用户名）

**优先级**：P3（需后端支持，可后续迭代）

### 4.2 错误报告日志系统改进

#### 改进点 1：FaultRecord 字段扩展（P0）

**目标**：每条故障记录都包含完整的环境信息。

**方案**：修改 `src/main/utils/logger.ts` 的 `FaultRecord` 接口：

```typescript
export interface FaultRecord {
  id: string
  timestamp: string
  type: FaultType
  summary: string
  detail: string
  context?: Record<string, unknown>
  file: string
  // 新增字段
  appVersion: string        // package.json version
  electronVersion: string   // process.versions.electron
  nodeVersion: string       // process.versions.node
  platform: string          // process.platform
  osVersion: string         // os.release()
  pid: number               // process.pid
  uptime: number            // process.uptime()
}
```

`logFault` 内部自动填充这些字段，调用方无需关心。

**优先级**：P0（所有后续改进的基础）

#### 改进点 2：渲染进程错误捕获三件套（P0）

**目标**：渲染进程任何 JS 错误都能被记录。

**方案**：

**2.1 新建 `src/renderer/components/common/ErrorBoundary.tsx`**：
```typescript
export class ErrorBoundary extends React.Component<
  { children: React.ReactNode },
  { hasError: boolean; error?: Error }
> {
  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error }
  }
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // 调用 IPC 写入主进程 faults 日志
    window.electronAPI?.log?.reportRendererError({
      message: error.message,
      stack: error.stack,
      componentStack: info.componentStack,
      source: 'ErrorBoundary'
    })
  }
  render() {
    if (this.state.hasError) {
      return <ErrorFallback error={this.state.error!} onRetry={() => location.reload()} />
    }
    return this.props.children
  }
}
```

**2.2 在 `src/renderer/main.tsx` 包裹全局 ErrorBoundary**：
```typescript
ReactDOM.createRoot(document.getElementById('root')!).render(
  <React.StrictMode>
    <ErrorBoundary>
      <App />
    </ErrorBoundary>
  </React.StrictMode>
)
```

**2.3 新建 `src/renderer/utils/global-error-handler.ts`**：
```typescript
// 捕获资源加载失败、脚本异常
window.addEventListener('error', (event) => {
  window.electronAPI?.log?.reportRendererError({
    message: event.message,
    filename: event.filename,
    lineno: event.lineno,
    colno: event.colno,
    stack: event.error?.stack,
    source: 'window.onerror'
  })
})

// 捕获未处理 Promise 拒绝
window.addEventListener('unhandledrejection', (event) => {
  const reason = event.reason instanceof Error ? event.reason : new Error(String(event.reason))
  window.electronAPI?.log?.reportRendererError({
    message: reason.message,
    stack: reason.stack,
    source: 'unhandledrejection'
  })
})
```

**2.4 主进程新增 IPC 处理器**（`src/main/index.ts`）：
```typescript
ipcMain.handle('log:reportRendererError', (_event, payload) => {
  const err = new Error(payload.message)
  err.stack = payload.stack
  logFault('rendererError', err, {
    source: payload.source,
    componentStack: payload.componentStack,
    location: payload.filename ? `${payload.filename}:${payload.lineno}:${payload.colno}` : undefined
  }).catch(() => {})
})
```

**优先级**：P0（渲染进程错误盲区是最大问题）

#### 改进点 3：IPC 错误统一记录（P1）

**目标**：关键 IPC handler 失败时记录到 faults。

**方案**：新建 `src/main/utils/ipc-wrapper.ts`：

```typescript
export function wrapIpcHandler<T extends (...args: any[]) => any>(
  channel: string,
  handler: T
): T {
  return (async (...args: Parameters<T>) => {
    try {
      return await handler(...args)
    } catch (err) {
      logFault('ipcError', err instanceof Error ? err : new Error(String(err)), {
        channel,
        args: args.slice(1).map(a => typeof a === 'string' ? a.slice(0, 200) : typeof a)
      }).catch(() => {})
      throw err
    }
  }) as T
}
```

在关键 IPC handler（文件操作、扫描、视频处理、数据库写入）外包一层。

**优先级**：P1

#### 改进点 4：错误上下文日志缓冲（P1）

**目标**：故障发生时附带最近 50 条主日志。

**方案**：`logger.ts` 新增环形缓冲区：

```typescript
const RECENT_LOG_BUFFER_SIZE = 50
const recentLogs: Array<{ timestamp: string; level: string; message: string }> = []

function pushToRecentBuffer(level: string, message: string) {
  recentLogs.push({ timestamp: new Date().toISOString(), level, message })
  if (recentLogs.length > RECENT_LOG_BUFFER_SIZE) {
    recentLogs.shift()
  }
}

export function getRecentLogs() {
  return [...recentLogs]
}
```

`logFault` 调用时自动附加 `recentLogs` 到 context：

```typescript
context.recentLogs = getRecentLogs()
```

**优先级**：P1

#### 改进点 5：listFaults 性能优化（P2）

**目标**：万条故障列表不卡 UI。

**方案**：
- 新增 `listFaultsPaged(offset, limit, filter?)` 支持分页
- 内存缓存 fault 列表，文件 mtime 变化时失效
- `getStats` 不再调用 `listFaults`，改为独立计数

**优先级**：P2

#### 改进点 6：exportLogsAsZip 跨平台 + 完整诊断包（P1）

**目标**：一键导出完整诊断包，跨平台支持。

**方案**：
- 替换 PowerShell 为 `archiver` npm 包（跨平台 zip）
- 导出内容扩展：
  - `logs/` 目录（所有 main-*.log + faults-*.jsonl）
  - `crashes/` 目录（所有 dump 文件）
  - `startup-errors.log`
  - `system-info.json`（自动生成：版本、系统、硬件、进程列表）
  - `recent-operations.json`（最近 20 条用户操作历史）

**诊断包结构**：
```

├── logs/
│   ├── main-2026-07-05.log
│   └── faults-2026-07-05.jsonl
├── crashes/
│   └── xxx-renderer.dmp
├── startup-errors.log
├── system-info.json
└── recent-operations.json
```

**优先级**：P1

#### 改进点 7：容量管理增强（P2）

**目标**：避免日志占用过多空间。

**方案**：
- 总上限从 5GB 调整为 2GB（桌面应用足够）
- 新增时间维度过期：超过 90 天的日志自动清理
- startup-errors.log 纳入 logs 目录管理（移动到 `logs/startup-errors.log`）
- 单日 faults 文件大小上限 50MB，超限分片为 `faults-YYYY-MM-DD-1.jsonl`

**优先级**：P2

**方案**：新建 `src/renderer/components/common/ErrorFallback.tsx`：

**优先级**：P0

#### 改进点 3：主进程运行时错误弹窗（P1）

**目标**：主进程 `uncaughtException` 时通知用户。

**方案**：修改 `src/main/index.ts` 的 `process.on('uncaughtException')`：

```typescript
process.on('uncaughtException', (err) => {
  logStartupError('uncaughtException', err)
  try { logFault('uncaughtException', err, { source: 'process.uncaughtException' }).catch(() => {}) } catch {}

  // 首次错误弹窗提示，避免重复弹窗
  if (!hasShownFatalErrorDialog) {
    hasShownFatalErrorDialog = true
    dialog.showErrorBox(
      '应用遇到错误',
      `应用遇到未预期的错误：\n\n${err.message}\n\n` +
      `错误详情已记录到日志。建议：\n` +
      `1. 保存当前工作后重启应用\n` +
      `2. 如反复出现，请导出诊断包反馈给开发者\n\n` +
      `日志路径：${getLogDirectory()}`
    )
  }
})
```

**注意**：避免高频弹窗，使用 `hasShownFatalErrorDialog` 标志限制仅弹一次。

**优先级**：P1

#### 改进点 4：渲染进程崩溃恢复（P1）

**目标**：渲染进程崩溃后自动恢复，不白屏。

**方案**：修改 `createMainWindow` 中的 `render-process-gone` 处理：

```typescript
this.mainWindow.webContents.on('render-process-gone', async (_event, details) => {
  const err = new Error(`Renderer process gone: ${details.reason}`)
  logFault('rendererCrash', err, {
    reason: details.reason,
    exitCode: details.exitCode,
    source: 'webContents.render-process-gone',
    recentLogs: getRecentLogs()
  }).catch(() => {})

  // 询问用户是否重新加载
  const result = await dialog.showMessageBox(this.mainWindow!, {
    type: 'error',
    title: '页面崩溃',
    message: '页面已崩溃，需要重新加载',
    detail: `崩溃原因：${details.reason}\n退出代码：${details.exitCode}\n\n错误已记录到日志。`,
    buttons: ['重新加载', '关闭应用'],
    defaultId: 0,
    cancelId: 1
  })

  if (result.response === 0) {
    this.mainWindow?.reload()
  } else {
    app.quit()
  }
})
```

**优先级**：P1

#### 改进点 7：单实例锁冲突处理增强（P0）

**目标**：gotLock=false 时提供"一键清理僵尸进程"选项。

**方案**：修改 `gotLock = false` 分支：

```typescript
if (!gotLock) {
  const result = await dialog.showMessageBox({
    type: 'warning',
    title: '已有实例运行',
    message: '应用已在运行中',
    detail: '可能是之前的进程异常未退出。\n点击"清理并重启"自动结束残留进程。',
    buttons: ['清理并重启', '手动处理', '退出'],
    defaultId: 0,
    cancelId: 2
  })

  if (result.response === 0) {
    // 通过 taskkill 结束所有同名进程
    const { exec } = require('child_process')
    exec(`taskkill /F /IM "${process.env.APPNAME || '无限暖暖相册管理工具'}.exe"`, () => {
      app.relaunch()
      app.quit()
    })
  } else if (result.response === 1) {
    // 打开任务管理器
    require('child_process').exec('taskmgr')
    app.quit()
  } else {
    app.quit()
  }
  return
}
```

**优先级**：P0（直接解决本次问题）

### 4.4 进程退出诊断改进

#### 改进点 1：退出诊断日志（P0）

**目标**：进程退出时记录完整资源状态，便于诊断"不退出"问题。

**方案**：在 `performCleanup()` 中添加诊断日志：

```typescript
private async performCleanup(): Promise<void> {
  const diagnosis = {
    timestamp: new Date().toISOString(),
    pid: process.pid,
    uptime: process.uptime(),
    activeHandles: (process as any)._getActiveHandles().map((h: any) => h.constructor.name),
    activeRequests: (process as any)._getActiveRequests().map((r: any) => r.constructor.name),
    childProcessCount: activeProcesses.size,
    ffmpegCommandCount: killableCommands.size,
    timerCount: this.startupTimers.length
  }

  logStartupError('exit-diagnosis', new Error(JSON.stringify(diagnosis, null, 2)))

  // ... 原有清理逻辑
}
```

**记录内容**：
- 活跃句柄列表（HTTP server、socket、pipe 等）
- 活跃请求列表
- 子进程数量
- ffmpeg 命令数量
- 定时器数量
- 进程运行时长

**优先级**：P0（直接解决本次"进程不退出"诊断盲区）

#### 改进点 2：退出超时强制 kill（P1）

**目标**：`app.exit(0)` 之前给清理逻辑 2 秒时间，超时强制退出。

**方案**：

```typescript
app.on('before-quit', (event) => {
  if (this.isCleaningUp) return
  this.isCleaningUp = true
  event.preventDefault()

  // 清理逻辑最多给 2 秒
  const forceExitTimer = setTimeout(() => {
    logStartupError('force-exit-timeout', new Error('清理超时，强制退出'))
    app.exit(1)
  }, 2000)

  this.performCleanup().finally(() => {
    clearTimeout(forceExitTimer)
    app.exit(0)
  })
})
```

**优先级**：P1

## 七、风险与应对

### 7.1 技术风险

| 风险 | 影响 | 应对 |
|---|---|---|
| ErrorBoundary 无法捕获事件处理器中的错误 | 部分 JS 错误仍会丢失 | 配合 window.onerror + unhandledrejection 双重捕获 |

| 全局 Toast Provider 性能影响 | 所有页面重渲染 | 使用 Context + memo 优化 |

### 7.2 兼容性风险

| 风险 | 应对 |
|---|---|
| FaultRecord 字段扩展后旧日志不兼容 | listFaults 兼容处理：缺失字段显示为"未知" |

| 全局 Toast Provider 与设置页 Toast 冲突 | 重命名 + 渐进式迁移，设置页改用全局 Provider |

### 7.3 用户体验风险

| 风险 | 应对 |
|---|---|
| 运行时错误弹窗频繁打扰用户 | hasShownFatalErrorDialog 标志限制仅弹一次 |
| 错误兜底页破坏原有页面美观 | ErrorFallback 设计遵循 Fluent Design 风格 |
| 诊断包包含敏感信息 | 导出前自动脱敏（移除用户名路径） |
