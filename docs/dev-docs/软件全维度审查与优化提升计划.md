# 软件全维度审查与优化提升计划

> **项目名称**：无限暖暖相册管理工具（wxnn-photo-manager）
> **当前版本**：v2.0.0
> **技术栈**：Electron 28 + React 18 + TypeScript 5 + Vite 5 + better-sqlite3 + sharp + fluent-ffmpeg + Zustand + Tailwind CSS
> **审查日期**：2026-06-23
> **审查范围**：`src/main`（主进程）、`src/renderer`（渲染进程）、构建配置、数据库设计、IPC 通道、UI 组件、业务流程、全量代码
> **审查方法**：静态代码审查 + 架构分析 + 数据流追踪 + 安全建模 + 性能推演

---

## 一、项目审查概述

### 1.1 项目定位

本项目是一款面向《无限暖暖》游戏玩家的桌面相册管理工具，核心能力涵盖：游戏目录自动发现、图片/视频扫描索引、多视图浏览（网格/瀑布流/列表/时间线）、图片编辑（调整/滤镜/LUT/色调曲线/水印）、文件操作（复制/移动/重命名/删除/导出）、分类与标签管理、视频播放与截帧。

### 1.2 架构现状

项目采用 Electron 主进程-渲染进程分离架构，主进程负责原生能力（文件系统、数据库、ffmpeg、sharp），渲染进程负责 UI 与交互，通过 IPC 通信。整体技术选型合理，但**架构落地存在严重偏差**。

### 1.3 审查结论摘要

经全维度审查，本项目存在 **1 项最高优先级的根因性问题**——"重构未完成导致双套实现并存"，该问题衍生出大量重复代码、安全漏洞与性能隐患。围绕此根因，共发现：

| 维度 | 致命 | 严重 | 一般 | 优化 | 合计 |
|------|------|------|------|------|------|
| 底层架构 | 4 | 9 | 8 | 4 | 25 |
| UI 设计 | 3 | 11 | 16 | 12 | 42 |
| 功能设计 | 2 | 10 | 11 | 8 | 31 |
| 全量代码 | 5 | 10 | 13 | 15 | 43 |
| **去重合计** | **9** | **28** | **38** | **31** | **106** |

> 注：部分问题跨维度重复，去重后实际独立问题约 80 项。下文按维度呈现并标注跨维度关联。

### 1.4 整体风险评级

- **安全风险**：高（存在路径遍历、任意文件写入、协议白名单过宽等多个可利用漏洞）
- **数据完整性风险**：高（标签双写无事务、无退出清理、无单实例锁）
- **性能风险**：高（万级文件场景下多处必现卡顿/OOM）
- **可维护性风险**：极高（双套实现并存，计划文档与实际代码严重不符）
- **可用性风险**：中（扫描进度不可见、部分视图必崩、保存覆盖原图）

## 二、分维度问题清单

### 2.1 底层架构维度

#### 致命问题（Fatal）

**A-F1. 重构层为死代码，双套实现并存**
- **证据**：`src/main/index.ts` 第 42-1194 行 `Application` 类（1152 行上帝类）内联了全部 IPC 路由、协议处理、数据库维护逻辑；而 `src/main/ipc/ipc-router.ts`（687 行）、`src/main/repositories/*.ts`（5 个文件）、`src/main/services/database-maintenance-service.ts`、`src/main/services/theme-manager.ts`、`src/main/protocol/media-protocol-handler.ts` 已完整编写但**从未被 `index.ts` 导入**（已通过搜索验证：`index.ts` 中无 `IpcRouter`/`MediaProtocolHandler`/`DatabaseMaintenanceService` 等引用）。
- **衍生影响**：约 1500+ 行重构代码完全无效；`ipc-router.ts` 缺少 `ui-theme:get/set` 通道，已落后于 `index.ts`；维护者修改 Repository 层不会生效；计划文档 `架构优化与功能升级计划.md` 虚假标记 M7 重构"已完成"。

**A-F2. 数据库迁移无版本管理**
- **证据**：`src/main/database/connection.ts` 第 27-174 行 `runMigrations()` 使用 `CREATE TABLE IF NOT EXISTS` + try/catch `ALTER TABLE`，无 `PRAGMA user_version` 或独立 `schema_version` 表。第 176-209 行 `migrateSceneCategory()` 每次启动都执行 `WHERE scene_category IS NULL OR scene_category = 'other'` 全表扫描回填，万级记录库每次启动产生不必要 CPU/IO 开销。

**A-F3. 初始化顺序错误导致协议早期失效**
- **证据**：`src/main/index.ts` 第 62-96 行 `initialize()` 中，`registerMediaProtocol()`（第 64 行）在 `dbManager.initialize()`（第 66 行）之前执行。而 `isMediaPathAllowed()`（第 956 行）依赖数据库，此时返回 `null` 导致第 958 行返回 `false`，数据库就绪前的所有 `media://` 请求返回 403。

**A-F4. 无单实例锁与退出清理**
- **证据**：已验证 `src/main` 全目录无 `requestSingleInstanceLock`、`before-quit`、`will-quit` 调用。多实例并发操作同一 SQLite 数据库存在竞争；`DatabaseManager.close()`（`connection.ts` 第 241-246 行）从未被调用，WAL 文件未 checkpoint，异常退出可能丢数据。

#### 严重问题（Severe）

**A-S1. `media://` 协议白名单每次请求全表扫描（N+1）**
- **证据**：`src/main/index.ts` 第 956-993 行 `isMediaPathAllowed()`，每个 `media://` 请求执行：第 970-972 行 `SELECT 1 FROM media_files WHERE file_path = ? OR thumbnail = ?`（`thumbnail` 列无索引，全表扫描）；第 976-978 行 `SELECT DISTINCT source_path FROM media_files`（全表扫描）；第 979-986 行 JS 遍历所有 `source_path` 做前缀匹配。加载 100 张图的相册页触发 100 次该检查。

**A-S2. 协议白名单过宽导致任意文件读取**
- **证据**：`src/main/index.ts` 第 982 行 `if (normalizedRequest.startsWith(resolvedSource + path.sep)) return true`。若用户曾扫描 `D:\` 这样的宽目录，则 `media://` 协议允许访问整个 `D:\` 盘下任意文件，无法真正限制到已索引文件。

**A-S3. 关键查询路径缺少索引**
- **证据**：`connection.ts` 第 31-59 行 `media_files` 表：`thumbnail` 列无索引（被 A-S1 查询使用）、`source_path` 列无索引（被 A-S1 查询使用）；第 142-150 行 `media_tags` 表 `tag_id` 列无独立索引；第 121-128 行 `edit_history.media_id` 无索引；第 69 行 `categories.parent_id` 无索引。

**A-S4. 标签双存储无事务保护**
- **证据**：`connection.ts` 第 46 行 `tags TEXT` JSON 字段与第 142-150 行 `media_tags` 关联表并存。`index.ts` 第 569-595 行 `tag:add` 先写 `media_tags`（第 581 行）再写 `media_files.tags`（第 587 行），两步不在同一事务；第 655-665 行 `media:updateTags` 只更新 JSON 列，完全不同步关联表。

**A-S5. 扫描器递归无深度限制与符号链接环检测**
- **证据**：`src/main/scanner/index.ts` 第 217-275 行 `scan()` 递归调用（第 235 行）无深度限制（对比 `searchBySignature` 第 177 行有 `depth > 3` 限制）。含符号链接环的目录会导致无限递归栈溢出。

**A-S6. 扫描全量收集到内存后写入**
- **证据**：`scanner/index.ts` 第 203 行 `const files: MediaFile[] = []`，第 277 行 `await scan(rootPath)` 收集全部元数据到内存数组，第 329 行 `saveToDatabase(files, ...)` 一次性写入。万级文件峰值内存占用高，应流式分批写入。

**A-S7. 缩略图缓存无大小限制与 LRU 淘汰**
- **证据**：`src/main/thumbnail/generator.ts` 全文无缓存大小限制、无过期清理、无 LRU。`index.ts` 第 1084-1119 行 `cleanupOrphanThumbnails` 仅清理"数据库无引用的"，不清理"长期未访问的"。长期使用后缓存目录无限增长。

**A-S8. 缩略图缓存键基于路径+mtime，文件移动即失效**
- **证据**：`generator.ts` 第 103-105 行 `getFileHash()` 哈希输入为 `${filePath}:${mtime}`。文件移动后路径变化，即使内容不变也重新生成缩略图，旧缩略图变孤儿。

**A-S9. IPC 参数几乎无校验**
- **证据**：`index.ts` 全部 IPC 处理器直接信任渲染进程参数。第 159 行 `file:delete` 路径数组无校验；第 287 行 `editor:save` 的 `filePath` 任意可覆盖系统文件；第 442 行 `watermark:apply` 的 `config: unknown` 直接 cast；第 619 行 `media:updateRating` 的 `rating` 无 0-5 范围校验；第 141 行 `scanner:start` 的 `options.path` 任意可扫描系统目录。

#### 一般问题（General）

**A-G1. `media:list` 全表扫描无分页**：`index.ts` 第 703-740 行 `SELECT ... FROM media_files ORDER BY created_at DESC` 无 `LIMIT/OFFSET`，万级文件单次 IPC 传输数十 MB JSON。

**A-G2. 图片协议响应全量读入内存**：`index.ts` 第 941 行 `await fs.promises.readFile(normalizedPath)`，非视频文件（含大图）整体读入内存。视频已用流式（第 932 行），图片未用。

**A-G3. 启动时全量数据库扫描阻塞就绪**：`index.ts` 第 71 行 `await this.cleanupAndRepairDatabase()` 在 `setupIPC()` 之前同步执行，万级记录库启动时间显著增加，且渲染进程无法获得进度反馈。

**A-G4. 自动扫描用 setTimeout 魔数延迟**：`index.ts` 第 80-83 行 `setTimeout(..., 1500)`，应改为监听 `ready-to-show` 或 `did-finish-load` 事件。

**A-G5. 缺少关键 PRAGMA**：`connection.ts` 第 22-23 行仅设置 `journal_mode=WAL` 和 `foreign_keys=ON`，未设置 `synchronous=NORMAL`（WAL 下默认 FULL 性能差）、`busy_timeout`（并发时抛 SQLITE_BUSY）、`cache_size`。

**A-G6. `query/execute` 暴露原始 SQL 接口**：`connection.ts` 第 211-219 行接受任意 SQL 字符串，潜在 SQL 注入入口。

**A-G7. `close()` 未做 WAL checkpoint**：`connection.ts` 第 241-246 行直接关闭，未执行 `PRAGMA wal_checkpoint(TRUNCATE)`。

**A-G8. `media_files.category_id` 无外键约束**：`connection.ts` 第 31-59 行 `media_files` 表的 `category_id` 字段无 `REFERENCES categories(id)`，虽第 23 行启用了 `foreign_keys`，对该字段无效。删除分类后 `category_id` 成为悬空引用。

#### 优化项（Optimization）

**A-O1. 视频扩展名列表 4 处重复硬编码**：`generator.ts` 第 57 行、`scanner/index.ts` 第 39 行、`index.ts` 第 928 行、`media-protocol-handler.ts` 第 9 行，应提取共享常量。

**A-O2. 动态 import p-limit 在热路径**：`scanner/index.ts` 第 280 行、`index.ts` 第 1057/1144 行在热路径动态 import，应顶部静态 import。

**A-O3. 缩略图参数硬编码不可配置**：`generator.ts` 第 12-14 行 `maxWidth=320, maxHeight=320, quality=85`。

**A-O4. 缩略图无子目录分桶**：`generator.ts` 第 47 行所有缩略图平铺单目录，文件系统对单目录文件数有性能上限。

### 2.2 UI 设计维度

#### 致命问题（Fatal）

**U-F1. ToneCurve 画布使用 CSS 变量导致网格/对角线/文字不渲染**
- **证据**：`src/renderer/components/editor/ToneCurve.tsx` 第 28 行 `ctx.strokeStyle = 'var(--divider)'`、第 43 行 `'var(--text-tertiary)'`、第 89 行 `ctx.fillStyle = 'var(--text-secondary)'`。Canvas 2D API 不解析 CSS 变量，该值非法被静默忽略，回退到默认黑色或上次有效值。
- **影响**：曲线编辑器参考网格完全失效，用户无法判断色调映射基准。

**U-F2. MasonryView 无虚拟滚动且硬编码列数**
- **证据**：`src/renderer/components/gallery/MasonryView.tsx` 第 22 行 `columnCount: 4` 硬编码、第 24 行 `files.map(...)` 一次性渲染全部文件。数千张图片时 DOM 节点爆炸导致页面卡死；小窗口下列宽过窄。

**U-F3. EditorPage 保存直接覆盖原始文件，无备份无确认**
- **证据**：`src/renderer/pages/EditorPage.tsx` 第 126-129 行 `handleSave` 直接写回 `media.file_path`，无备份、无二次确认、无版本保留。PNG 原图保存时还传 `quality: 95`（PNG 无损，quality 无意义但可能触发转码）。
- **影响**：误操作保存即丢失原图，不可恢复。

#### 严重问题（Severe）

**U-S1. ScanProgress 组件从未被挂载，扫描进度对用户不可见**
- **证据**：已验证 `ScanProgress.tsx` 仅在 `ScanButton.tsx`（写入 store）和 `mediaStore.ts` 中被引用，`GalleryPage.tsx`、`AppShell.tsx`、`App.tsx` 均未 import 该组件。扫描过程中用户完全看不到进度、已找到文件数、当前路径。

**U-S2. PageTransition 使用 key 强制重挂载，导致页面状态丢失与重复请求**
- **证据**：`src/renderer/components/layout/PageTransition.tsx` 第 7 行 `<div key={currentView}>`。每次视图切换完全卸载/重挂载页面，`GalleryPage` 的 `useEffect` 重新执行 `loadMediaFromDatabase()`，从详情页返回图库时丢失滚动位置与组件内部状态。

**U-S3. TimelineView 虚拟化仅在分组层级，组内全量渲染**
- **证据**：`src/renderer/components/gallery/TimelineView.tsx` 第 81-83 行仅过滤可见分组，第 119 行 `group.files.map(...)` 每个可见分组内所有文件全量渲染。单日 500 张照片时即使仅 20 张在视口也渲染全部 500 个 DOM 节点。

**U-S4. VirtualImageGrid 选中态查找 O(n*m)**
- **证据**：`src/renderer/components/gallery/VirtualImageGrid.tsx` 第 132 行 `selectedIds.includes(file.id)` 每个可见项线性扫描数组。全选 5000 张时每帧 30 项 × 5000 次 = 15 万次比较，应改用 `Set`。

**U-S5. ContextMenu 无边界检测且 useEffect 依赖未记忆化**
- **证据**：`src/renderer/components/common/ContextMenu.tsx` 第 83-87 行直接用鼠标坐标定位，未检测菜单尺寸是否超出视口；第 25 行 `focusableIndices` 每次渲染新数组，第 75 行 effect 依赖它导致每帧重渲染、键盘导航焦点反复跳回第一项。

**U-S6. useEditHistory 的 historyIndexRef 与 pushHistory 竞态**
- **证据**：`src/renderer/hooks/useEditHistory.ts` 第 17-22 行 ref 在 useEffect 中更新（render 后执行），第 33-44 行 `pushHistory` 读取 ref。同一 render 周期内连续调用多次 `pushHistory`（如批量调滑块）时 ref 还是旧值，`slice(0, historyIndexRef.current + 1)` 错误截断历史栈。

**U-S7. WatermarkPanel 滑块缺少 onCommit，水印调整无法撤销**
- **证据**：`src/renderer/components/editor/WatermarkPanel.tsx` 第 336-338 行等所有水印滑块未传 `onCommit={pushHistory}`，水印参数调整不入历史栈，Ctrl+Z 失效。

**U-S8. 多个模态框无焦点陷阱与 ARIA 属性**
- **证据**：`TagManager.tsx` 第 30-34 行、`WatermarkDialog.tsx` 第 28-29 行、`RenameDialog.tsx` 第 50 行、`CategoriesPage.tsx` 第 523-524 行均无 `role="dialog"`/`aria-modal="true"`/焦点陷阱/Esc 处理。键盘用户可 Tab 到模态框背后元素。

**U-S9. useTheme 与 themeStore 是两套互不关联的主题系统**
- **证据**：`hooks/useTheme.ts`（管理 light/dark/system，操作 `.dark` 类）与 `stores/themeStore.ts`（管理 default/soft-pink-luxury，操作主题类名）并存。`globals.css` 和 `soft-pink-luxury.css` 无任何 `.dark` 样式定义，`useTheme` 是死代码。暗色模式完全不可用。

**U-S10. FullscreenViewer 无触摸/滑动导航支持**
- **证据**：`src/renderer/components/gallery/FullscreenViewer.tsx` 第 29-41 行仅键盘事件、第 183-202 行仅按钮导航，无触摸滑动切换。触屏设备全屏浏览体验缺失。

**U-S11. SettingsPage 使用原生 alert()，与应用风格不一致**
- **证据**：`src/renderer/pages/SettingsPage.tsx` 第 55 行 `alert('本地数据已清除')`、第 57 行 `alert(...)`。原生 alert 阻塞主进程、样式无法定制、与毛玻璃 UI 风格违和。其他页面均用 `Toast` 组件。

#### 一般问题（General）

**U-G1. DetailPage ZoomableImage 拖拽后释放触发 click 误打开全屏**：`DetailPage.tsx` 第 311-316 行启动拖拽、第 345 行 `onClick` 仍触发 `openFullscreen`，缺少拖拽距离阈值。

**U-G2. EditorPage "取消"按钮误导用户**：`EditorPage.tsx` 第 297 行 `onExit={() => navigateTo('detail')}`，按钮文字"取消"但实际不丢弃编辑参数，再次进入编辑器修改仍在。

**U-G3. Toast 无入场/退场动画、无堆叠、无手动关闭**：`Toast.tsx` 第 9-22 行瞬间出现/消失，同时只能显示一条，新消息覆盖旧消息计时器，重要错误提示可能被成功提示覆盖。

**U-G4. ConfirmDialog danger 三元表达式两分支相同**：`ConfirmDialog.tsx` 第 85 行 `confirmVariant === 'danger' ? 'btn-primary' : 'btn-primary'`，无意义代码，缺少 `.btn-danger` 样式类。

**U-G5. SliderControl 缺少双击复位与单位显示**：`SliderControl.tsx` 第 13-42 行数值仅显示裸数字（色温应显示 K），无双击复位，键盘方向键调整时不触发 `onCommit`。

**U-G6. ScanProgress 进度条上限硬编码 10000**：`ScanProgress.tsx` 第 33 行 `scanProgress.scanned / 10000`，超 1 万提前满格，少于 1 万永远不满。

**U-G7. VideoPlayer 自定义按钮与原生 controls 位置冲突**：`VideoPlayer.tsx` 第 150 行 `bottom-14` 与原生控制栏视觉重叠，`opacity-0 group-hover:opacity-100` 在触屏无法触发。

**U-G8. VideoPlayer 进度持久化无清理机制**：`VideoPlayer.tsx` 第 53-60 行存入 localStorage，从不清理已删除视频记录，长期使用无限增长。

**U-G9. TagManager 不检查重复标签**：`TagManager.tsx` 第 17-22 行 `handleAddTag` 直接 `[...media.tags, newTag]`，不查重。

**U-G10. CategoriesPage 内联 Toast 与全局 Toast 组件不一致**：`CategoriesPage.tsx` 第 501-511 行自行渲染 fixed div，未使用 `<Toast>` 组件，样式有差异。

**U-G11. AppShell 状态栏仅在 gallery 视图显示**：`AppShell.tsx` 第 20-28 行，详情页/编辑器/分类页不显示任何状态信息。

**U-G12. Sidebar 未使用 viewStack/goBack，无返回导航**：`Sidebar.tsx` 第 41-53 行仅正向导航，`uiStore` 的 `goBack`（第 95-104 行）从未被调用。

  - **方案 B**：删除未使用的重构模块（`ipc-router.ts`、`repositories/`、`services/database-maintenance-service.ts`、`services/theme-manager.ts`、`protocol/media-protocol-handler.ts`），承认 `index.ts` 为唯一实现，并修正计划文档虚假标记。

**预期收益**：消除 1500+ 行死代码与全部重复代码；统一架构入口；恢复计划文档可信度；避免双重 IPC 注册崩溃风险。

#### 方案 A-2：数据库版本化迁移与索引补齐（解决 A-F2 / A-S3 / A-G5 / A-G7 / A-G8）

**改动方向**：
- 引入 `PRAGMA user_version` 版本化迁移机制，每次迁移前读取版本号、按版本号条件执行、执行后递增版本号，避免每次启动重复执行 `migrateSceneCategory` 全表扫描。
- 补齐索引：`media_files.thumbnail`、`media_files.source_path`、`media_tags.tag_id`、`edit_history.media_id`、`categories.parent_id`。
- 补齐 PRAGMA：`synchronous=NORMAL`、`busy_timeout=5000`、`cache_size=-64000`（64MB）。
- `media_files.category_id` 添加 `REFERENCES categories(id) ON DELETE SET NULL` 外键约束。
- `close()` 执行 `PRAGMA wal_checkpoint(TRUNCATE)` 后再关闭。

**预期收益**：`media://` 协议白名单查询从全表扫描降为索引查找（万级记录从 ~50ms 降至 ~0.5ms）；启动迁移开销从 O(n) 降为 O(1)；数据完整性保障。

#### 方案 A-3：协议安全与性能重构（解决 A-S1 / A-S2 / C-F5 / A-G2）

**改动方向**：
- `isMediaPathAllowed` 改为内存缓存模式：启动时加载一次 `source_path` 集合到内存 `Set`，请求时内存前缀匹配，避免每次请求全表扫描。
- 收紧白名单：`source_path` 前缀匹配仅允许访问"该 source_path 下且文件扩展名在媒体类型列表内"的文件，而非整个目录树；或改为仅允许访问数据库中已索引的 `file_path` 精确匹配 + 缩略图缓存目录。
- 图片协议响应改用 `fs.createReadStream` 流式返回（与视频一致），避免大图全量读入内存。

**预期收益**：万级图库浏览时主进程 CPU 飙升问题消除；消除任意文件读取漏洞；大图加载内存峰值降低 80%+。

#### 方案 A-4：初始化顺序与生命周期修复（解决 A-F3 / A-F4 / A-G3 / A-G4）

**改动方向**：
- 调整 `initialize()` 顺序：`dbManager.initialize()` → `registerMediaProtocol()` → `createMainWindow()` → `cleanupAndRepairDatabase()`（改为后台异步，不阻塞 `setupIPC`）→ `setupIPC()`。

- 添加 `app.requestSingleInstanceLock()`，第二实例聚焦已有窗口并退出。
- 添加 `app.on('before-quit')` 处理器：停止扫描器、关闭数据库连接、清理临时文件。

**预期收益**：消除协议早期失效；启动时间从"阻塞全量扫描"降为"后台异步修复"；多实例竞争消除；异常退出数据丢失风险消除。

#### 方案 A-5：IPC 安全与统一返回值（解决 A-S9 / C-F3 / C-F4 / C-G5）

**改动方向**：
- 引入 IPC 参数校验中间件：路径类参数校验白名单（仅允许已索引媒体或用户选择目录）、数值类参数校验范围（如 `rating` 限 0-5）、数组类参数校验长度上限。
- `file:rename` 的 `invalidChars` 正则补充 `/` 与 `..`，防止路径遍历。
- `editor:save`/`file:copy`/`file:move`/`file:export` 添加源/目标路径白名单校验。
- 统一所有 IPC 返回值为 `IpcResult<T>`（`{ success: boolean; data?: T; message?: string }`），移除裸数组/裸值返回。

**预期收益**：消除路径遍历、任意文件写入等安全漏洞；IPC 契约一致，渲染层类型推断更安全。

#### 方案 A-6：标签存储统一与事务保护（解决 A-S4 / C-F2）

**改动方向**：
- 以 `media_tags` 关联表为唯一真源，`media_files.tags` JSON 列改为查询时动态聚合的视图（或完全移除该列，查询时 JOIN）。
- 若保留双写，所有标签操作包裹在 `db.transaction()` 内，保证原子性。
- `media:updateTags` 必须同步更新 `media_tags` 表（先 DELETE 旧关联再 INSERT 新关联）。

**预期收益**：消除数据不一致风险；标签查询走关联表索引而非 JSON 解析。

#### 方案 A-7：扫描器与缩略图优化（解决 A-S5 / A-S6 / A-S7 / A-S8 / C-S3）

**改动方向**：
- `scan()` 添加深度限制参数（默认 10）与符号链接环检测（维护已访问 inode `Set`）。
- 扫描结果流式分批写入：每 500 条 `INSERT` 一次，降低峰值内存。
- 缩略图缓存引入 LRU 淘汰：记录每个缩略图的最后访问时间，总大小超阈值（默认 2GB）时淘汰最久未访问的。
- 缩略图缓存键改为内容 hash（文件前 1MB 的 sha256），文件移动后缓存仍有效。
- `generateThumbnailsForUnprocessed` 添加运行标志互斥锁，防止并发重复生成。

**预期收益**：消除递归栈溢出风险；万级文件扫描峰值内存降低 70%；缩略图缓存可控；并发竞态消除。
