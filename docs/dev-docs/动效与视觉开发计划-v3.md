## 一、为什么需要这份新计划

### 1.1 v2.0 计划的失败教训

v2.0 计划（P0-P5 + 清理阶段）执行后产生三类负面结果：

| 问题 | 根因 | 后果 |
|---|---|---|
| 首次打开后无法重新打开 | P5-1 在 `app.whenReady()` 之前调用 `app.getPath('userData')`，打包后写入异常配置文件 | 严重可用性 bug，用户必须重装 |
| 页面切换"一闪"感 | View Transitions API 整页快照交叉淡入，新旧页面半透明叠加时背景透出 | 体验倒退，不如 2.2.6 原版 |
| 毛玻璃效果无感 | body 是 `#F5F5F5` 纯色，`backdrop-filter` 模糊纯色无可见效果 | 改动无效，徒增复杂度 |

### 1.2 检查机制失灵的原因

v2.0 执行期间的"全面检查"只验证了「任务是否完成」「tsc 是否通过」，没有验证：
- 打包后 exe 是否能正常启动和重启
- 改动是否真正达成预期视觉效果
- 改动是否引入运行时风险

**本计划强制要求**：每个阶段完成后必须打包 exe 实测，验收清单见各阶段末尾。

### 1.3 本计划的定位

本计划是**保守增强**：在已验证稳定的基础上，增加 OriginOS 6 风格的高斯模糊过渡效果和毛玻璃分层应用。核心原则：
1. **不引入 View Transitions API**（已证明有闪烁问题）
2. **不动主进程启动流程**（P5-1 的教训：ready 前不碰 `app.getPath`）
3. **不破坏 keep-alive 机制**（保留滚动位置与组件状态）
4. **每个改动必须打包 exe 实测**
5. **可回滚**：每个阶段独立，失败则回滚该阶段，不影响其他阶段
6. **每阶段有降级方案**：效果不达预期或性能不达标时，按降级链调整

## 二、当前项目真实现状（2026-07-11 调研）

### 2.1 body 背景与毛玻璃

- **默认主题**：`body { background: var(--bg-primary); }`，`--bg-primary: #F5F5F5`，纯色无渐变
- **soft-pink-luxury 主题**：body 有浅粉渐变 `linear-gradient(180deg, #fdf9fb 0%, #f7f0f4 100%)`
- **毛玻璃可见性**：默认主题下，title-bar / sidebar / glass-card 下方是纯色，`backdrop-filter` 几乎不可见；仅在叠加于图片/视频之上的元素（FullscreenViewer 控制条、SlideshowPlayer 控制条、Toast）才真正显现
- **backdrop-filter 使用点**：CSS 中 7 处声明 + TSX 内联 9 处 = 16 个唯一使用点

### 2.2 页面切换机制（关键技术细节）

**App.tsx keep-alive 实现**（第 134-146 行）：
```tsx
{ALL_VIEWS.map((view) => {
  if (!visitedViews.has(view)) return null
  const isActive = view === currentView
  return (
    <div
      key={view}
      className={isActive ? 'h-full page-enter' : 'h-full'}
      style={{ display: isActive ? 'block' : 'none' }}
    >
      {renderView(view)}
    </div>
  )
})}
```

**动画触发机制分析**：
- React 复用 DOM 节点（key=view 不变），不重新挂载
- 切换时：className 从 `'h-full'` 变为 `'h-full page-enter'`，display 从 `none` 变为 `block`
- Chromium 行为：元素从 `display:none` 变为 `display:block` 且有 animation 属性时，**会重新触发 CSS animation**
- **风险**：React 批量更新 DOM，className 变化和 display 变化在同一 commit 中完成，浏览器只看到最终状态。需实测确认动画是否可靠重播

**uiStore.ts 导航逻辑**（第 163-183 行）：
- `navigateTo(view)`：push 到 viewStack，设置 currentView
- `goBack()`：pop viewStack，设置 currentView
- 两者都只做 setState，无 JS 介入动画

**当前 page-enter 动画**：
```css
.page-enter {
  animation: slideInRight 300ms cubic-bezier(0.4, 0, 0.2, 1) forwards;
}
```

**soft-pink-luxury.css 覆盖**（第 265-269 行）：
```css
.soft-pink-luxury .page-enter {
  animation: slideInRight 200ms cubic-bezier(0.4, 0, 0.2, 1) forwards;
}
.soft-pink-luxury .page-exit { ... }
```

### 阶段 C：侧边栏微渐变与毛玻璃分层应用（中风险）

**用户确认方向**：采用侧边栏微渐变方案，毛玻璃效果重点用于二/三/四级界面，一级界面不强化。

#### 界面层级定义

按视图类型固定映射层级（不使用 viewStack.length，因为导航路径不固定）：

| 层级 | 对应视图 | 毛玻璃策略 | 判定依据 |
|---|---|---|---|
| 一级 | `gallery`、`categories`、`settings`、`recycle-bin`、`favorites`、`duplicates`、`launcher-cache` | 不强化，保持 `--bg-secondary: rgba(255,255,255,0.75)` | 侧边栏直接切换的主模块 |
| 二级 | `detail` | 降至 `rgba(255,255,255,0.65)`，模糊更可见 | 从一级进入的详情页 |
| 三级 | `editor` | 降至 `rgba(255,255,255,0.55)`，效果显著 | 从二级进入的编辑器 |
| 四级 | editor 内部子面板 | 降至 `rgba(255,255,255,0.5)`，层次最深 | 嵌套面板（滤镜、调色、水印等） |

#### 实现思路

**侧边栏微渐变**：
- 在 AppShell 的 sidebar 容器添加极淡的彩色渐变背景

**毛玻璃分层应用**：

  [data-view-level="2"] .glass-card { background: rgba(255, 255, 255, 0.65); }

### 阶段 D：OriginOS 6 风格高斯模糊过渡（中风险，核心需求）

**目标**：**每一次界面切换**（侧边栏切换模块、进入详情、进入编辑器、返回操作）都应用高斯模糊浮现过渡，类似 OriginOS 6 从桌面启动应用的丝滑效果。

#### 覆盖范围

本阶段修改的是全局 `.page-enter` CSS 类，所有通过 `navigateTo` / `goBack` 触发的界面切换都会应用：

| 切换场景 | 触发方式 | 覆盖情况 |
|---|---|---|
| 侧边栏切换模块 | `navigateTo('gallery'/'settings'/...)` | ✅ 覆盖 |
| 图库点击图片进入详情 | `navigateTo('detail')` | ✅ 覆盖 |
| 详情页进入编辑器 | `navigateTo('editor')` | ✅ 覆盖 |
| 返回上一级 | `goBack()` | ✅ 覆盖 |
| 全屏查看器开关 | FullscreenViewer 独立组件 | ❌ 不覆盖（已有 View Transitions API） |
| 对话框开关 | motion 组件 | ❌ 不覆盖（已有 motion 动画） |

#### 效果描述

参考 OriginOS 6 启动应用的过渡特征：
1. 新页面从**模糊状态**（`filter: blur(12px)`）渐变到**清晰**（`blur(0)`）
2. 配合轻微**缩放**（`scale(1.03)` → `scale(1)`），营造"从远处浮现"的层次感
3. 配合**淡入**（`opacity: 0` → `1`），避免突兀
4. 整体过渡时长约 350ms，缓动曲线 `cubic-bezier(0.4, 0, 0.2, 1)`
5. **无方向性**：前进和返回用相同的模糊浮现，避免方向矛盾

#### 技术方案

**纯 CSS 实现，不引入 JS 介入，不破坏 keep-alive**：

修改 `globals.css` 中的 `page-enter` 关键帧：

```css
@keyframes pageBlurIn {
  from {
    filter: blur(12px);
    transform: scale(1.03);
    opacity: 0;
  }
  to {
    filter: blur(0);
    transform: scale(1);
    opacity: 1;
  }
}

.page-enter {
  animation: pageBlurIn 350ms cubic-bezier(0.4, 0, 0.2, 1) forwards;
  will-change: filter, transform, opacity;
}
```

**同步修改 soft-pink-luxury.css**（第 265-266 行）：
```css
.soft-pink-luxury .page-enter {
  animation: pageBlurIn 350ms cubic-bezier(0.4, 0, 0.2, 1) forwards;
}
```
（soft-pink-luxury 原本用 200ms，统一改为 350ms 保持效果一致）

#### 关键技术风险与应对

**风险 1：keep-alive 下动画是否可靠重播**

App.tsx 中 React 复用 DOM 节点，切换时 className 从 `'h-full'` 变为 `'h-full page-enter'`，display 从 `none` 变为 `block`。

- **Chromium 行为**：元素从 `display:none` 变为可见且有 animation 属性时，会重新触发动画
- **验证方式**：D-7 打包实测时必须重点验证「已访问过的页面切换回来时动画是否重新播放」
- **兜底方案**：如果动画不重播，在 App.tsx 中用 `key={currentView + '-' + navigationId}` 强制重新挂载（navigationId 每次导航递增）。但这会丢失 keep-alive 状态，需权衡

**风险 2：filter:blur 与 backdrop-filter 冲突**

父元素 `filter: blur()` 会创建新的合成层和堆叠上下文，可能导致子元素的 `backdrop-filter` 在动画期间失效。

- **表现**：动画期间毛玻璃不显示，动画结束后恢复
- **验证方式**：D-7 实测时观察切换过程中 glass-card 的毛玻璃是否闪烁
- **应对方案**：
  1. 如果冲突明显，将 blur 从 `filter` 改为 `backdrop-filter`（仅模糊背景，不影响子元素）
  2. 但 `backdrop-filter` 无法做"整个页面模糊"效果，只能做元素自身背景模糊
  3. 最终如果冲突无法解决，降级为仅 `scale + opacity`（去掉 blur）

**风险 3：filter:blur(0) 仍创建合成层**

`filter: blur(0)` 虽然视觉上无模糊，但仍会创建合成层，可能持续影响 backdrop-filter。

- **应对方案**：动画结束后用 JS 移除 filter 属性。但这增加 JS 介入，违背纯 CSS 原则
- **权衡**：如果实测发现动画结束后毛玻璃异常，再考虑此方案。否则保持 `forwards` 填充模式

#### 退场页面处理

保持当前 keep-alive 逻辑，非活动页直接 `display: none`，不做退场动画。

**理由**：
- 退场动画需协调时序（等动画结束再 display:none），增加 App.tsx 复杂度
- 新页面模糊浮现已足够营造过渡感，用户视觉焦点自然转移到新内容
- 不做退场动画反而更利落，避免新旧页面动画叠加的混乱
- 降低风险，保持 keep-alive 简单性

**用户关注的"丝滑"由入场动画承担**：旧页面瞬间消失 + 新页面从模糊中浮现，整体感受是"新页面从远处清晰化"，而非"旧页面淡出"。

#### 性能考量

**风险点**：`filter: blur()` 在大尺寸元素上有 GPU 开销，可能导致首次帧卡顿。

**缓解措施**：
1. `will-change: filter, transform, opacity` 提前创建合成层（阶段 B 已添加）
2. blur 半径 12px 适中，不会过大
3. 过渡时长 350ms 足够让 GPU 完成渲染
4. Electron 28（Chromium 120）的 filter 性能已优化
5. keep-alive 下非活动页 `display:none`，不参与渲染，无额外开销

**降级方案**：如果实测发现卡顿（FPS < 50），按以下顺序降级：
1. 降级一：blur 半径从 12px 降至 6px
2. 降级二：去掉 filter: blur，仅保留 `scale(1.03) → scale(1)` + `opacity: 0 → 1`
3. 降级三：恢复原版 `slideInRight 300ms`

## 四、执行顺序与依赖关系

```
阶段 A（死代码清理）── 无依赖，可立即执行
    │
    ├── 阶段 B（GPU 加速提示）── 依赖 A 完成（避免清理时冲突）
    │       │
    │       └── 阶段 D（高斯模糊过渡）── 依赖 B 完成（B 的 will-change 为 D 做性能准备）
    │               │
    │               └── 阶段 C（侧边栏微渐变 + 毛玻璃分层）── 依赖 D 完成
    │                   （D 确认 page-enter 效果稳定后，再叠加毛玻璃分层，
    │                    避免 filter:blur 与 backdrop-filter 冲突难以定位）
    │
    └── 阶段 E（审计收尾）── 在 B/C/D 完成后执行
```

**推荐执行顺序**：A → B → D → C → E

**顺序理由**：
1. **A 先行**：清理死代码，避免后续改动时混淆
2. **B 次之**：加 will-change 提示，为 D 的高斯模糊动画做性能准备
3. **D 第三**：实现高斯模糊过渡，核心视觉效果，需单独验证帧率和 filter/backdrop-filter 冲突
4. **C 第四**：在 D 稳定后叠加毛玻璃分层。如果 D 的 filter:blur 与 backdrop-filter 有冲突，需先在 D 解决再叠加 C 的不透明度调整
5. **E 收尾**：审计全部改动，更新文档
