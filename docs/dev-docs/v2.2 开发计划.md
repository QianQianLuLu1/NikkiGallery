## 一、明确不纳入本期版本的内容

基于用户反馈与可行性调研，以下功能本期**不做**：

| 功能 | 不做原因 |
|---|---|
| 自动更新（electron-updater） | 用户明确表示不需要 electron-updater 自动更新机制；改为在"关于"页提供「检查更新」按钮，点击后跳转 GitHub 仓库（见 T07），由用户手动下载新版本 |
| macOS 版本 | 用户明确表示不需要 |
| 导出时 EXIF / 位置数据清理 | 无限暖暖游戏截图本身不含位置信息，多此一举 |
| QQ 直接发送文件 | 调研结论：仅 `tencent://` 拉起会话窗口，**无公开文件接口** → 改用剪贴板方案，见 T09 |
| 微信直接发送文件 | 调研结论：`weixin://` 仅拉起主界面，**无公开文件接口** → 改用剪贴板方案，见 T09 |
| 邮箱分享（mailto 折中方案） | 用户明确取消；mailto 不支持附件，体验割裂 |
| 蓝牙全自动发送 | 需自实现 OBEX 协议层，开发量过大；折中方案（拉起 fsquirt.exe 向导）体验割裂，本期不做 |

> **说明**：QQ / 微信虽无"直接发送文件"的公开接口，但采用「复制文件到系统剪贴板 + 弹窗引导粘贴」的轻量方案可绕开接口限制，体验接近原生，详见 T09。

## 三、P0 数据安全底线

### T01 · SQLite 数据库自动备份

**目标**：防止 `media.db` 损坏导致元数据（标签、分类、套装、操作历史）全部丢失。

**设计**：
- 备份策略三触发：① 应用启动时备份 ② 每日首次扫描后备份 ③ 设置页手动备份按钮
- 备份位置：`app.getPath('userData')/backups/`
- 命名规则：`media-YYYY-MM-DD-HHmm.db`
- 保留策略：最多保留 10 份，超限自动删除最旧
- 备份方式：SQLite 的 `VACUUM INTO`（在线一致性备份，无需停库）

**影响范围**：
- 新增 `src/main/services/backup-service.ts`
- 修改 `src/main/index.ts`（启动时调用 + IPC handler）
- 修改 `src/main/preload.ts`、`src/renderer/vite-env.d.ts`（暴露 `backup.create` / `backup.list` / `backup.restore`）
- 新增 `src/renderer/pages/settings/BackupManagePage.tsx`（数据管理 → 数据库备份 三级页面，对齐 T06 层级结构；若 T06 尚未完成可临时挂在 `SettingsPage.tsx`，T06 完成后迁移）

**验收标准**：
- [ ] 启动应用后 `backups/` 目录自动生成一份当日备份
- [ ] 手动点击"立即备份"后立即生成新备份并 Toast 提示
- [ ] 备份列表可查看（时间、大小），可手动恢复任意一份
- [ ] 备份数量超过 10 份时自动清理最旧的
- [ ] 备份文件大小合理（接近原 db 大小，不翻倍膨胀）

**风险点**：
- `VACUUM INTO` 需要 SQLite 3.27+，better-sqlite3 9.x 已支持
- 恢复备份时需关闭当前连接再替换文件，需做互斥锁

### T02 · 媒体文件完整性校验（missing 状态标记）

**目标**：用户在资源管理器里删了文件后，相册不应"虚标"——能在扫描时检测并标记 `missing` 状态，UI 给出灰显/提示。

**设计**：
- `media_files` 表新增字段 `is_missing INTEGER DEFAULT 0`、`missing_count INTEGER DEFAULT 0`（连续缺失计数）
- 扫描流程结束后，对所有数据库记录做一次文件存在性校验（`fs.existsSync` 批量并发）
- 缺失时 `missing_count += 1`，仅在 `missing_count >= 2` 时才置 `is_missing = 1`（避免移动盘临时不可用误判）
- 重新出现时立即 `missing_count = 0` 并置 `is_missing = 0`（一次出现即恢复，对用户更友好）
- 媒体列表渲染时：missing 项灰显 + 角标"已丢失"
- 详情页 missing 项显示"文件已被移动或删除，建议从库中移除"

**影响范围**：
- 修改 `src/main/database/connection.ts`（加 `is_missing` + `missing_count` 字段 + 迁移）
- 修改 `src/main/scanner/index.ts`（扫描后追加完整性校验步骤）
- 修改 `src/main/services/file-service.ts`（提供 `markMissing` / `cleanupMissing` 接口）
- 修改 `src/renderer/components/gallery/ImageGrid.tsx`、`MasonryView.tsx`、`ListView.tsx`、`TimelineView.tsx`（missing 视觉样式）
- 新增独立入口"丢失文件"（**不**放进回收站页，避免与"用户主动软删除"混淆）：可在相册顶部筛选条件加"仅看丢失"开关，或设置页"数据管理"加"清理丢失记录"按钮

**验收标准**：
- [ ] 外部删除一张图，连续两次扫描后该条目自动标记为 missing
- [ ] 仅一次扫描缺失时不会被立即标记，避免移动盘临时不可用误判
- [ ] missing 项在所有视图（网格 / 瀑布流 / 列表 / 时间轴）均有灰显样式
- [ ] 详情页给出友好提示并提供"从库中移除"按钮
- [ ] 文件被恢复后再次扫描，missing 标记立即清除（一次出现即恢复）
- [ ] 相册筛选可"仅看丢失"，并可一键清理所有 missing 记录

**风险点**：
- 大库（万级）校验耗时，需用并发控制（复用 `concurrency.ts`）+ 进度回调
- missing 不应与"用户主动软删除的回收站"混淆，二者状态独立、入口独立

### T05 · 智能相似图查找（pHash 聚类）

**目标**：当前 DuplicatesPage 仅按文件 hash 去重，无法发现"同姿势不同分数""同场景不同光线"的近似图。升级为感知哈希聚类，帮助用户挑选最佳一张。

**设计**：
- 扫描时为每张图片计算 pHash（8x8 DCT，64 位 hash）
- 数据库新增 `phash` 字段（TEXT，64 字符 0/1 串）
- 相似度判定：汉明距离 ≤ 5 视为相似
- DuplicatesPage 重命名为"相似图片"，分组展示相似图簇
- 每组提供"保留最佳一张"建议（按分辨率 / 清晰度 / 文件大小综合评分）

**影响范围**：
- 修改 `src/main/database/connection.ts`（加 phash 字段 + 迁移）
- 修改 `src/main/thumbnail/generator.ts`（生成缩略图时同步计算 pHash）
- 修改 `src/main/services/file-service.ts`（提供 `findSimilarGroups` 接口）
- 重写 `src/renderer/pages/DuplicatesPage.tsx`（相似图簇分组视图）

**验收标准**：
- [ ] 新增图片扫描后自动计算 pHash 并入库
- [ ] 相似图分组准确率 ≥ 80%（肉眼判定）
- [ ] 每组提供清晰的"保留建议"标识
- [ ] 支持手动选择保留项后一键删除其余
- [ ] 万级数据下相似度查询响应 ≤ 2 秒

**风险点**：
- 现有图片需补算 pHash（一次性迁移，扫描时进度提示）
- 纯色 / 极简构图图片 pHash 区分度低，可能误聚类，需配合分辨率做二次过滤

## 五、P2 设置层级化与关于信息完善

### T06 · 设置页面二级 / 三级 / 四级层级化重构

**目标**：当前设置页是单页平铺，外观 / 扫描 / 路径 / 数据管理 / 日志 / 关于 6 个区块堆在一起显得拥挤。重构为左侧分类 + 右侧详情的二级结构，部分内容下钻到三级 / 四级。

**设计方案**：

```
设置（二级页面：左侧分类列表 + 右侧详情）
├── 通用
│   ├── 启动行为（三级：自动扫描 / 增量扫描 / 启动时打开上次视图等开关；本期不做"检查更新"项）
│   └── 文件操作（三级：删除确认 / 批量操作阈值）
├── 外观
│   ├── 主题（三级：主题切换 + 主题预览）
│   └── 字体与显示（三级：字号 / 紧凑模式 / 动效减弱）
├── 扫描与路径
│   ├── 扫描选项（三级：增量 / 并发数 / 缩略图尺寸）
│   └── 游戏路径（三级：自定义路径列表 + 内置默认路径折叠展示）
├── 数据管理
│   ├── 数据库备份（三级：立即备份 / 备份列表 / 恢复 → 四级：恢复确认对话框）
│   ├── 缓存管理（三级：缩略图缓存大小 / 清理 LRU）
│   └── 清除数据（三级：危险操作区，需二次确认）
├── 日志与诊断
│   ├── 日志管理（三级：故障列表 + 导出 / 清除）
│   └── 崩溃报告（三级：崩溃记录列表，T13 新增）
└── 关于
    ├── 应用信息（三级：版本号 / 技术栈 / 开发者 / 「检查更新」按钮，详见 T07）
    ├── 联系方式（三级：GitHub / QQ 群 / 社交媒体，详见 T07）
    └── 致谢与开源协议（三级：依赖列表 + License）
```

**设计要点**：
- 二级页面采用"左侧固定宽度导航 + 右侧自适应内容区"经典布局
- 每个三级页面是独立组件，懒加载（React.lazy）
- 四级页面以模态框 / 抽屉形式呈现，避免页面层级过深导致迷失
- 保留顶部"设置"标题与返回按钮，确保用户随时可回到主页
- 配色与圆角遵循现有 Fluent Design 规范（frosted glass + 16px 圆角）

**影响范围**：
- 重构 `src/renderer/pages/SettingsPage.tsx`（拆分为容器 + 路由）
- 新增 `src/renderer/pages/settings/` 目录，包含各三级页面组件
- 修改 `src/renderer/components/layout/Sidebar.tsx`（设置入口保持不变）
- 修改 `src/renderer/stores/uiStore.ts`（新增 settings 子页面状态）

**验收标准**：
- [ ] 二级页面分类清晰，单页内容不滚动超过 1.5 屏
- [ ] 三级页面之间切换流畅，无明显卡顿
- [ ] 四级模态框打开 / 关闭动效自然
- [ ] 原有所有功能（主题切换 / 路径配置 / 日志管理 / 清除数据）在新结构中均可找到
- [ ] 键盘 Tab 导航符合逻辑顺序

**风险点**：

**目标**：基于调研结论，WiFi 局域网传输是唯一"公开接口 + 开发量小"的方案，本期实现。

**设计**：
- 主进程内嵌一个 WebDAV 服务（基于 `webdav-server` 或自写简易 HTTP 服务）
- 用户在详情页 / 批量操作中选"分享到手机"
- 弹窗展示：二维码（含 `http://电脑IP:端口`）+ 服务启停按钮
- 手机扫码或输入地址后，浏览器打开文件下载页
- 单次分享会话结束后自动关闭服务（默认 10 分钟超时）

**影响范围**：
- 新增 `src/main/services/share-wifi-service.ts`（WebDAV / HTTP 服务封装，与 T09 剪贴板服务分离，避免职责混杂）
- 修改 `src/main/index.ts`（IPC handler + 服务生命周期管理）
- 修改 `src/main/preload.ts`、`src/renderer/vite-env.d.ts`（暴露 `share.startWifi` / `share.stopWifi`）
- 新增 `src/renderer/components/common/WifiShareDialog.tsx`（二维码 + 服务状态展示，与 T09 引导弹窗形态不同，不复用）

**验收标准**：
- [ ] 同一局域网下手机浏览器可访问分享地址
- [ ] 二维码可被主流手机浏览器 / 微信扫一扫识别
- [ ] 服务启动 / 停止可控制，超时自动关闭
- [ ] 分享单张图片 / 批量分享均可
- [ ] 大文件（>100MB 视频）下载稳定不断流

**风险点**：
- 防火墙拦截：首次启动时需提示用户允许 Node 通过防火墙
- 局域网 IP 探测需处理多网卡情况（优先返回可达 IP）
- 安全性：服务仅绑定到本机局域网 IP，不暴露公网

### T09 · 媒体文件一键分享（微信 / QQ / vivo办公套件，剪贴板方案）

**目标**：新增「分享到微信 / 分享到QQ / 分享到vivo办公套件」三个渠道。全程采用「自动复制文件到系统剪贴板 + 弹窗引导粘贴」的统一轻量方案，不接入任何第三方 SDK、不申请额外接口权限，无合规与兼容性风险，所有逻辑完全本地实现，不破坏现有架构与功能。

**核心设计原则**：
1. 完全复用现有文件选择逻辑，支持单文件、多文件批量分享，兼容图片与视频格式
2. 三个渠道采用完全一致的实现逻辑，仅引导文案不同，保证体验统一
3. 入口轻量化植入，不改动现有主布局，不干扰核心图库浏览功能
4. 样式自动适配所有现有主题（当前为「默认简约」「柔粉轻奢」两套；若后续新增「未来科技」等主题需同步适配），视觉风格统一

**一、功能入口设计**：

1. **右键菜单入口**：在图库文件右键菜单中新增「分享」二级子菜单，包含三个选项：
   - 分享到微信
   - 分享到QQ
   - 分享到vivo办公套件
2. **顶部工具栏入口**：主界面顶部操作栏新增「分享」按钮，点击展开同款下拉菜单
3. **批量操作适配**：批量选中多个文件后，右键菜单与顶部批量操作栏的分享入口同步生效
4. **空选校验**：未选中任何文件时点击分享，弹出轻量提示「请先选择要分享的媒体文件」

**二、统一交互逻辑**（三个渠道通用）：

用户点击对应渠道后，执行两步固定流程：

1. **自动复制**：后台将选中的所有图片 / 视频文件，以系统文件格式（CF_HDROP）写入 Windows 剪贴板，确保所有软件均可识别并粘贴
2. **引导弹窗**：屏幕居中弹出轻量提示弹窗，按渠道展示专属引导文案：

   | 渠道 | 引导文案 |
   |---|---|
   | 微信 | ✅ 文件已复制到剪贴板<br/>请打开微信聊天窗口，粘贴即可发送图片/视频 |
   | QQ | ✅ 文件已复制到剪贴板<br/>请打开 QQ 聊天窗口，粘贴即可发送图片/视频 |
   | vivo办公套件 | ✅ 文件已复制到剪贴板<br/>请打开 vivo 办公套件，粘贴即可同步到手机 |

3. **弹窗细节**：
   - 底部「知道了」按钮，点击关闭
   - 弹窗支持 3 秒后自动关闭，不打断用户操作
   - 弹窗无多余广告与跳转，仅做提示

**三、附加适配逻辑**：

1. **安装状态检测（可选增强）**：点击分享时自动检测系统是否安装对应软件
   - 未检测到时，文案调整为：「未检测到对应软件，文件已复制到剪贴板，你可手动打开软件后粘贴发送」
   - 不强制校验，即使检测不到也始终执行复制操作，保证功能可用
2. **异常容错**：文件路径无效、权限不足、文件过大导致复制失败时，弹出明确错误提示，不造成程序崩溃
3. **性能要求**：单文件复制响应时间 ≤ 100ms，批量 20 张以内图片复制响应 ≤ 500ms，无界面卡顿

**四、技术实现要求**：

1. **剪贴板写入**：本工程为 Electron（非 Qt），采用 Electron 原生 `clipboard` API，通过 `clipboard.writeBuffer('CF_HDROP', buffer)` 写入 CF_HDROP 格式，兼容 Windows 全版本，确保微信 / QQ / vivo办公套件均可正常识别粘贴
   - CF_HDROP 缓冲区结构：`DROPFILES` 头（20 字节，pFiles=20, fWide=1）+ UTF-16LE 文件路径列表（以 `\0` 分隔，双 `\0` 结尾）
   - 多文件批量写入时按上述结构依次拼接
2. **不引入任何第三方库 / SDK / 外部依赖**，纯本地实现，不增加安装包体积
3. **样式复用**：所有弹窗、菜单样式完全复用现有组件体系（ConfirmDialog / ContextMenu / Toast 等），自动适配现有主题的色彩、圆角、材质效果
4. **逻辑解耦**：分享逻辑独立封装到 `share-clipboard-service.ts`，与现有图库模块解耦；后续新增渠道仅需扩展菜单项与文案配置，无需修改核心逻辑

**影响范围**：
- 新增 `src/main/services/share-clipboard-service.ts`（`copyFilesToClipboard` + `detectAppInstalled` + 渠道文案配置表）
- 修改 `src/main/index.ts`（IPC handler：`share:copyFiles` / `share:detectApp`）
- 修改 `src/main/preload.ts`、`src/renderer/vite-env.d.ts`（暴露 `share.copyFiles` / `share.detectApp`）
- 新增 `src/renderer/components/common/ShareGuideDialog.tsx`（引导弹窗，3 秒自动关闭 + 「知道了」按钮）
- 修改 `src/renderer/components/common/ContextMenu.tsx`（新增「分享」二级子菜单）
- 修改 `src/renderer/components/gallery/GalleryToolbar.tsx`、`BatchActions.tsx`（顶部 / 批量操作栏加分享按钮）
- 修改 `src/renderer/icons/index.tsx`（新增微信 / QQ / vivo 图标）

**验收标准**：
- [ ] 单张图片、多张图片、单个视频、多个视频混合选择，均能成功复制到剪贴板
- [ ] 在微信 / QQ / vivo办公套件中粘贴后可正常发送
- [ ] 三个渠道入口正常显示，点击后弹窗文案准确，3 秒自动关闭与「知道了」手动关闭均正常
- [ ] 未选中文件时弹出「请先选择要分享的媒体文件」提示
- [ ] 未安装对应软件时文案调整为兜底版本，复制操作仍执行
- [ ] 文件路径无效 / 权限不足 / 文件过大等异常有明确错误提示，无崩溃
- [ ] 单文件复制 ≤ 100ms，20 张以内批量 ≤ 500ms
- [ ] 现有两套主题（默认简约、柔粉轻奢）下菜单 / 弹窗样式适配正常，无视觉错乱
- [ ] 不影响扫描 / 分类 / 批量编辑等所有核心功能
- [ ] 无额外网络请求、无隐私数据上传，全程纯本地操作

**风险点**：
- CF_HDROP 缓冲区字节对齐与编码（UTF-16LE）需严格按 Windows 规范，否则粘贴时乱码或丢失文件

- vivo办公套件安装路径不固定，检测逻辑需走注册表 `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\*` 模糊匹配应用名

### T10 · 缩略图缓存 LRU 淘汰

**目标**：缩略图目录会随库增长无限膨胀，需加最大容量限制 + LRU 淘汰机制。

**设计**：
- 缩略图目录：`app.getPath('userData')/thumbnails/`
- 默认上限：2GB（可在设置中调整）
- 淘汰策略：每次启动扫描后检查总大小，超限时按 atime 升序删除最旧文件，直到降至阈值以下
- 设置页"数据管理 → 缓存管理"提供手动清理入口

**影响范围**：
- 新增 `src/main/services/thumbnail-cache-service.ts`
- 修改 `src/main/index.ts`（启动后检查 + IPC handler）
- 修改 `src/renderer/pages/settings/CacheManagePage.tsx`（T06 的三级页面之一）

**验收标准**：
- [ ] 启动后自动检查缩略图目录大小
- [ ] 超限时自动清理，清理后 Toast 提示释放空间
- [ ] 设置页可查看当前缓存大小、手动清理、调整上限
- [ ] 清理后再次访问被清理的图片能重新生成缩略图

**风险点**：
- 清理过程中若有图片正在被渲染需做保护（引用计数或延迟删除）
- 文件 atime 在某些文件系统上可能被禁用，需 fallback 到 mtime

#### T05 · 智能相似图查找（pHash 聚类）（✅ 完成）

**pHash 计算工具**（新增 `src/main/utils/phash.ts`）：
- `calculatePHash(filePath)`：基于 32×32 DCT-II 的 64 位感知哈希
  - 算法流程：sharp 缩放到 32×32 灰度 → 二维 DCT-II（行列两次一维 DCT）→ 取左上 8×8 低频系数 → 排除 DC 分量后求均值 → 64 位与均值比较生成 0/1 串
  - DC 分量固定为 '0'，避免它影响相似度判断
  - 预计算 DCT 余弦矩阵 `cosMatrix[k][n] = cos(π/N * (n+0.5) * k)`，避免重复计算
  - 计算失败返回 null（文件损坏、sharp 不支持的格式等）
- `hammingDistance(hash1, hash2)`：返回不同位的数量；长度不一致返回 -1
- 修复编译错误：`info.stride` 在 sharp `OutputInfo` 类型上不存在，改用 `info.width`（greyscale 后 channels=1，stride 即等于 width）

**数据库迁移**（`src/main/database/connection.ts`）：
- `media_files` 表新增 `phash TEXT` 字段
- 新增索引 `idx_media_files_phash`
- 新增 `migratePhash()` 迁移方法（ALTER TABLE + CREATE INDEX，try-catch 容错重复执行）

**主进程**（`src/main/index.ts`）：
- `processThumbnailForRow` 同步计算图片 pHash（仅图片，视频不计算）：
  - UPDATE 语句扩展为 `width, height, thumbnail, phash`
  - 计算失败时 phash 保持 null，不阻塞缩略图生成
- 新增 `generatePhashForUnprocessed()` 异步补算方法：
  - 查询 `file_type = 'image' AND phash IS NULL` 的记录
  - 使用 `runWithConcurrency` 控制并发（与缩略图生成共用 `THUMBNAIL_CONCURRENCY`）
  - 返回 `{ processed, total }` 供前端显示进度
  - `performStartupScan` 中触发后台补算（与缩略图生成解耦，避免阻塞）
- 新增 `media:findSimilar` IPC handler（Union-Find 聚类）：
  - 参数：`threshold` 汉明距离阈值（默认 5，范围 0-64）
  - 查询 `phash IS NOT NULL AND phash != ''` 的未软删除图片
  - 两两比较汉明距离，≤ threshold 时 union；路径压缩优化（`parent[x] = parent[parent[x]]`）
  - 按 root 分组，仅保留 ≥ 2 张的相似组
  - 组内按 `modified_at` 降序，组间按可释放空间降序
  - 返回与 `findDuplicates` 相同结构 + `threshold` + `hashedFiles` 额外字段
- 新增 `media:generatePhash` IPC handler：手动触发补算，返回处理进度

**IPC 暴露层**：
- `src/main/preload.ts`：`media` 对象添加 `findSimilar` 和 `generatePhash` 方法
- `src/renderer/vite-env.d.ts`：补全 `findSimilar` 和 `generatePhash` 类型声明
