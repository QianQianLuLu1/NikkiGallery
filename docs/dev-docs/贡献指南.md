## 一、新手友好：从反馈开始

如果你是零基础，**不用写代码也能贡献**：

1. **使用软件**：发现 bug 或不顺手的地方
2. **反馈问题**：按 [第二节](#二提交-issue反馈问题或建议) 提交 Issue
3. **提建议**：想到新功能或改进点
4. **帮忙测试**：在 Issue 中确认其他人反馈的问题是否能复现
5. **改进文档**：发现文档不清楚的地方，提 PR 修改

这些都是有价值的贡献，我们会认真对待每一条反馈。

## 二、提交 Issue（反馈问题或建议）

### 2.1 什么时候提 Issue

- 发现 bug（程序出错、行为异常）
- 有功能建议（希望增加新功能、改进现有功能）
- 文档有问题（描述不清楚、有错误）
- 不清楚如何使用某个功能

### 2.2 提交前必做

**搜索现有 Issue**：你的问题可能已经被提过。

1. 打开项目仓库的 Issues 页面
2. 在搜索框输入关键词（如"扫描不到"、"卡顿"）
3. 看是否有相同问题
4. 如果有，**在下方评论补充你的情况**（不要新开 Issue）
5. 如果没有，再新开 Issue

### 2.3 Issue 模板

**反馈 Bug**请用以下格式：

```markdown
**问题描述**
<一句话说明发生了什么>

**复现步骤**
1. 打开程序
2. 点击 '...'
3. 看到 '...'

**期望结果**
<应该发生什么>

**实际结果**
<实际发生了什么>

**环境信息**
- 程序版本：<设置 → 关于 中查看>
- 操作系统：<如 Windows 11 23H2>
- 是否每次都能复现：<是 / 否 / 偶尔>

**诊断包**
<按 docs/排障手册.md 中"导出崩溃报告"章节操作，把 ZIP 文件拖入 Issue>
```

**提功能建议**请用以下格式：

```markdown
**功能类型**
<新功能 / 改进现有功能 / 优化体验>

**使用场景**
<在什么情况下需要这个功能？解决了什么问题？>

**期望效果**
<希望怎么用？能给个简单的示例吗？>

**备选方案**
<有没有其他方式也能解决？为什么选这个？>

**附加信息**
<截图、参考其他软件的类似功能等>
```

### 2.4 Issue 的生命周期

1. **提交**：你创建 Issue
2. **分类**：维护者给 Issue 加标签（bug / feature / docs 等）
3. **讨论**：维护者或其他用户讨论方案
4. **处理**：有人认领处理（或维护者自己处理）
5. **关闭**：问题解决后关闭 Issue

**请耐心等待回复**，维护者可能在忙其他事情。如果一周没回复，可以在 Issue 下 @维护者提醒。

## 三、提交 PR（贡献代码）

### 3.1 准备工作

1. **注册 GitHub 账号**：[github.com](https://github.com)
2. **安装 Git**：[git-scm.com](https://git-scm.com)
3. **安装 Node.js**（建议 18+）：[nodejs.org](https://nodejs.org)
4. **Fork 仓库**：在项目 GitHub 页面点击 "Fork" 按钮
5. **克隆到本地**：

```bash
git clone https://github.com/<你的用户名>/wxnn-photo-manager.git
cd wxnn-photo-manager
```

6. **添加上游仓库**（同步最新代码用）：

```bash
git remote add upstream https://github.com/<原始仓库>/wxnn-photo-manager.git
```

### 3.2 开发流程

#### 第 1 步：同步最新代码

```bash
git checkout main
git pull upstream main
```

#### 第 2 步：创建功能分支

**命名规则**：
- 新功能：`feat/<简短描述>`，如 `feat/outfit-grouping`
- 修复 bug：`fix/<bug-描述>`，如 `fix/duplicate-scoring-stability`
- 改文档：`docs/<描述>`，如 `docs/update-troubleshooting`

```bash
git checkout -b feat/outfit-grouping
```

#### 第 3 步：写代码

参考 [第四节代码规范](#四代码规范)。

#### 第 4 步：本地验证（必做）

```bash
# 1. 安装依赖（首次或依赖更新时）
npm install

# 2. 格式化代码
npm run format

# 3. 类型检查
npm run typecheck

# 4. 跑测试
npm test

# 5. 启动开发模式预览效果
npm run dev
```

**所有命令必须通过**才能提交。

#### 第 5 步：提交代码

参考 [第五节提交规范](#五提交规范)。

```bash
git add .
git commit -m "feat(gallery): 添加按套装分组按钮"
```

#### 第 6 步：推送到你的仓库

```bash
git push origin feat/outfit-grouping
```

#### 第 7 步：创建 PR

1. 打开你 Fork 的仓库 GitHub 页面
2. 点击 "Compare & pull request" 按钮
3. 填写 PR 描述（见下方模板）
4. 点击 "Create pull request"

### 3.3 PR 描述模板

```markdown
## 改动说明
<这个 PR 做了什么？为什么？>

## 改动类型
- [ ] Bug 修复（不破坏现有功能）
- [ ] 新功能（添加新能力）
- [ ] 重构（不改外部行为）
- [ ] 文档更新
- [ ] 测试补充

## 关联 Issue
Fixes #<Issue 编号>  <!-- 如 Fixes #123 -->

## 测试情况
- [ ] 本地已跑 `npm test`，全部通过
- [ ] 本地已跑 `npm run typecheck`，无错误
- [ ] 手动验证受影响功能正常

## 截图
<如果是 UI 改动，请附上前后对比截图>
```

### 3.4 PR 评审流程

1. **自动检查**：CI 会自动跑测试和类型检查
2. **代码评审**：维护者会 review 代码
3. **修改建议**：根据反馈修改，追加 commit 后推送（不要 squash）
4. **合并**：通过评审后，维护者合并 PR

**评审通常需要 1-3 天**，请耐心等待。如果有紧急情况，可以在 PR 中 @维护者。

### 3.5 同步上游更新

如果开发期间主仓库有更新，需要同步：

```bash
git fetch upstream
git checkout main
git merge upstream/main
git push origin main

# 切回功能分支，合并主分支的最新改动
git checkout feat/outfit-grouping
git rebase main  # 或 git merge main
# 解决冲突（如果有）
git push origin feat/outfit-grouping
```

## 四、代码规范

### 4.1 复用优先（最重要）

**写代码前必须先检查是否有现成的可复用组件**。

参考 [docs/AI开发规范.md](./AI开发规范.md) 第二章，重点：

- 工具函数：先查 `src/common/utils/`（`@common/utils`）
- UI 组件：先查 `src/renderer/components/common/`
- 设置行：先查 `src/renderer/pages/settings/shared.tsx`
- CSS 类：先查 `src/renderer/styles/globals.css`

**禁止重复实现**已有的工具或组件。

### 4.2 命名约定

| 类型 | 规范 | 示例 |
|---|---|---|
| 函数/变量 | camelCase | `formatSize`、`isLoading` |
| 类型/接口/组件 | PascalCase | `MediaRepository`、`BaseDialog` |
| 常量 | UPPER_SNAKE_CASE | `MAX_FILE_SIZE` |
| 文件名 | 与默认导出同名 | `BaseDialog.tsx` 导出 `BaseDialog` |
| 测试文件 | `<源文件名>.test.ts` | `format.ts` → `format.test.ts` |

### 4.3 代码风格

- **函数要小**：单个函数不超过 50 行，参数不超过 4 个
- **嵌套要浅**：用提前返回（Guard Clauses）处理异常情况
- **分支要简**：if/else 嵌套不超过 3 层
- **TypeScript 严格模式**：禁止 `any`（除非外部 API 返回类型未知）

### 4.4 注释规范

**只写"为什么"**，不写"是什么"。

好注释：
```ts
// 修复 C-S3：并发生成同一文件的缩略图时存在竞态条件
// 使用 Map<string, Promise> 作为互斥锁
private generatingLocks: Map<string, Promise<string | null>> = new Map()
```

不需要的注释（代码本身已经说明）：
```ts
// 遍历数组（不需要写）
for (const item of items) { ... }
```

### 4.5 错误处理

- 所有外部输入（用户、API、文件）默认不可信
- 错误必须被处理：返回有意义的值或向上抛，**禁止吞掉**
- 资源用完就关：监听器、定时器、连接、文件句柄

### 4.6 测试要求

- **纯函数工具**：必须有单元测试，覆盖正常路径 + 边界（空值/NaN/异常输入）
- **React 组件**：暂不强制要求测试（项目未配置 jsdom 环境）
- 测试文件与源文件同目录，命名 `<源文件名>.test.ts`

### 4.7 文件组织

```
src/
├── common/              # 跨进程共享工具（主进程 + 渲染进程都能用）
│   └── utils/           # 纯函数工具
├── main/                # Electron 主进程代码
│   ├── database/        # 数据库相关
│   ├── scanner/         # 扫描器
│   ├── services/        # 业务服务
│   └── utils/           # 主进程专用工具（依赖 Node API）
├── renderer/            # 渲染进程代码（React）
│   ├── components/      # React 组件
│   │   └── common/     # 公共组件（跨页面复用）
│   ├── pages/           # 页面
│   ├── stores/          # Zustand 状态管理
│   ├── hooks/           # 自定义 Hooks
│   └── utils/           # 渲染进程专用工具
└── shared/              # 主进程与渲染进程共享的类型定义
```

**新增文件时**：
- 跨进程都能用的纯函数 → `src/common/utils/`
- 只在主进程用 → `src/main/...`
- 只在渲染进程用 → `src/renderer/...`
- 公共 UI 组件 → `src/renderer/components/common/`

## 五、提交规范

### 5.1 提交信息格式

```
<type>(<scope>): <subject>

<body>
```

### 5.2 type 取值

| type | 含义 | 示例 |
|---|---|---|
| `feat` | 新功能 | `feat(gallery): 添加按套装分组按钮` |
| `fix` | Bug 修复 | `fix(scanner): 修复 Steam 库路径解析错误` |
| `refactor` | 重构（不改外部行为） | `refactor(repository): 抽取 SQL 访问层` |
| `test` | 新增/修改测试 | `test(repository): 补全 MediaRepository 单元测试` |
| `docs` | 文档 | `docs: 新增排障手册和贡献指南` |
| `chore` | 构建/工具/依赖 | `chore(deps): 升级 vitest 到 1.6.0` |
| `style` | 格式化（不改逻辑） | `style: 应用 Prettier 格式化` |
| `perf` | 性能优化 | `perf(grid): 优化虚拟滚动性能` |

### 5.3 scope 取值

按模块划分，常用值：

- `gallery` —— 图库页面相关
- `scanner` —— 扫描器
- `repository` —— 数据访问层
- `editor` —— 图片编辑器
- `settings` —— 设置页
- `database` —— 数据库
- `thumbnail` —— 缩略图
- `dialog` —— 对话框组件
- `utils` —— 工具函数
- `deps` —— 依赖
- 不写 scope —— 跨模块或无法归类时

### 5.4 subject 要求

- **简洁明了**：一句话说明本次提交做了什么
- **祈使句**：用"添加"而不是"添加了"，用"修复"而不是"修复了"
- **不加句号**：结尾不加 `.`
- **不超过 50 字**（中文）/ 72 字符（英文）

**好示例**：
```
feat(gallery): 添加按套装分组按钮
fix(scanner): 修复 Steam 库路径解析错误
docs: 新增排障手册
```

**差示例**：
```
update code           # 太模糊
修复了一些 bug         # 没说修了什么
添加按套装分组按钮。    # 多了句号
新增功能并且重构了部分代码  # 一次提交做了两件事
```

### 5.5 单次提交的范围限制

**每次提交只解决一个问题**：

| 类型 | 允许范围 | 禁止 |
|---|---|---|
| Bug 修复 | 1 个 bug + 必要的回归测试 | 顺手重构周边代码 |
| 新功能 | 1 个完整功能点 + 测试 | 同时实现多个相关功能 |
| 重构 | 1 个文件或 1 组紧密相关的函数 | 跨模块大重构 |
| 依赖升级 | 1 个依赖 + 验证 | 同时升级多个依赖 |

**如果一次改了多个东西，拆成多个 commit**：

```bash
git add src/common/utils/format.ts
git commit -m "feat(utils): 新增 formatSize 文件大小格式化"

git add src/main/services/backup-service.ts
git commit -m "refactor(backup): 复用 @common/utils/formatSize"
```

### 5.6 提交前检查清单

提交前请逐项确认：

- [ ] 代码已跑 `npm run format`
- [ ] 代码已跑 `npm run typecheck`，无错误
- [ ] 代码已跑 `npm test`，全部通过
- [ ] 手动验证受影响功能正常
- [ ] 提交信息符合规范
- [ ] 单次提交只解决一个问题

## 六、适合新手参与的任务

### 6.1 文档类（零基础可做）

- **改进文档**：发现文档不清楚、过时、有错误的地方，提 PR 修改
- **翻译文档**：把文档翻译成其他语言
- **补充示例**：给现有功能写使用示例
- **整理 FAQ**：收集常见问题，整理成 FAQ 文档

**入门任务示例**：
- 修正 [docs/排障手册.md](./排障手册.md) 中的错别字
- 给 [README](../README.md) 添加截图
- 补充某个功能的详细使用说明

### 6.2 测试类（有基础可做）

- **补全测试**：给未覆盖的工具函数补单元测试
- **补充边界测试**：现有测试覆盖不全的边界情况
- **写集成测试**：跨模块的集成测试用例

**入门任务示例**：
- 给 `src/common/utils/` 下的工具函数补充边界测试
- 给 `src/main/utils/` 下的工具函数补充异常输入测试

### 6.3 Bug 修复类（有经验可做）

- 在 Issue 列表中找标注了 `good first issue` 标签的问题
- 找标注了 `help wanted` 标签的问题
- 修复自己使用中发现的 bug

**入门任务示例**：
- 修复 UI 显示错位（CSS 调整）
- 修复文案错误
- 修复某个按钮点击无响应

### 6.4 小功能类（有经验可做）

- 添加一个新的筛选条件
- 添加新的快捷键支持
- 添加新的导出格式
- 添加新的右键菜单项

**入门任务示例**：
- 给图片列表添加"按文件大小排序"选项
- 给批量操作添加"全选"按钮
- 添加一个切换深色/浅色主题的快捷键

### 6.5 不适合新手的任务

以下任务需要深入了解项目架构，**新手不要碰**：

- 数据库 schema 变更
- 扫描器核心逻辑修改
- 主进程退出流程修改
- 跨进程通信协议变更
- 性能关键路径优化（虚拟滚动、缓存策略）

## 七、开发环境搭建

### 7.1 系统要求

- **操作系统**：Windows 10 1803+（必须，因使用 tar.exe）
- **Node.js**：18+（推荐 20 LTS）
- **内存**：8GB+（推荐 16GB）
- **磁盘空间**：5GB+（含依赖和构建产物）

### 7.2 安装步骤

```bash
# 1. 克隆仓库
git clone https://github.com/<你的用户名>/wxnn-photo-manager.git
cd wxnn-photo-manager

# 2. 安装依赖（首次较慢，可能 5-10 分钟）
npm install

# 3. 启动开发模式
npm run dev
```

### 7.3 常用命令

| 命令 | 作用 |
|---|---|
| `npm run dev` | 启动开发模式（带热重载） |
| `npm run build` | 构建生产版本 |
| `npm test` | 跑所有测试 |
| `npm run test:watch` | 监听模式跑测试 |
| `npm run test:coverage` | 跑测试并生成覆盖率报告 |
| `npm run typecheck` | 类型检查 |
| `npm run format` | 用 Prettier 格式化代码 |
| `npm run format:check` | 检查格式是否规范（不修改） |
| `npm run lint` | ESLint 检查 |
| `npm run pack` | 打包成 exe（不含安装程序） |
| `npm run dist` | 打包成完整安装包 |

### 7.4 项目结构速查

参考 [第四节 4.7 文件组织](#47-文件组织)。

**核心模块**：

| 模块 | 位置 | 职责 |
|---|---|---|
| 主进程入口 | `src/main/index.ts` | Electron 启动、IPC、窗口管理 |
| 数据库 | `src/main/database/` | SQLite 访问层 |
| 扫描器 | `src/main/scanner/` | 游戏截图扫描 |
| 缩略图 | `src/main/thumbnail/` | 缩略图生成与缓存 |
| 公共工具 | `src/common/utils/` | 跨进程共享纯函数 |
| 渲染进程入口 | `src/renderer/main.tsx` | React 应用入口 |
| 页面 | `src/renderer/pages/` | 各个页面 |
| 公共组件 | `src/renderer/components/common/` | 跨页面复用组件 |
| 状态管理 | `src/renderer/stores/` | Zustand stores |

### 7.5 调试技巧

- **主进程日志**：在终端中可以看到 `console.log` 输出
- **渲染进程日志**：在 Electron 窗口按 `Ctrl+Shift+I` 打开 DevTools
- **数据库调试**：用 DB Browser for SQLite 打开 `%APPDATA%\wxnn-photo-manager\wxnn_photo_manager.db`
- **看日志文件**：`%APPDATA%\wxnn-photo-manager\logs\main-YYYY-MM-DD.log`

## 八、常见问题

### Q1：我没有写过代码，能参与吗？

**可以**。从以下方式开始：
1. 使用软件，反馈 bug
2. 改进文档
3. 翻译文档
4. 等熟悉了再尝试小功能

### Q2：我提交了 PR，多久能得到回复？

通常 1-3 天内会有初步回复。如果一周没回复，可以在 PR 中 @维护者。

### Q3：我的 PR 被要求修改怎么办？

这是正常流程，不代表你的代码不好。维护者会给出具体的修改建议，按建议修改后重新推送即可。**不要关闭 PR 重开**，直接在原 PR 上追加 commit。

### Q4：我可以同时做多个任务吗？

**不建议**。一次只做一个任务，做完再开始下一个。这样可以专注，避免半途而废。

### Q5：我的代码和别人的 PR 冲突了怎么办？

先和对方在 PR 中讨论，看谁先合并。后合并的一方需要 rebase 到最新的 main 分支，解决冲突。

### Q6：我想做的功能不在 Issue 列表中怎么办？

**先提 Issue** 说明你想做什么，等维护者讨论后再开始写代码。避免辛辛苦苦写完发现不符合项目方向。

### Q7：我发现了安全问题怎么办？

**不要在公开 Issue 中提交安全问题**。请直接发邮件到 [安全邮箱]，标题以 `[SECURITY]` 开头。我们会在 24 小时内回复。

### Q8：如何成为项目的 Collaborator（协作者）？

持续贡献高质量代码（提交 PR、参与 review、帮助新手），维护者会主动邀请你加入。**不要主动要求**。

### Q9：项目用什么技术栈？

- **桌面框架**：Electron
- **前端**：React 18 + TypeScript
- **构建**：Vite
- **状态管理**：Zustand
- **数据库**：better-sqlite3
- **测试**：Vitest
- **样式**：Tailwind CSS + 自定义 CSS 变量
- **动画**：Motion（前身 Framer Motion）

### Q10：我应该用中文还是英文交流？

都可以。项目主要使用中文，但英文也欢迎。**用你舒服的语言表达**即可。
