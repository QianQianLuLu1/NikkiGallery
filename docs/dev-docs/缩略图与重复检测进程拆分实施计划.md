# 缩略图与重复检测进程拆分实施计划

> 目标：将「缩略图批量生成」「pHash 批量补算 + 重复标记」两类 CPU 密集任务迁移到独立 utilityProcess，复用已落地的 scanner worker 架构模式，解决批量处理时主进程 UI 卡顿问题。原有功能输出与上层调用方式 100% 不变。

## 二、两类任务的接入逻辑

### 2.1 复用现有 worker 架构的方式

**架构模式复用**（非代码继承）：media worker 完整复用 scanner worker 已验证的 7 项架构要素，但**进程独立、消息协议独立、bridge 独立**，避免任务类型耦合。

| 架构要素 | scanner worker 实现 | media worker 复用方式 |
|---|---|---|
| 进程启动 | `utilityProcess.fork(path.join(__dirname, 'worker-entry.js'), [], { serviceName, stdio: 'pipe' })` | 完全一致，`serviceName: 'media-worker'`，`worker-entry.js` 路径解析方式相同 |
| 启动握手 | `WORKER_READY` 信号 + `readyPromise` | 完全一致 |
| DB 连接 | worker 内 `new Database(dbPath)` + WAL + `busy_timeout=5000` | 完全一致（WAL 多连接并发已验证） |
| 消息协议 | discriminated union `WorkerCommand` / `WorkerEvent` | 新建 `MediaWorkerCommand` / `MediaWorkerEvent`，结构相同 |
| 进度推送 | `SCAN_PROGRESS` → bridge 分发 → `EventEmitter.emit` → IPC `webContents.send` | 完全一致，新增 `thumbnail:progress` / `phash:progress` / `duplicate:progress` 三个 IPC 推送 channel |
| 取消机制 | 模块级 `shouldStop` + `SCAN_STOP` 消息 | 完全一致，三个独立标志 `shouldStopThumbnail` / `shouldStopPhash` / `shouldStopDuplicate` |
| 异常退出兜底 | `handleWorkerExit` 合成 `SCAN_COMPLETE(success=false)` | 完全一致，合成对应任务的 `*_COMPLETE(success=false)` |
| 退出清理 | `SCAN_DISPOSE` → 500ms 等待 → `kill()` 兜底 | 完全一致，`MEDIA_WORKER_DISPOSE` → 500ms 等待 → `kill()` 兜底 |

### 2.2 消息协议设计（`worker-protocol.ts`）

```typescript
// ============ 主进程 → worker ============
export type MediaWorkerCommand =
  // 缩略图批量生成（同时补算缺phash的图片，保持 processThumbnailForRow 三路并行）
  | {
      type: 'THUMBNAIL_BATCH_START'
      payload: {
        dbPath: string
        cacheDir: string            // 缩略图缓存目录，由主进程 ThumbnailGenerator.getCacheDir() 提供
        thumbnailQuality: 'low' | 'standard' | 'high'
      }
    }
  // pHash 批量补算（仅对已有缩略图但缺 phash 的图片）
  | {
      type: 'PHASH_BATCH_START'
      payload: { dbPath: string }
    }
  // 重复标记（O(n²) Union-Find 聚类）
  | {
      type: 'DUPLICATE_MARK_START'
      payload: { dbPath: string }
    }
  // 取消指令（各自独立，互不影响）
  | { type: 'THUMBNAIL_STOP' }
  | { type: 'PHASH_STOP' }
  | { type: 'DUPLICATE_STOP' }
  // 进程清理
  | { type: 'MEDIA_WORKER_DISPOSE' }

// ============ worker → 主进程 ============
export type MediaWorkerEvent =
  | { type: 'WORKER_READY' }
  | {
      type: 'THUMBNAIL_PROGRESS'
      payload: { processed: number; total: number; currentFile: string }
    }
  | {
      type: 'THUMBNAIL_COMPLETE'
      payload: { success: boolean; message: string; processed: number; total: number }
    }
  | {
      type: 'PHASH_PROGRESS'
      payload: { processed: number; total: number; currentFile: string }
    }
  | {
      type: 'PHASH_COMPLETE'
      payload: {
        success: boolean
        message: string
        processed: number
        total: number
        duplicatesResult?: { markedDuplicates: number; totalGroups: number }  // phash 完成后自动链式 markDuplicates
      }
    }
  | {
      type: 'DUPLICATE_PROGRESS'
      payload: { compared: number; totalPairs: number }   // O(n²) 进度，totalPairs = n*(n-1)/2
    }
  | {
      type: 'DUPLICATE_COMPLETE'
      payload: { success: boolean; message: string; markedDuplicates: number; totalGroups: number }
    }
  | { type: 'MEDIA_WORKER_LOG'; payload: { level: 'info' | 'warn' | 'error'; message: string; args?: unknown[] } }
  | { type: 'WORKER_ERROR'; payload: { message: string; stack?: string } }
```

### 2.4 进度与结果回传机制

| 任务 | 原有进度 | 新增进度 | 完成事件 |
|---|---|---|---|
| 缩略图批量 | 无（仅完成时 `media:updated`） | `thumbnail:progress`（processed/total/currentFile） | `thumbnail:complete` + 原有 `media:updated` |
| pHash 批量 | 无（仅 console.log） | `phash:progress`（processed/total/currentFile） | `phash:complete`（含 duplicatesResult） |
| 重复标记 | 无（仅 console.log） | `duplicate:progress`（compared/totalPairs） | `duplicate:complete` |

**节流策略**：worker 内部进度推送间隔 ≥ 500ms（与 scanner worker 一致），避免 IPC 洪水。

**完成事件保证**：
- worker 正常完成 → 发送 `*_COMPLETE(success=true)`
- worker 异常退出 → bridge 合成 `*_COMPLETE(success=false, message='进程异常退出')`
- worker 收到 STOP → 完成当前批次后发送 `*_COMPLETE(success=true, message='已取消')`
- 主进程 `pendingPromise` 在收到 `*_COMPLETE` 时必然 resolve，**不会卡死**

### 2.5 worker 内部的缩略图生成实现

worker 不依赖主进程的 `ThumbnailGenerator` 类，内部实现轻量版 `generateThumbnailInWorker(filePath, cacheDir, quality)`：

| 功能 | 主进程 ThumbnailGenerator | worker 内部轻量版 | 说明 |
|---|---|---|---|
| sharp 缩放 / jpeg 编码 | ✅ | ✅ | 完全一致，直接 `import sharp from 'sharp'` |
| ffmpeg 视频抽帧 | ✅ `extractVideoFrame` | ✅ 复用同函数 | 直接 import，ffmpeg 路径通过 `ffmpegPath` 解析 |
| LRU 访问时间记录 | ✅ `accessTimes` Map | ❌ 不维护 | worker 只写文件，LRU 由主进程通过文件 mtime 推断 |
| 内容 hash 缓存 | ✅ `contentHashCache` | ❌ 不缓存 | worker 是短生命周期，缓存意义不大 |
| 生成锁 `generatingLocks` | ✅ | ❌ 不需要 | worker 内 `runWithConcurrency` 已串行化每个文件 |
| 缓存目录创建 | ✅ `setDir` | ✅ 启动时 `fs.mkdirSync(cacheDir, { recursive: true })` | 主进程已创建，worker 兜底 |
| 元数据持久化 | ✅ `cache-metadata.json` | ❌ 不写 | worker 退出时主进程通过 `enforceLimitNow` 重建元数据 |

**关键设计**：worker 生成的缩略图文件路径与主进程 ThumbnailGenerator 完全一致（`<cacheDir>/<contentHash>.jpg`），主进程后续 `thumbnail:generate` 单文件调用时若文件已存在则直接返回路径，不会重复生成。

## 三、任务取消与进程清理逻辑

### 3.1 任务取消

每个任务独立维护 `shouldStop` 标志，互不影响（例如取消缩略图不影响正在运行的 pHash）。

```typescript
// worker-entry.ts 模块级
let shouldStopThumbnail = false
let shouldStopPhash = false
let shouldStopDuplicate = false

// 每个任务的处理循环中检查
for (const task of tasks) {
  if (shouldStopThumbnail) break
  await task()
}
```

**取消后的行为**：
- 已处理的数据已写入 DB（事务粒度=单条），不回滚
- 当前批次的 `Promise.all` 等待完成后退出（最多等待 1 个 sharp/ffmpeg 任务，约 200ms-30s）
- 发送 `*_COMPLETE(success=true, message='已取消', processed, total)`，主进程 Promise 正常 resolve

**取消入口**：
- `MediaWorkerManager.stopThumbnailBatch()` / `stopPhashBatch()` / `stopDuplicateMark()`
- 上层（如 `media:cancelPhash` IPC，若有需求）可直接调用

### 3.2 进程清理（dispose）

#### 主动清理（应用退出时）

`index.ts` `performCleanup` 中，在 `scannerWorkerBridge.dispose()` 之后、`dbManager.close()` 之前新增：

```typescript
// 3.6 media worker 清理（带 1s 超时保护）
try {
  await Promise.race([
    this.mediaWorkerBridge.dispose(),
    new Promise<void>((resolve) => setTimeout(resolve, 1000))
  ])
} catch {}
```

`MediaWorkerBridge.dispose()` 内部流程（复用 scanner bridge 模式）：

```
1. postMessage(MEDIA_WORKER_DISPOSE)
2. 等待 500ms（让 worker 关闭 DB + kill 子进程 + exit）
3. try { worker.kill() } catch {}   ← 兜底
4. worker = null
5. 三个 pendingPromise 全部 reject（标记为 '进程已退出'）
6. isRunning* = false
```

#### worker 内部清理（`cleanupAndExit`）

```typescript
function cleanupAndExit(): void {
  // 1. 关闭 DB 连接（同步）
  try { if (db) db.close() } catch {}
  // 2. kill 所有 ffmpeg 子进程（worker 内独立的 process-registry 实例）
  try { killAllProcesses('SIGKILL') } catch {}
  // 3. 退出
  process.exit(0)
}

// 触发点
process.on('exit', cleanupAndExit)
process.on('uncaughtException', (err) => {
  sendEvent({ type: 'WORKER_ERROR', payload: { message: err.message, stack: err.stack } })
  cleanupAndExit()
})
process.on('unhandledRejection', (reason) => {
  sendEvent({ type: 'WORKER_ERROR', payload: { message: String(reason) } })
  cleanupAndExit()
})
```

#### 异常退出兜底

`MediaWorkerBridge.handleWorkerExit(code)`：
- 若 `isThumbnailRunning` → 合成 `THUMBNAIL_COMPLETE(success=false, message='扫描进程异常退出')`
- 若 `isPhashRunning` → 合成 `PHASH_COMPLETE(success=false, ...)`
- 若 `isDuplicateRunning` → 合成 `DUPLICATE_COMPLETE(success=false, ...)`
- 全部 `pendingPromise` reject
- `worker = null`，下次任务时 `ensureWorker` 自动重启

### 3.3 资源互斥与并发控制

| 场景 | 处理方式 |
|---|---|
| 同一任务重复触发（如缩略图批量进行中又触发一次） | `MediaWorkerManager` 内 `isThumbnailRunning` 标志拦截，直接返回 `{ success: false, message: '已有批量任务在运行' }` |
| 缩略图与 pHash 同时运行 | 允许（worker 内部串行处理两条任务队列，避免 sharp 实例并发过多） |
| scanner 与 media worker 同时运行 | 允许（两个独立 utilityProcess，互不干扰） |
| `media:findDuplicates` / `media:findSimilar` 与批量任务同时运行 | 允许（主进程同步计算与 worker 异步计算独立） |
| 应用退出时有任务运行 | `dispose` 先发 `MEDIA_WORKER_DISPOSE`，worker 内 `cleanupAndExit` 不等待任务完成，直接 kill 子进程 + 关闭 DB + 退出 |

### 3.4 与原 `thumbnailsGenerating` 互斥标志的关系

原 `ctx.isThumbnailsGenerating()` / `ctx.setThumbnailsGenerating(v)` 位于 Application 实例，用于防止启动自动扫描与手动 `scanner:start` 触发的批量任务并发。

**保留方式**：
- `generateThumbnailsForUnprocessed` 函数体改为：
  ```typescript
  export async function generateThumbnailsForUnprocessed(ctx: HandlerContext): Promise<void> {
    if (ctx.isThumbnailsGenerating()) return
    ctx.setThumbnailsGenerating(true)
    try {
      await ctx.mediaWorkerManager.startThumbnailBatch({
        cacheDir: ctx.thumbnailGen.getCacheDir(),
        thumbnailQuality: 'standard'
      })
      ctx.getMainWindow()?.webContents.send('media:updated')
    } finally {
      ctx.setThumbnailsGenerating(false)
    }
  }
  ```
- 互斥标志仍在主进程，worker 不感知，避免跨进程同步

### 4.1 上层调用方式不变

| 调用方 | 原调用方式 | 改造后调用方式 | 变化 |
|---|---|---|---|

| `ipc/handlers/media.ts` media:markDuplicates | `await markDuplicates(ctx)` | 完全一致 | 无 |
| 渲染层 | `media:updated` 监听 | 完全一致 + 新增可选 `thumbnail:progress` / `phash:progress` / `duplicate:progress` / `thumbnail:complete` / `phash:complete` / `duplicate:complete` | 新增不破坏旧 |

### 4.2 功能输出完全一致

| 功能 | 原输出 | 改造后输出 | 一致性保证 |
|---|---|---|---|
| 缩略图文件 | `<cacheDir>/<contentHash>.jpg` | 完全一致 | worker 用相同 contentHash 算法 + 相同 sharp pipeline |
| media_files.thumbnail 字段 | 缩略图文件路径字符串 | 完全一致 | UPDATE SQL 相同 |
| media_files.width / height | 图片原始宽高 | 完全一致 | sharp.metadata 相同 |
| media_files.phash 字段 | 64 字符 0/1 串 | 完全一致 | calculatePHash 算法相同 |
| media_files.is_duplicate / original_id | 0/1 + id | 完全一致 | markDuplicates 算法相同 |
| 缩略图缓存目录 | 主进程 ThumbnailGenerator 管理 | 完全一致 | worker 仅写入文件，不管理目录 |
| LRU 淘汰 | 主进程 enforceLimitNow | 完全一致 | worker 退出后主进程照常 LRU |
| 重复分组结果 | markDuplicates 返回 `{markedDuplicates, totalGroups}` | 完全一致 | worker 返回相同结构 |

### 4.3 进程级隔离保证

- media worker 进程崩溃不影响主进程与 scanner worker
- media worker 进程崩溃后，下次任务自动重启（`ensureWorker` 检测 `worker === null`）
- scanner worker 与 media worker 完全独立，无共享状态、无消息往来

### 4.4 不涉及的功能（明确边界）

| 功能 | 是否涉及 | 理由 |
|---|---|---|
| `media:findDuplicates`（sha256 精确重复） | ❌ 不迁移 | 手动触发，同步返回结果，频率低 |
| `media:findSimilar`（pHash 相似查询） | ❌ 不迁移 | 同上 |
| `autoAnalyzeSceneTime`（场景亮度分析） | ❌ 不迁移 | 独立 sharp 任务，本期不涉及 |
| `cleanupAndRepairDatabase` 中的 `processThumbnailForRow` 调用 | ❌ 不迁移 | 数据库修复路径，频率极低 |
| 单文件 `thumbnail:generate` IPC | ❌ 不迁移 | 主进程 ThumbnailGenerator 单文件 generate 保留 |
| `cache:` 系列 IPC | ❌ 不迁移 | 缓存管理保留主进程 |

## 六、风险说明

### 6.1 已识别风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| worker 内 sharp 实例与主进程 ThumbnailGenerator 并发生成同一文件 | 极低概率文件冲突 | 主进程 `generatingLocks` 保留；worker 批量任务由 `thumbnailsGenerating` 互斥标志拦截主进程单文件 generate 的并发（实际单文件 generate 由用户手动触发，频率极低） |
| WAL 多连接并发写入冲突 | 极低（busy_timeout=5000 自动重试） | 已在 scanner worker 验证；media worker 复用相同 PRAGMA |
| worker 异常退出导致 pendingPromise 卡死 | 上层调用方永久 await | bridge `handleWorkerExit` 合成 `*_COMPLETE(success=false)`，强制 resolve |
| worker 启动失败（native 模块加载失败） | 任务无法执行 | `ensureWorker` 抛错 → 上层 catch → logFault 记录 → 用户可见错误 |
| 进程退出时 worker 未及时清理 | 子进程僵尸、单实例锁未释放 | `dispose` 500ms 等待 + `kill()` 兜底 + `performCleanup` 1s 超时 |
| 缩略图文件路径不一致 | 主进程单文件 generate 找不到 worker 生成的文件 | worker 用相同 contentHash 算法 + 相同 cacheDir，路径必然一致 |
| `media:findDuplicates` / `media:findSimilar` 仍在主进程 O(n²) | 手动触发时仍可能卡顿 | 本期不涉及；后续可单独迁移（需改 IPC 为异步推送） |
| 渲染层未订阅新增的 `*:progress` / `*:complete` | 进度信息不被展示 | 不影响功能；渲染层可后续按需订阅 |

### 6.2 不在范围内的事项

1. **不迁移 `media:findDuplicates` / `media:findSimilar`**：这两个 IPC 是手动触发、同步返回结果的查询，迁移需要改 IPC 协议（同步 → 异步推送），超出"上层调用方式不变"的承诺。如需迁移，另立计划。
2. **不迁移 `autoAnalyzeSceneTime`**：独立的场景亮度分析任务，与缩略图/phash 无强耦合，本期不涉及。
3. **不迁移 `cleanupAndRepairDatabase` 中的 `processThumbnailForRow`**：数据库修复路径，频率极低，且与修复事务强耦合。
4. **不改动 ThumbnailGenerator 类**：单文件 generate、LRU、cache: IPC 全部保留主进程。
5. **不新增渲染层 UI**：不增加进度条、取消按钮等 UI 元素（用户未要求）。
6. **不改动 package.json**：版本号 v2.3.0 不变，asarUnpack 配置已覆盖所有原生依赖。
7. **不改动 scanner worker**：两个 worker 完全独立，互不干扰。
