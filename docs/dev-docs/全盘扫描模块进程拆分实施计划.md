# 全盘扫描模块进程拆分实施计划

> 适用版本：wxnn-photo-manager v2.3.0
> 制定日期：2026-07-18
> 涉及代码：`src/main/scanner/`、`src/main/index.ts`、`src/main/preload.ts`（仅类型同步，不改 API）、`src/main/ipc/handlers/misc.ts`（不改逻辑）
> 版本号：**保持 v2.3.0 不变**

---

## 一、拆分目标

将全盘扫描模块从 Electron 主进程中拆分到 `utilityProcess` 独立工具进程中执行，解决以下问题：

1. **扫描期间 UI 卡顿**：`findMediaDirectories` 全盘签名深度搜索、`scanDirectory` 递归遍历、`probeVideoMetadata` 调用 ffprobe、`checkMissingFiles` 万级文件存在性校验等操作都在主进程事件循环中执行，CPU/IO 密集任务阻塞主进程，导致窗口拖动、按钮点击、动画掉帧。
2. **better-sqlite3 同步调用阻塞**：`saveBatchToDatabase` 的事务提交、`repairLegacyData` 的全表 LIKE 扫描、`checkMissingFiles` 的事务更新等同步 DB 调用会阻塞主进程事件循环数百毫秒。
3. **future-proof**：未来扫描逻辑扩展（如 EXIF 解析、AI 标签）若仍在主进程执行，卡顿将更严重。

**核心承诺**：扫描功能 100% 不变，渲染层调用方式完全不变，用户无感知。

## 二、整体架构说明

### 2.1 拆分前架构

```
[渲染进程] ──IPC──► [主进程]
                     ├─ ScannerManager（EventEmitter）
                     │   ├─ findMediaDirectories（CPU 密集）
                     │   ├─ scanDirectory（IO 密集 + 递归）
                     │   ├─ probeVideoMetadata（spawn ffprobe）
                     │   ├─ saveBatchToDatabase（同步事务）
                     │   ├─ checkMissingFiles（万级 fs.access）
                     │   └─ repairLegacyData（全表 LIKE）
                     ├─ DatabaseManager（better-sqlite3 同步连接）
                     └─ ThumbnailGenerator / FileService / ...
```

主进程同时承担 UI 响应、IPC 调度、数据库读写、扫描 CPU/IO 任务，扫描期间必然卡顿。

### 2.2 拆分后架构

```
[渲染进程] ──IPC──► [主进程] ◄──MessagePort──► [utilityProcess: scanner-worker]
                     ├─ ScannerManager（薄壳）                     ├─ 扫描核心逻辑（迁移自 ScannerManager）
                     │   └─ 转发 start/stop/status                  │   ├─ findMediaDirectories
                     │       + 桥接 progress/complete 事件          │   ├─ scanDirectory
                     ├─ DatabaseManager（保留原连接，业务 IPC 用）   │   ├─ probeVideoMetadata
                     ├─ ThumbnailGenerator                          │   ├─ saveBatchToDatabase（独立 DB 连接）
                     ├─ FileService / VideoService / ...            │   ├─ checkMissingFiles
                     └─ ScannerWorkerBridge（进程管理）              │   └─ repairLegacyData
                                                                  └─ 独立 better-sqlite3 连接（WAL 多连接）
```

### 2.3 各方职责

| 角色 | 职责 |
|---|---|
| **渲染进程** | **零改动**。`window.electronAPI.scanner.start/stop/status/onProgress/onComplete` 调用方式完全不变 |
| **主进程** | 1. 接收渲染进程 IPC（`scanner:start/stop/status`）<br>2. 通过 `ScannerWorkerBridge` 管理 worker 进程生命周期<br>3. 转发扫描参数到 worker，接收 worker 推送的进度/完成事件，再通过 `mainWindow.webContents.send` 推送给渲染进程<br>4. 维持业务 IPC 使用的 `DatabaseManager` 连接（与 worker 的连接独立）<br>5. 在 `before-quit` 中显式终止 worker 进程 |
| **工作进程（utilityProcess）** | 1. 接收主进程的 `SCAN_START/STOP` 消息<br>2. 执行全部扫描核心逻辑（迁移自 `ScannerManager`）<br>3. 独立打开 better-sqlite3 连接（同一 db 文件，WAL 模式支持多连接）<br>4. 调用 ffprobe（通过 `resolveAsarUnpackedPath` 解析路径）<br>5. 通过 MessagePort 向主进程推送 `progress/complete/error/log` 事件<br>6. 接收 `SCAN_DISPOSE` 时主动关闭 DB 连接 + kill 子进程 + 退出 |

### 2.4 通信链路

```
渲染进程                主进程                 worker
   │  ipcRenderer          │  MessagePort         │
   │  .invoke              │  .postMessage        │  process.on('message')
   │ ─────────────────────►│ ────────────────────►│
   │  scanner:start        │  SCAN_START          │
   │                       │                      │
   │  webContents.send     │  MessagePort         │  port.postMessage
   │ ◄─────────────────────│ ◄────────────────────│
   │  scanner:progress     │  SCAN_PROGRESS       │
   │  scanner:complete     │  SCAN_COMPLETE       │
```

- 渲染层 ↔ 主进程：保持现有 IPC + `webContents.send` 推送，**接口零改动**
- 主进程 ↔ worker：使用 `utilityProcess.fork()` 返回的 `UtilityProcess` 实例的通信管道（`child.postMessage` / `child.on('message')`），无需 `MessageChannelMain`（仅单管道即可满足需求）

## 四、核心实现细节

### 4.1 工作进程入口与逻辑迁移

#### 4.1.1 入口文件位置

`src/main/scanner/worker-entry.ts`

通过 `utilityProcess.fork(path.join(__dirname, 'worker-entry.js'))` 启动。打包后 `__dirname` 为 `app.asar/dist/main/main/`，与现有 `index.js` 同目录，无需额外路径配置。

#### 4.1.2 核心逻辑迁移方案

将 `ScannerManager`（`src/main/scanner/index.ts`）中以下成员**整体迁移**到 `worker-entry.ts`：

**常量迁移**（保持原值，禁止改值）：
- `MAX_SCAN_DEPTH`、`SCAN_BATCH_SIZE`
- `GAME_SIGNATURES`、`GAME_SIGNATURES_LOWER`
- `MEDIA_FOLDER_SIGNATURES`、`MEDIA_FOLDER_SIGNATURES_LOWER`
- `SYSTEM_DIRS_TO_SKIP`
- `DRIVE_SCAN_CONCURRENCY`

**接口迁移**（保持定义，禁止改字段）：
- `ScanOptions`、`ScanProgress`、`MediaFile`

**方法迁移**（保持实现，禁止改逻辑）：
- `findMediaDirectories`
- `searchBySignature` / `searchBySignatureShallow` / `searchByMediaFolderSignatureShallow`
- `findSteamGamePaths` / `findEpicGamePaths` / `querySteamInstallPath`
- `getAvailableDrives` / `getUserDataPaths`
- `scanDirectory`
- `saveBatchToDatabase`
- `checkMissingFiles`
- `repairLegacyData`
- `insertScanHistory` / `updateScanHistory`
- `cachedRealpath`

**改造点**（仅必要的最小改动）：
1. **DB 连接**：原 `this.db` / `this.dbManager` 注入改为 worker 内部 `new Database(dbPath)`，并配置与主进程一致的 PRAGMA（`journal_mode=WAL` / `synchronous=NORMAL` / `cache_size=-20000` / `temp_store=MEMORY` / `mmap_size=268435456` / `foreign_keys=ON` / `wal_autocheckpoint=1000`）。`schema_migrations` 表在主进程 `DatabaseManager.initialize()` 时已创建，worker 不再执行 migrations。
2. **dbManager 调用**：`repairLegacyData` 中的 `this.dbManager.isMigrationApplied` / `markMigrationApplied` 改为直接查询 `schema_migrations` 表（worker 不持有 `DatabaseManager` 实例，使用降级路径）。
3. **EventEmitter emit**：原 `this.emit('progress', ...)` 改为 `process.parentPort.postMessage({ type: 'SCAN_PROGRESS', payload: ... })`。
4. **logger**：worker 内不能使用主进程的 `logger`（依赖 `app.getPath`），改为通过 `SCAN_LOG` 消息转发到主进程，由主进程 `logger` 统一记录。
5. **shouldStop 标志**：原 `this.shouldStop` 改为模块级变量，由 `SCAN_STOP` 消息处理函数设置。
6. **ffprobe 路径**：直接 `import { ffprobePath } from '../utils/ffmpeg-paths'`，`resolveAsarUnpackedPath` 已包含路径修正。
7. **insertStmtCache / realpathCache**：保持原缓存机制，每次 `SCAN_START` 时清空。

#### 4.1.3 worker 启动流程

```ts
// worker-entry.ts 伪代码
let db: Database.Database | null = null
let shouldStop = false

process.parentPort.on('message', async (msg) => {
  if (msg.type === 'SCAN_START') {
    // 1. 打开 DB（如未打开）
    if (!db) db = openDatabase(msg.payload.dbPath)
    // 2. 执行扫描（迁移自 ScannerManager.startScan）
    const result = await startScan(db, msg.payload.options)
    // 3. 推送完成事件
    process.parentPort.postMessage({ type: 'SCAN_COMPLETE', payload: result })
  } else if (msg.type === 'SCAN_STOP') {
    shouldStop = true
  } else if (msg.type === 'SCAN_DISPOSE') {
    cleanupAndExit()
  }
})
```

### 4.2 IPC 通信消息类型定义

在 `src/main/scanner/worker-protocol.ts` 中定义：

```ts
// ============ 主进程 → worker ============
export type WorkerCommand =
  | { type: 'SCAN_START'; payload: { dbPath: string; options: ScanOptions } }
  | { type: 'SCAN_STOP' }
  | { type: 'SCAN_DISPOSE' }  // 主进程退出前调用，worker 主动清理并退出

// ============ worker → 主进程 ============
export type WorkerEvent =
  | { type: 'SCAN_PROGRESS'; payload: ScanProgress }
  | { type: 'SCAN_COMPLETE'; payload: { success: boolean; message: string; filesFound?: number } }
  | { type: 'SCAN_LOG'; payload: { level: 'info' | 'warn' | 'error'; message: string; args?: unknown[] } }
  | { type: 'WORKER_READY' }  // worker 启动完成，可用于扫描
  | { type: 'WORKER_ERROR'; payload: { message: string; stack?: string } }
```

**消息发送频率约束**：
- `SCAN_PROGRESS` 保持原 500ms 节流（worker 内部节流，不在主进程二次节流）
- `SCAN_LOG` 仅 `warn` / `error` 级别跨进程转发，`info` 级别在 worker 内部丢弃（避免跨进程 IPC 开销）
- `SCAN_COMPLETE` 每次扫描仅发一次
- `WORKER_READY` 仅 worker 启动时发一次

### 4.3 数据库连接策略

| 项目 | 主进程 | worker 进程 |
|---|---|---|
| 连接对象 | `DatabaseManager.db`（已存在） | `new Database(dbPath)`（worker 内创建） |
| 打开时机 | `app.whenReady()` 后 `dbManager.initialize()` | 收到首个 `SCAN_START` 时惰性打开 |
| 关闭时机 | `before-quit` 中 `dbManager.close()` | 收到 `SCAN_DISPOSE` 或 `process.on('exit')` 时关闭 |
| WAL 模式 | 已启用 | 共享同一 WAL 文件（WAL 支持多连接并发读 + 单写） |
| migrations | 主进程独占执行 | worker **不执行 migrations**，仅使用已有 schema |
| 写入并发 | 业务 IPC 偶发写入 | 扫描期间高频写入（500 条/批次事务） |

**并发安全分析**：
- SQLite WAL 模式允许多个连接同时读取，写入通过 WAL 文件串行化
- 主进程业务 IPC 写入极少（用户操作触发），与 worker 扫描写入冲突概率低
- 即便冲突，WAL 的 `busy_timeout` 会自动重试（better-sqlite3 默认 5000ms）
- worker 启动时显式设置 `db.pragma('busy_timeout = 5000')`

**不共享连接的理由**：better-sqlite3 的 `Database` 实例不能跨进程传递（Native Handle 绑定到创建进程），必须各自打开。

### 4.4 主进程侧桥接层

`src/main/scanner/scanner-worker-bridge.ts` 职责：

```ts
export class ScannerWorkerBridge {
  private worker: UtilityProcess | null = null
  private ready: Promise<void> | null = null
  private messageHandlers: ((event: WorkerEvent) => void)[] = []

  /** 惰性启动 worker（首次 startScan 时调用） */
  private async ensureWorker(): Promise<void> {
    if (this.worker && !this.worker.killed) return
    this.worker = utilityProcess.fork(path.join(__dirname, 'worker-entry.js'))
    this.worker.on('message', (msg: WorkerEvent) => {
      this.messageHandlers.forEach((h) => h(msg))
    })
    this.worker.on('exit', (code) => {
      // 异常退出处理（见 4.6）
      this.handleWorkerExit(code)
    })
    // 等待 WORKER_READY
    await this.waitForReady()
  }

  /** 发送扫描指令 */
  async startScan(dbPath: string, options: ScanOptions): Promise<void> {
    await this.ensureWorker()
    this.worker!.postMessage({ type: 'SCAN_START', payload: { dbPath, options } })
  }

  /** 发送停止指令 */
  stopScan(): void {
    this.worker?.postMessage({ type: 'SCAN_STOP' })
  }

  /** 注册 worker 事件回调（progress/complete/log/error） */
  onEvent(handler: (event: WorkerEvent) => void): () => void {
    this.messageHandlers.push(handler)
    return () => {
      this.messageHandlers = this.messageHandlers.filter((h) => h !== handler)
    }
  }

  /** 主进程退出前调用：发送 DISPOSE → 等待 500ms → 强制 terminate */
  async dispose(): Promise<void> {
    if (!this.worker) return
    try {
      this.worker.postMessage({ type: 'SCAN_DISPOSE' })
      // 给 worker 500ms 时间清理（关闭 DB + kill 子进程）
      await new Promise((resolve) => setTimeout(resolve, 500))
    } catch {}
    if (!this.worker.killed) {
      try { this.worker.kill() } catch {}
    }
    this.worker = null
  }
}
```

### 4.5 ScannerManager 改造

改造后的 `ScannerManager` 仅保留接口外壳：

```ts
export class ScannerManager extends EventEmitter {
  private bridge: ScannerWorkerBridge
  private currentStatus: ScanProgress = { scanned: 0, found: 0, currentPath: '', status: 'idle' }
  private unsubscribe: (() => void) | null = null

  constructor(bridge: ScannerWorkerBridge) {
    super()
    this.bridge = bridge
    this.subscribeBridgeEvents()
  }

  private subscribeBridgeEvents(): void {
    this.unsubscribe = this.bridge.onEvent((event) => {
      switch (event.type) {
        case 'SCAN_PROGRESS':
          this.currentStatus = event.payload
          this.emit('progress', { ...event.payload })
          break
        case 'SCAN_COMPLETE':
          this.currentStatus.status = event.payload.success ? 'completed' : 'failed'
          this.emit('complete', event.payload)
          break
        case 'SCAN_LOG':
          // 转发到主进程 logger
          if (event.payload.level === 'error') logger.error(`[ScannerWorker] ${event.payload.message}`)
          else if (event.payload.level === 'warn') logger.warn(`[ScannerWorker] ${event.payload.message}`)
          break
        case 'WORKER_ERROR':
          logger.error(`[ScannerWorker] 异常: ${event.payload.message}`)
          break
      }
    })
  }

  async startScan(options?: ScanOptions): Promise<{ success: boolean; message: string; filesFound?: number }> {
    if (this.currentStatus.status === 'running') {
      return { success: false, message: '扫描正在进行中' }
    }
    this.currentStatus = { scanned: 0, found: 0, currentPath: '', status: 'running' }
    // dbPath 由 Application 注入（通过 setDbPath 或构造参数）
    const dbPath = this.dbPath
    if (!dbPath) return { success: false, message: '数据库路径未初始化' }

    // 发送 SCAN_START，等待 SCAN_COMPLETE
    return new Promise((resolve) => {
      const onceComplete = (result: { success: boolean; message: string; filesFound?: number }) => {
        this.removeListener('complete', onceComplete)
        resolve(result)
      }
      this.on('complete', onceComplete)
      this.bridge.startScan(dbPath, options ?? {}).catch((err) => {
        this.removeListener('complete', onceComplete)
        resolve({ success: false, message: `扫描启动失败: ${err instanceof Error ? err.message : String(err)}` })
      })
    })
  }

  async stopScan(): Promise<{ success: boolean }> {
    this.bridge.stopScan()
    return { success: true }
  }

  getStatus(): ScanProgress {
    return { ...this.currentStatus }
  }

  /** Application 初始化时注入数据库路径 */
  setDbPath(dbPath: string): void {
    this.dbPath = dbPath
  }
}
```

### 4.6 资源清理与异常退出处理

#### 4.6.1 worker 进程内清理

```ts
// worker-entry.ts
function cleanupAndExit(): void {
  try {
    if (db) {
      db.pragma('wal_checkpoint(PASSIVE)')
      db.close()
      db = null
    }
  } catch {}
  // kill 所有 worker 启动的 ffprobe 子进程（通过 process-registry）
  try { killAllProcesses('SIGKILL') } catch {}
  process.exit(0)
}

process.on('message', (msg: WorkerCommand) => {
  if (msg.type === 'SCAN_DISPOSE') {
    cleanupAndExit()
  }
})

// 兜底：parentPort 断开（主进程已退出）时主动清理
process.parentPort.on('close', cleanupAndExit)
process.on('exit', cleanupAndExit)
process.on('uncaughtException', (err) => {
  process.parentPort.postMessage({ type: 'WORKER_ERROR', payload: { message: err.message, stack: err.stack } })
  cleanupAndExit()
})
```

#### 4.6.2 主进程侧清理（`performCleanup` 中）

在现有 `performCleanup` 的步骤 3（`scannerManager.stopScan`）之后新增：

```ts
// 3.1 终止扫描 worker 进程（在 dbManager.close 之前）
try {
  await Promise.race([
    this.scannerWorkerBridge.dispose(),
    new Promise<void>((resolve) => setTimeout(resolve, 1000))  // 超时保护
  ])
} catch {}
```

#### 4.6.3 worker 异常退出处理

`ScannerWorkerBridge` 监听 `worker.on('exit')`：

```ts
private handleWorkerExit(code: number): void {
  if (code !== 0) {
    logger.error(`[ScannerWorker] 异常退出，exit code=${code}`)
  }
  // 若扫描正在进行中，发送一个失败的 SCAN_COMPLETE 事件，避免主进程 ScannerManager 卡在 running 状态
  if (this.currentStatus?.status === 'running') {
    this.messageHandlers.forEach((h) => h({
      type: 'SCAN_COMPLETE',
      payload: { success: false, message: '扫描进程异常退出' }
    }))
  }
  this.worker = null
}
```

#### 4.6.4 主进程崩溃时的兜底

主进程 `uncaughtException` / `before-quit` 强制退出路径中，**不**等待 worker 清理，直接 `app.exit(1)`。worker 通过 `process.parentPort.on('close')` 检测到主进程退出后自行清理。OS 会在主进程退出后回收 worker 进程（utilityProcess 是主进程的子进程）。

## 七、风险说明与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| **R1: utilityProcess API 兼容性** | Electron 28 中 `utilityProcess` 已稳定，但 `process.parentPort` 在不同 Electron 版本行为可能微调 | 使用 `utilityProcess.fork()` + `process.parentPort.on('message')` 标准 API；不依赖任何实验性 API；打包后在目标 Windows 环境实测 |
| **R2: better-sqlite3 native 模块加载** | worker 进程加载 `better-sqlite3` 时，需找到 `app.asar.unpacked` 下的 native 二进制 | `package.json` 的 `asarUnpack` 已配置 `**/better-sqlite3/**`；worker 中 `import Database from 'better-sqlite3'` 会自动解析到 unpacked 路径；若失败，回退到主进程扫描（保留原 `ScannerManager` 实现作为兜底，但本计划不实现兜底，仅在出现问题时回退代码） |
| **R3: WAL 多连接写入冲突** | 主进程业务 IPC 与 worker 扫描同时写入同一 DB，可能触发 `SQLITE_BUSY` | WAL 模式 + `busy_timeout=5000` 自动重试；业务 IPC 写入频率极低（用户操作触发），冲突概率 <0.1%；冲突时 worker 事务自动重试 |
| **R4: worker 进程退出时 WAL 未 checkpoint** | worker 异常退出可能导致 WAL 文件膨胀，影响下次启动 | worker `cleanupAndExit` 中显式 `wal_checkpoint(PASSIVE)` + `db.close()`；主进程下次启动时 `DatabaseManager.initialize` 中已有 WAL/SHM 损坏重试逻辑 |
| **R5: 跨进程 IPC 开销抵消隔离收益** | 高频 `SCAN_PROGRESS` 消息（500ms/次）跨进程传递有序列化开销 | 500ms 节流已在原实现中存在，跨进程 IPC 开销 <1ms/次，可忽略；`SCAN_LOG` 仅 warn/error 跨进程转发，info 级别在 worker 内丢弃 |
| **R6: ffmpeg/ffprobe 子进程在 worker 中无法启动** | `ffprobe-static` 路径在 asar 内，worker 加载时可能找不到 | `resolveAsarUnpackedPath` 已处理路径修正；`asarUnpack` 已包含 `**/ffprobe-static/**`；与现有 `video-probe.ts` 使用同一份 `ffmpeg-paths.ts`，路径解析逻辑一致 |
| **R7: 扫描进度事件丢失或乱序** | MessagePort 是单向 FIFO，理论上不丢失，但需确认 | `SCAN_PROGRESS` 携带 `{scanned, found, currentPath}` 全量字段（非增量），即便丢失某次进度也不影响最终状态；`SCAN_COMPLETE` 仅发一次，必达 |
| **R8: worker 在开发模式（`npm run dev`）下路径不同** | 开发模式 `__dirname` 为 `dist/main/main/`，worker-entry.js 同目录 | `utilityProcess.fork(path.join(__dirname, 'worker-entry.js'))` 在开发和打包环境下路径解析一致 |
| **R9: 主进程 `before-quit` 超时强制退出时 worker 未清理** | `performCleanup` 有 2-5s 超时，若 worker `dispose()` 耗时过长可能被强制中断 | `dispose()` 内部 500ms 等待 + 1s 超时保护（见 4.6.2），总耗时 <1.5s；worker 收到 DISPOSE 后自行清理，即便主进程强制退出，worker 也会通过 `parentPort.on('close')` 检测后清理 |
| **R10: `scan_history` 写入时机变化** | 拆分前 `insertScanHistory` 在主进程同步执行，拆分后在 worker 中执行 | worker 与主进程共享同一 db 文件，`scan_history` 表记录写入时机一致；主进程通过同一 db 读取 `scan_history` 时数据已提交 |
| **R11: 渲染层 `scanner:status` 调用时机** | `ScanButton` 挂载时调用 `scanner:status` 同步状态；若此时 worker 未启动，`ScannerManager.currentStatus` 为 idle | `ScannerManager.getStatus` 返回 `currentStatus`，初始为 `{status: 'idle'}`，与拆分前行为一致；worker 未启动时不影响 status 查询 |
| **R12: `misc.ts` 中 `scanner:start` 的 `generateThumbnailsForUnprocessed` 调用** | 拆分后扫描在 worker 中完成，但缩略图生成仍在主进程 | `misc.ts` 不改动，`scanner:start` handler 调用 `ctx.scannerManager.startScan` 后根据返回值 `result.success && result.filesFound > 0` 触发 `generateThumbnailsForUnprocessed`，与拆分前一致 |
