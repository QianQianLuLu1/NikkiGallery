## 2. 分维度问题与优化/删改方案

### 2.1 底层架构

#### 2.1.1 严重问题

**【严重-A1】4 组核心函数在 index.ts 与 handlers/ 存在双份活跃副本**
- **位置**：
  - `src/main/index.ts:1264` `generateThumbnailsForUnprocessed` ↔ `src/main/ipc/handlers/misc.ts:78`
  - `src/main/index.ts:1300` `generatePhashForUnprocessed` ↔ `src/main/ipc/handlers/media.ts:39`
  - `src/main/index.ts:1348` `markDuplicates` ↔ `src/main/ipc/handlers/media.ts:87`
  - `src/main/index.ts:1226` `processThumbnailForRow` ↔ `src/main/ipc/handlers/misc.ts:35`
- **根因**：P0-A1 重构将 IPC handler 从 `Application` 类迁移到独立文件时采用「复制」而非「移动」策略。原 index.ts 中的私有方法保留供启动流程（`performAutoScan`）使用，新 handler 文件创建接受 `HandlerContext` 的自由函数副本。两组副本当前都在被活跃调用，逻辑相同。**注意**：misc.ts 版本通过 `ctx.isThumbnailsGenerating()`/`ctx.setThumbnailsGenerating()` 回调共享 Application 实例的互斥标志，因此两份代码不是完全独立的，但代码重复仍导致维护时需同步修改两处。
- **删改建议**：抽取到 `src/main/services/thumbnail-phash-service.ts`，导出 `generateThumbnailsForUnprocessed(ctx)`、`generatePhashForUnprocessed(ctx)`、`markDuplicates(ctx)`、`processThumbnailForRow(ctx, row)` 四个函数。index.ts 的 `performAutoScan` 改为调用共享模块；misc.ts/media.ts 已有的自由函数改为 re-export 或直接调用共享模块；删除 index.ts 中的 4 个私有方法副本。
- **预期收益**：消除约 200 行重复代码；修复 bug 时只需改一处；避免两份代码逐渐分叉导致启动路径与 IPC 路径行为不一致。

**【严重-A2】backup-service 数据库恢复非原子操作，中途失败导致数据永久丢失**
- **位置**：`src/main/services/backup-service.ts:188`
- **根因**：`restoreBackup` 直接使用 `fs.copyFileSync(backupPath, dbPath)` 覆盖当前数据库文件。`copyFileSync` 是非原子操作——若中途因磁盘空间不足、权限变更、进程异常终止等原因失败，`dbPath` 处于「部分写入」的损坏状态。此时原数据库已被覆盖、WAL/SHM 已删除（191-192 行），用户既无法使用当前数据库，也无法自动回滚。
- **删改建议**：改为「先复制到临时文件 → 校验完整性 → 原子 rename 覆盖」三步式：
  ```typescript
  const tmpPath = dbPath + '.restoring'
  fs.copyFileSync(backupPath, tmpPath)
  // 用 better-sqlite3 临时以只读模式打开 tmpPath 执行 PRAGMA integrity_check
  fs.renameSync(tmpPath, dbPath)  // 同盘 rename 是原子操作
  ```
- **预期收益**：消除数据库恢复过程中的数据损坏风险；恢复失败时原数据库不受影响，用户可重试或选择其他备份。

**【严重-A3】backup-service 恢复前不校验备份文件完整性**
- **位置**：`src/main/services/backup-service.ts:175`（仅 `existsSync`，未校验是否为有效 SQLite）
- **根因**：`restoreBackup` 仅检查备份文件是否存在，不验证其是否为有效的 SQLite 数据库。若备份文件因磁盘错误、不完整写入等原因损坏，恢复后应用无法启动（better-sqlite3 打开损坏数据库会抛错），且由于 A2 的非原子覆盖，原数据库也已丢失。
- **删改建议**：恢复前用 better-sqlite3 临时以只读模式打开备份文件，执行 `PRAGMA integrity_check` 和 `PRAGMA quick_check`，通过后才执行恢复。
- **预期收益**：防止用户因恢复损坏备份而丢失可用数据库；在 UI 层提前给出「备份文件已损坏」的明确提示。

**【严重-A4】watermark:apply handler 完全缺失参数校验**
- **位置**：`src/main/ipc/handlers/watermark.ts:10-24`
- **根因**：`watermark:apply` 接收 `config: unknown`、`filePaths: string[]`、`targetDir: string` 三个参数，但未调用任何校验函数：未调用 `validateFilePathArray(filePaths)`、未调用 `validateFilePath(targetDir) + validateNonSensitivePath(targetDir)`、`config` 直接 `as WatermarkConfig` 强制转换无结构校验。对比同类写操作 handler（`file:copy`/`file:move`/`file:export`）均完整调用了上述校验。这是 P1-A5 安全加固时的遗漏。
- **删改建议**：补充与其他写操作 handler 一致的三层校验：
  ```typescript
  const v1 = validateFilePathArray(filePaths)
  if (!v1.valid) return { success: false, message: v1.message }
  const v2 = validateFilePath(targetDir)
  if (!v2.valid) return { success: false, message: v2.message }
  const s = validateNonSensitivePath(targetDir)
  if (!s.valid) return { success: false, message: s.message }
  ```
- **预期收益**：消除路径遍历和敏感目录写入风险；与项目其他写操作保持一致的安全基线。

**【严重-A5】video 域 4 个写操作 handler 缺失 validateNonSensitivePath**
- **位置**：
  - `src/main/ipc/handlers/video.ts:50-59` `video:export`
  - `src/main/ipc/handlers/video.ts:76-88` `video:trim`
  - `src/main/ipc/handlers/video.ts:92-99` `video:changeSpeed`
  - `src/main/ipc/handlers/video.ts:103-113` `video:exportLivePhoto`
- **根因**：这四个 handler 都使用 `validateFilePath(targetDir)` 校验了路径合法性和遍历，但均未调用 `validateNonSensitivePath(targetDir)`。虽然 `file:export` 对同类操作做了敏感目录检查，视频域 handler 在 P1-A5 加固时被遗漏。渲染进程被入侵后可将视频导出到 `C:\Program Files\` 等敏感目录。
- **删改建议**：在四个 handler 的 `validateFilePath(targetDir)` 之后补充 `validateNonSensitivePath(targetDir)` 校验。
- **预期收益**：统一写操作安全基线；防止恶意/异常调用向系统目录写入媒体文件。

**【严重-A6】editor-service 备份文件永不清理，磁盘空间隐性泄漏**
- **位置**：`src/main/services/editor-service.ts:72-74`（备份存储在 backupService 目录）+ `src/main/services/backup-service.ts:229-241`（`pruneOldBackups` 仅匹配 `wxnn_photo_manager_*.db`）
- **根因**：`EditorService.save` 在写入前将原图备份到 `backupService.getBackupDirectory()`，文件名为 `{mediaId}_{timestamp}.{ext}`。但 `BackupService.pruneOldBackups` 的清理正则是 `/^wxnn_photo_manager_.*\.db$/`，完全不匹配编辑器备份的文件名模式。每次编辑保存都会在备份目录留下一个原图副本（高分辨率照片可达 5-20MB），永远不会被自动清理，也不会出现在备份列表 UI 中。
- **删改建议**：方案一（推荐）：编辑器备份使用独立子目录 `backups/editor-snapshots/`，在 `editor-service.ts` 中实现独立的 LRU 清理逻辑（如保留最近 50 个）。方案二：修改 `pruneOldBackups` 同时匹配编辑器备份文件名模式。
- **预期收益**：防止频繁编辑操作导致磁盘空间隐性泄漏；用户可在 UI 中看到并管理编辑器备份。

**【严重-A7】hasPathTraversal 函数实际无效，提供虚假安全感**
- **位置**：`src/main/utils/ipc-validate.ts:35-41`
- **根因**：函数实现为 `path.resolve(p)` 后检测路径段是否含 `..`。但 `path.resolve` 的核心行为就是展开所有 `..` 段——输入 `C:\foo\..\..\windows\system32` 会 resolve 为 `C:\windows\system32`，不再包含 `..` 段。因此 `segments.includes('..')` 永远返回 `false`（除非输入本身就是异常的混合形式）。**注释已说明"检测异常情况"**，防护范围有限，仅能检测极罕见的混合形式路径。当前安全主要依赖 `validateNonSensitivePath` 的黑名单兜底，但黑名单仅覆盖 5 个固定路径，无法防御如 `C:\Users\OtherUser\...` 等越权访问。
- **删改建议**：移除此低效检查（对绝大多数输入返回 false，徒增函数调用开销）；或在需要预期根目录的场景改为 `isPathWithin(filePath, allowedRoot)` 模式提供真实防护。在注释中明确说明当前依赖 `validateNonSensitivePath` 兜底。
- **预期收益**：消除虚假安全感；避免后续开发者误以为已做路径遍历防护而放松警惕。

**【严重-A8】validateNonSensitivePath 的 startsWith 缺失分隔符边界**
- **位置**：`src/main/utils/ipc-validate.ts:98`
- **根因**：`normalized.startsWith(d.toLowerCase())` 会产生前缀误匹配。例如敏感目录 `C:\Program Files` 会同时阻止 `C:\Program Files (x86)\foo`（正确）和 `C:\Program Filesabc`（错误——这是不同的合法目录）。当前 `SYSTEM_SENSITIVE_DIRS` 恰好都带分隔符或括号，不会触发误匹配，但这是脆弱检查。
- **删改建议**：改为精确的目录边界检查：
  ```typescript
  const d = d.toLowerCase()
  if (normalized === d || normalized.startsWith(d + path.sep)) {
    return { valid: false, ... }
  }
  ```
- **预期收益**：消除前缀误匹配风险；使检查语义精确为「目录或其子目录」。

**【严重-A9】ffprobePath 对 undefined 无回退处理**
- **位置**：`src/main/utils/ffmpeg-paths.ts:27-32`
- **根因**：`ffmpegPath` 对 `ffmpegStatic` 可能为 `undefined` 做了防御（回退到 PATH 中的 `'ffmpeg'`），但 `ffprobePath` 直接访问 `ffprobeStatic.path` 无任何检查。如果 `ffprobe-static` 包安装异常或路径解析失败，`resolveAsarUnpackedPath(undefined)` 会将 `undefined` 转为字符串 `'undefined'` 并返回，导致 ffprobe 调用失败且错误信息不明确。
- **删改建议**：
  ```typescript
  export const ffprobePath: string = ffprobeStatic?.path
    ? resolveAsarUnpackedPath(ffprobeStatic.path)
    : 'ffprobe'
  ```
- **预期收益**：在 ffprobe-static 包异常时给出明确的回退行为，与 ffmpegPath 保持一致的防御策略。

**【严重-A10】backup-service 三个方法使用同步 fs API 阻塞主进程**
- **位置**：`src/main/services/backup-service.ts:139-164`（`listBackups`）、`213-225`（`deleteBackup`）、`229-241`（`pruneOldBackups`）
- **根因**：使用 `fs.existsSync`、`fs.readdirSync`、`fs.statSync` 三个同步 API。该方法被 IPC handler `backup:list` 直接调用阻塞主进程事件循环；网络挂载目录或慢速磁盘场景下同步 `statSync` 可能导致可感知的卡顿。与项目其他模块（log-service、crash-service、file-service）统一使用 `fsp` 异步 API 的模式不一致。
- **删改建议**：将三个方法改为 `async`，使用 `fsp` API。`pruneOldBackups` 在 `createBackup` 中用 `await` 调用。
- **预期收益**：消除主进程阻塞风险；与项目其他 service 的异步模式统一。

**【严重-A11】8 个 setDir/resetDir handler 完全重复**
- **位置**：
  - `backup.ts:74-98` / `cache.ts:67-91` / `crash.ts:64-88` / `log.ts:141-165`（setDir × 4）
  - `backup.ts:101-116` / `cache.ts:94-109` / `crash.ts:91-106` / `log.ts:168-183`（resetDir × 4）
- **根因**：8 个 handler 的逻辑完全相同，仅 `SETTING_KEYS` 的 key 不同和返回的 service `getDir` 方法不同。每个 `setDir` 都是：校验 dir 是字符串 → mkdir → 写测试文件 → unlink 测试文件 → `setSetting` → 返回 `needRestart`。共约 120 行重复代码。
- **删改建议**：在 `utils/dir-manager.ts` 或新建 `utils/ipc-dir-helpers.ts` 中提取两个工厂函数：
  ```typescript
  export function makeSetDirHandler(ctx: HandlerContext, settingKey: string) { ... }
  export function makeResetDirHandler(ctx: HandlerContext, settingKey: string) { ... }
  ```
  各 handler 文件改为 `ipcMain.handle('backup:setDir', makeSetDirHandler(ctx, SETTING_KEYS.backupDir))`。
- **预期收益**：消除约 100 行重复代码；修改校验逻辑只需改一处。
