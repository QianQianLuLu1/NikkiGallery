### 1.2 风险点识别

| 风险 ID | 等级 | 描述 | 缓解方案 |
|---------|------|------|----------|
| R1 | **高** | JSON 实际结构未验证。当前 L246 仅 log 了 `step1Text.slice(0, 120)` 前缀和 length，完整字段名和层级未确认 | P0-T1 先用 `logger.info` 输出完整 JSON 文本（与现有 L246 风格一致，不加 `process.env` 条件判断），确认结构后再编写解析器 |
| R2 | **中** | CameraParams 存在 V1/V2 两种格式。V2 含 `momo` 字段（大喵隐藏），当前解析器仅处理 V1（31 元素） | 解析器增加版本检测：数组长度 >31 时按 V2 处理 |
| R3 | **中** | `portrait_mode` 数据源不一致。当前从 CameraParams arr[1] 读取，但 nikki_albums 从 JSON 的 `portrait_mode_handler.portrait_mode` 读取 | 优先从 JSON `portrait_mode_handler` 读取，arr[1] 作为兜底 |
| R4 | **中** | 枚举映射表（cloth_type/weather_type/pose_id/light_id/filter_id）值未知 | 初期显示原始 ID/数值，通过实际照片逐步收集映射关系 |
| R5 | **中** | 坐标→地名映射。nikki_albums 的 `parse_location` 在 Rust 二进制中（未开源） | 初期显示原始坐标，后续通过社区数据或逆向建立映射表 |
| R6 | **低** | 多面板重复调用 `useGameParams`。5 个新面板各自调用同一 hook，产生 5 次 IPC 请求 | LRU 缓存自动去重（首次请求后其余命中缓存），或改为父组件统一获取后 props 传递 |
| R7 | **低** | `ClockInPhoto` 和 `Collage` 类型 JSON 结构与 `NikkiPhoto` 不同 | 本计划仅处理 `NikkiPhoto` 类型，其他类型后续迭代 |
| R8 | **低** | `handleCopyAll` 需扩展。当前仅复制基础信息 + EXIF | 扩展为包含全部面板数据的格式化文本 |

## 二、架构设计

### 2.1 数据流改造

```
改造前（当前）:
  文件 → DLL Step1(UID Key) → step1Text(JSON)
       → 正则提取 CameraParams Base64 → DLL Step3(Cam Key) → parseCameraParams(17字段)
       → DecryptionResult { camera, hasParams }

改造后（目标）:
  文件 → DLL Step1(UID Key) → step1Text(JSON)
       ├→ JSON.parse(step1Text) → SocialPhoto 对象
       │    ├→ parsePhotographyInfo(socialPhoto)  → PhotographyInfo
       │    ├→ parseNikkiInfo(socialPhoto)        → NikkiParams
       │    ├→ parseDressing(socialPhoto)         → DressingParams
       │    └→ parseInteractions(socialPhoto)     → InteractionParams
       │
       └→ 正则提取 CameraParams Base64 → DLL Step3(Cam Key) → parseCameraParams
            → mergeRichCamera(cameraArr, socialPhoto.photo_info) → RichCameraParams

  最终返回: DecryptionResult {
    camera: RichCameraParams,       // 富相机参数（含 zoom/rotation/pose）
    photography: PhotographyInfo,    // 拍摄信息（扩展现有类型）
    nikki: NikkiParams,            // 暖暖信息
    dressing: DressingParams,      // 搭配信息
    interactions: InteractionParams,// 交互物信息
    hasParams: true
  }
```

### 2.3 缩放计算公式

参考 nikki_albums `converter.rs` L186-192：

```typescript
function calculateZoom(photoInfo: SocialPhotoPhotoInfo): number {
  const dx = photoInfo.nikki_loc_x - photoInfo.camera_actor_loc_x
  const dy = photoInfo.nikki_loc_y - photoInfo.camera_actor_loc_y
  const dz = photoInfo.nikki_loc_z - photoInfo.camera_actor_loc_z
  const distance = Math.sqrt(dx * dx + dy * dy + dz * dz)
  return -0.0035 * distance + 6.45
}
```

#### P0-T3: 解析器实现

| 属性 | 值 |
|------|-----|
| **文件** | `src/main/services/decryption-service.ts` |
| **位置** | L161-202 区域（`parseCameraParams` 附近） |
| **内容** | 新增 4 个解析函数 + 重构 `decodeCameraParamsFromFile` |
| **依赖** | P0-T1, P0-T2 |
| **验收** | exe 运行后解密 NikkiPhoto 照片，返回完整的 DecryptionResult（含 photography/nikki/dressing/interactions） |

**新增函数**:

1. `parseSocialPhoto(step1Text: string): SocialPhotoJSON | null`
   - `JSON.parse(step1Text)` 解析完整 JSON
   - 防御性检查：解析失败返回 null
   - 返回 `SocialPhotoJSON`（初期为 `Record<string, unknown>`，P0-T1 确认后补充精确字段）

2. `parsePhotographyInfo(socialPhoto: SocialPhotoJSON): PhotographyInfo`
   - 提取 edit_photo_handler → EditPhotoState
   - 提取 social_photo.time → date/time（字段名使用 `minute`/`second`，与现有类型一致）
   - 提取 social_photo.photo_info.nikki_loc_x/y/z → LocationInfo
   - 提取 social_photo.weather_type → weather
   - 提取 photo_wall_plugin → photoWall
   - 提取 puzzle_game_plugin / risk_photo / interactive_photo → tasks

3. `parseNikkiParams(socialPhoto: SocialPhotoJSON): NikkiParams`
   - 提取 social_photo.giant_state → giantState
   - 提取 social_photo.photo_info.nikki_hidden → hidden
   - 提取 social_photo.photo_info.nikki_loc/rot/scale

4. `parseDressingParams(socialPhoto: SocialPhotoJSON): DressingParams`
   - 提取 social_photo.photo_info.nikki_clothes → ClothParams[]
   - 提取 social_photo.photo_info.magicball_color_ids → EurekaParams[]

5. `parseInteractionParams(socialPhoto: SocialPhotoJSON): InteractionParams`
   - 提取 social_photo.mount_info → mount
   - 提取 social_photo.carrier_info → carrier
   - 提取 social_photo.interactions → interactions[]

6. `mergeRichCameraParams(cameraArr: CameraParams, socialPhoto: SocialPhotoJSON): RichCameraParams`
   - 合并 CameraParams 数组数据 + photo_info 数据
   - 计算 zoom（缩放公式）
   - 提取 rotation（camera_actor_rot_roll）
   - 提取 pose（pose_id）
   - portraitMode 优先从 portrait_mode_handler 读取（覆盖 arr[1] 的值，不新增字段）

**重构 `decodeCameraParamsFromFile`**:
```typescript
function decodeCameraParamsFromFile(
  fns: BoundFunctions,
  fileData: Buffer,
  uidKey: koffi.PointerObject,
  camKey: koffi.PointerObject
): DecryptionResult {
  // Step 1-4: 保持不变（获取 step1Text + camera 数组解析）
  // ...

  // 新增 Step 1.5: 完整 JSON 解析
  const socialPhoto = parseSocialPhoto(step1Text)

  // Step 5: 合并 RichCameraParams
  const richCamera = socialPhoto
    ? mergeRichCameraParams(camera, socialPhoto)
    : camera  // fallback: JSON 解析失败时使用原始 CameraParams

  // Step 6: 解析其他信息
  const photography = socialPhoto ? parsePhotographyInfo(socialPhoto) : undefined
  const nikki = socialPhoto ? parseNikkiParams(socialPhoto) : undefined
  const dressing = socialPhoto ? parseDressingParams(socialPhoto) : undefined
  const interactions = socialPhoto ? parseInteractionParams(socialPhoto) : undefined

  return {
    hasParams: true,
    camera: richCamera,
    photography,
    nikki,
    dressing,
    interactions,
  }
}
```

**防御性设计原则**:
- 每个解析函数独立 try-catch，单个字段解析失败不影响其他字段
- JSON.parse 失败时 fallback 到原有逻辑（仅返回 CameraParams）
- Optional 字段用 `?.` 链式访问，不存在时返回 null/undefined

## 六、注意事项

1. **不修改版本号**：按项目约定，不修改 `package.json` 版本号
2. **HTML 与 exe 同步**：所有 UI 改动必须同步到 `preview.html`
3. **Import 更新**: `GameParamsPanel.tsx` 重命名为 `CameraInfoPanel.tsx` 后，直接更新 3 处 import（`PropertiesDialog.tsx` L4、`DetailPage.tsx` L15、`FullscreenViewer.tsx` L13），不保留别名导出
4. **防御性解析**：所有 JSON 字段解析使用 `?.` 和 `try-catch`，单字段失败不阻断整体
5. **DLL 串行化**：现有 `withMutex` 保护仍然适用，新增的 JSON.parse 在内存中执行，不涉及 DLL 调用
6. **日志安全**：JSON 日志中可能包含 UID 等信息，生产环境关闭完整 JSON 日志
7. **缓存兼容**：扩展 `GameParamsData` 类型后，旧缓存条目缺少新字段。缓存 TTL 5 分钟后自动过期，无需手动清理
