> 项目位置：`./wxnn-photo-manager/`

## 先了解一个大背景：什么是 Electron？

这个软件是用 **Electron** 框架做的。Electron 把一个软件分成两个互相通信的部分：

- **主进程（main process）**：可以理解为「程序后台」。它能直接读写电脑硬盘、调用系统功能（比如读注册表、调用 ffmpeg 处理视频）。在 Node.js 环境运行。
- **渲染进程（renderer process）**：可以理解为「程序界面」。就是你看到的窗口、按钮、图片缩略图。它跑在一个浏览器内核里，用 React 写。
- **IPC（Inter-Process Communication，进程间通信）**：后台和界面不能直接互调函数，必须通过 IPC 这个「写信通道」互相发消息。比如界面想让后台扫描文件，就发一封名为 `scanner:start` 的信，后台收到后干活，干完再发一封 `scanner:complete` 的信告诉界面「我干完了」。

记住这三件事，后面的内容就很好懂了。

## 二、核心功能流程（两条链路）

### 2.1 链路一：用户点击「自动定位游戏目录」后发生了什么？

**目标**：程序自动找到《无限暖暖》存在哪个文件夹，把里面的图片视频列出来。

**用大白话讲一遍流程**：

1. **用户点击按钮**
   - 按钮文件：`src/renderer/components/scanner/ScanButton.tsx`
   - 用户在下拉菜单点「自动定位游戏目录」，触发 `handleScan('full')` 函数。

2. **界面给后台发一封信**
   - 调用 `window.electronAPI.scanner.start({ fullScan: true })`。
   - 这个 `scanner.start` 实际上是发了一封名为 `scanner:start` 的 IPC 信。
   - 信封格式定义在：`src/main/preload.ts`（这个文件是「界面与后台之间的桥」）。

3. **后台收到信，开始找游戏目录**
   - 收信文件：`src/main/ipc/handlers/misc.ts` 第 34-80 行
   - 后台把活派给扫描器主管：`ctx.scannerManager.startScan(options)`
   - 主管文件：`src/main/scanner/index.ts` 的 `ScannerManager` 类

4. **扫描器主管怎么找到游戏目录？**
   主管会按下面顺序找（在 `findMediaDirectories` 函数里，第 552 行）：

   - **第 1 步：查注册表找 Steam**
     - 函数：`findSteamGamePaths`（第 713 行）
     - 读 Windows 注册表里 Steam 的安装位置，再读 `libraryfolders.vdf` 文件列出所有游戏库，看里面有没有 `InfinityNikki` 文件夹。

   - **第 2 步：查注册表找 Epic**
     - 函数：`findEpicGamePaths`（第 768 行）
     - 读注册表里 Epic Games 的安装位置，看里面有没有 `InfinityNikki` 文件夹。

   - **第 3 步：查用户数据目录**
     - 函数：`getUserDataPaths`（第 900 行）
     - 看 `%LOCALAPPDATA%\InfinityNikki` 和 `文档\InfinityNikki` 这两个常见位置。

   - **第 4 步：如果前面都找不到，全盘搜索**
     - 函数：`searchByMediaFolderSignatureShallow`（第 683 行）
     - 把 C 盘到 Z 盘所有存在的盘符都遍历一遍（最多 4 个盘符同时扫，叫并发），找名字叫 `NikkiPhotos_HighQuality`、`MagazinePhotos`、`ClockInPhoto` 等 16 种「游戏专属文件夹名」（叫**签名**，就是文件名特征）的目录。
     - 注意：`ScreenShot` 这个名字太通用（其他游戏也有），所以扫到它还要额外验证路径里是不是真的有 `infinitynikki` 或 `x6game` 字样，防止误判。

   - **找到后过滤**：只保留路径里含 `infinitynikki` 的目录。

5. **找到目录后，开始递归扫描里面的文件**
   - 函数：`scanDirectory`（第 937 行）
   - 一层一层进文件夹，遇到 `.jpg/.png/.mp4` 等图片视频扩展名就记录下来。
   - **限制最多 15 层深度**（`MAX_SCAN_DEPTH = 15`），防止钻太深卡死。
   - **跳过系统目录**（Windows、ProgramData、$recycle.bin、onedrive 等云同步目录）。
   - **跳过符号链接环**（防止 A 指向 B、B 又指向 A 把程序绕死）。
   - **进度每 500 毫秒报告一次**（不能每扫一个文件就报告，否则界面会卡）。

6. **扫描过程中，文件边扫边写进数据库**
   - 函数：`saveBatchToDatabase`（第 1125 行）
   - **不是扫完才写，而是每攒满 500 条就写一批**（叫「流式批量写入」，降低内存占用）。
   - 用的是 SQLite 数据库（本地文件数据库，不用联网）。
   - 写入位置：`{用户数据目录}/database/wxnn_photo_manager.db`
   - 表名：`media_files`，每张图片视频一行记录，存路径、大小、宽高、账号 UID、相册类型等。
   - **如果同一文件路径已存在，就更新除「场景时段」「套装」外的字段**（这两个字段保留用户手动设置，不覆盖）。

7. **后台一边扫，一边给界面发进度**
   - 后台每 500ms 调一次 `this.emit('progress', ...)`
   - 主入口 `src/main/index.ts` 订阅这个事件，转发给界面：`webContents.send('scanner:progress', progress)`
   - 界面在 `ScanButton.tsx` 里订阅，更新 `scanProgress` 状态，进度浮窗就会跟着动。

8. **扫描完成后，发「完成信」**
   - 后台在 `misc.ts` 第 77 行发 `scanner:complete` 事件。
   - 界面收到后，调 `loadMediaFromDatabase()` 从数据库把媒体列表读出来，更新图库显示。

**整个链路时序图**（简化版）：

```
[用户点按钮] ScanButton.tsx
      ↓ scanner.start({ fullScan: true })  ← 发 IPC 信
[后台收信] misc.ts → scannerManager.startScan()
      ↓ findMediaDirectories() ← 找游戏目录
      ↓   ├─ findSteamGamePaths()  查 Steam 注册表
      ↓   ├─ findEpicGamePaths()   查 Epic 注册表
      ↓   ├─ getUserDataPaths()    查 AppData
      ↓   └─ searchByMediaFolderSignatureShallow()  全盘找签名文件夹
      ↓ scanDirectory()  ← 递归扫文件
      ↓   每 500ms 发一次 progress ← 进度推送
      ↓ saveBatchToDatabase()  ← 每 500 条写一批
      ↓ checkMissingFiles()  ← 顺带检查文件是否还在
[扫描结束]
      ↓ 发 scanner:complete 事件
[界面收信] ScanButton.tsx
      ↓ loadMediaFromDatabase()  ← 从数据库读列表
      ↓ setMediaFiles(res.files)  ← 更新图库显示
[用户看到截图]
```

### 2.2 链路二：用户打开图片编辑器后，调色修改怎么生效并保存？

**目标**：用户拖动亮度滑块，图片要实时变亮；点保存后，原图要被备份，新图要写到硬盘。

**架构特点**：调色计算**全部在前端 JS 里做**（用 Canvas，不依赖服务器），保存时才交给后台用 sharp 库写到硬盘。这样拖滑块时是即时的，不会卡。

#### 调色怎么生效（实时预览）

1. **打开编辑器**
   - 页面文件：`src/renderer/pages/EditorPage.tsx`
   - 通过 `useImageProcessor` hook（自定义钩子函数）拿到所有调色状态和修改函数。

2. **用户拖动亮度滑块**
   - 滑块组件：`src/renderer/components/common/SliderControl.tsx`
   - 拖动时触发 `onChange` → 调 `updateParam('brightness', value)`

3. **`updateParam` 修改状态**
   - 文件：`src/renderer/hooks/useImageProcessor.ts` 第 173 行
   - 调 `setParams({ ...prev, brightness: value })` 更新状态。

4. **状态变化自动触发重新渲染预览（80ms 防抖）**
   - `useImageProcessor.ts` 第 78-105 行
   - **防抖**（debounce）= 连续操作时只在最后一次操作后等 80ms 才真正执行，避免拖滑块时每动一下都重算一次卡死。
   - 调 `processImageData(img, mergedParams, { maxSize: 预览尺寸, watermark })`。

5. **`processImageData` 是图像处理的总入口**
   - 文件：`src/renderer/utils/imageProcessor.ts` 第 490 行
   - 做法：
     - 用 `<canvas>` 把原图画出来
     - 用 `ctx.getImageData()` 拿到所有像素的 RGB 数据（一个超大的数组，每 4 个数代表一个像素的红绿蓝+透明度）
     - **用一个 for 循环遍历每个像素**，依次应用：亮度→对比度→高光阴影→色温色调→饱和度→自然饱和度→HSL→曲线→色调分离→去雾
     - 如果有 LUT（颜色查找表，相当于给图片套一层滤镜），用 `applyLut3D` 应用
     - 如果有清晰度/锐化/降噪，用卷积（一种像素邻域运算）处理，做了**内存优化**：用两块缓冲区交替使用（ping-pong），4096² 大图峰值内存从 448MB 降到 192MB
     - 如果有暗角/颗粒/褪色，再叠加
     - 如果有水印，最后画上去
     - 用 `ctx.putImageData()` 把处理后的像素画回 canvas
   - 处理完后转成 dataURL（一种把图片数据塞进字符串的格式）赋给 `<img>` 显示。

6. **关键算法函数清单**
   - `buildCurveMap`（第 244 行）：把曲线点转成 256 项查找表（这样不用每个像素都重算曲线，速度快）
   - `applyHSL`（第 282 行）：12 色 HSL 调整
   - `applyClarity`（第 350 行）：清晰度（3×3 反锐化）
   - `applySharpen`（第 380 行）：锐化（卷积核）
   - `applySplitTone`（第 449 行）：色调分离
   - `applyWatermark`（第 882 行）：水印总入口
   - `drawTextWatermark`（第 764 行）：文字水印
   - `drawImageWatermark`（第 824 行）：图片水印
   - `drawPolaroidFrame`（第 912 行）：拍立得白边框

#### 保存怎么落盘

1. **用户点「保存」**
   - 函数：`EditorPage.tsx` 第 186 行 `handleSave`
   - 先调 `exportDataUrl(true)`（`true` 表示全尺寸 4096px，预览时是小尺寸）
   - 再调 `window.electronAPI.editor.save(filePath, dataUrl, { format, quality: 95 })` 发 IPC 信

2. **后台收信**
   - 文件：`src/main/ipc/handlers/editor.ts` 第 20 行
   - 校验参数后委托给 `editorService.save()`

3. **`editorService.save` 的五步容灾保存**
   - 文件：`src/main/services/editor-service.ts` 第 104-199 行
   - **第 1 步：安全校验**——确认这个文件路径在 `media_files` 表里已存在，防止有人用 IPC 任意改写硬盘文件。
   - **第 2 步：备份原图**——把原图复制到 `editor-snapshots/{mediaId}_{时间戳}.{ext}` 子目录，**并校验备份大小一致**（防止磁盘满导致原图损坏）。
   - **第 3 步：用 sharp 库写入新图**——sharp 是 Node.js 的图像处理库，比 Canvas 快很多：
     ```
     sharp(buffer).jpeg({ quality: 95, progressive: true, mozjpeg: true }).toFile(原路径)
     ```
   - **第 4 步：失败回滚**——如果写入失败，再次校验备份大小，没问题就把备份复制回原路径恢复原图。
   - **第 5 步：记录历史**——往 `edit_history` 表插一条记录，存当时的调色参数（JSON 字符串）。

4. **历史记录的 LRU 清理**
   - 编辑快照（原图备份）超过 50 个时，按时间最旧先删（LRU = Least Recently Used，最近最少使用，缓存淘汰算法）。

#### 撤销/重做怎么实现

**重要：项目里有两套独立的历史机制，不要搞混**

- **编辑器内调色撤销**（关闭编辑器就丢）
  - 文件：`src/renderer/hooks/useEditHistory.ts`
  - 最多记 50 步
  - 滑块**释放时**（`onPointerUp`）才入栈，拖动过程不入栈
  - `undo` 取上一个状态回灌到四个 setter
  - **只在内存里，不存数据库**

- **全局文件操作撤销**（跨重启都能撤）
  - 文件：`src/renderer/stores/operationHistoryStore.ts`
  - 数据库表：`operation_history`（启动时自动清理 30 天前的记录）
  - 只管文件级操作（移动/重命名/复制/删除/收藏/评分/分类/标签/笔记）
  - **不管图片调色**——调色保存后不可通过这个栈撤销，但原图备份在 `editor-snapshots` 目录里可以手动恢复

#### LUT 和滤镜的区别

- **滤镜（filter）**：内置预设，每个预设只是「相对默认值的偏移量」（如 `{ brightness: 10, saturation: 15 }`），用 `mergeFilterParams` 按强度比例混合到当前参数。
- **LUT（Look-Up Table，颜色查找表）**：3D 颜色映射表，支持导入 `.cube` 格式文件，做的是「把每个 RGB 颜色查表换成另一个颜色」的高级调色。文件位置：`src/renderer/utils/lut.ts`，5 个内置预设（原色/青橙电影/暖调复古/冷调戏剧/高对比）。

**完整链路时序图**：

```
[用户拖亮度滑块] SliderControl.tsx
      ↓ onChange → updateParam('brightness', value)
[修改状态] useImageProcessor.ts setParams()
      ↓ 状态变化触发 useMemo 重算 mergedParams
      ↓ useEffect 触发 renderPreview (80ms 防抖)
[处理像素] imageProcessor.ts processImageData()
      ↓ canvas.drawImage + getImageData 拿到像素
      ↓ for 循环遍历每个像素应用调色
      ↓ putImageData 画回 canvas
      ↓ imageToDataUrl() 转 dataURL
[显示预览] <img src={previewUrl} />
[用户点保存]
      ↓ exportDataUrl(true) ← 全尺寸重算一次
      ↓ editor.save(filePath, dataUrl, options) ← 发 IPC 信
[后台收信] editor.ts → editorService.save()
      ↓ 安全校验：filePath 必须在 media_files 表里
      ↓ 备份原图到 editor-snapshots/
      ↓ 校验备份大小
      ↓ sharp 转码写入原路径
      ↓ 失败则回滚（从备份恢复）
      ↓ 写 edit_history 表
[返回成功] → 界面刷新媒体列表
```

### 模块 5：重复检测

**作用**：找出图库里完全一样的图（精确重复）和看起来很像的图（相似重复），并智能推荐保留哪一张。

| 项 | 内容 |
|---|---|
| 入口页面 | `src/renderer/pages/DuplicatesPage.tsx` |
| IPC 注册 | `src/main/ipc/handlers/media.ts`（`media:findDuplicates` 第 398 行 / `media:findSimilar` 第 505 行） |
| 完整 hash 计算 | `src/main/utils/file-utils.ts` 的 `calculateFileHash`（第 120 行，sha256 流式读取） |
| 感知哈希计算 | `src/main/utils/phash.ts` 的 `calculatePHash`（第 70 行） |
| 智能评分 | `src/main/utils/duplicate-scoring.ts` 的 `pickBestId`（第 87 行） |
| pHash 批量回填 | `src/main/services/thumbnail-phash-service.ts` 的 `generatePhashForUnprocessed`（第 102 行） |
| 重复标记入库 | `src/main/services/thumbnail-phash-service.ts` 的 `markDuplicates`（第 154 行） |

**两种检测机制**：

**A. 精确重复（exact）**
- 第 1 轮：按 `file_size` 分组，只保留 ≥2 张的组（大小不同肯定不重复）
- 第 2 轮：对每组每张图算完整 sha256（流式读取整个文件，64 字符 hex 串）
- 按 hash 二次分组，长度>1 的就是精确重复组

**B. 相似重复（similar）**
- 数据源：`media_files` 表里 `phash IS NOT NULL` 的记录
- 算法：**Union-Find（并查集）**——两两图片算 Hamming 距离（两个 64 位 hash 不同的位数），距离 ≤ 阈值就合并到同一组
- 阈值档位：2（极严格）/ 5（默认）/ 10 / 15（极宽松）

**pHash 是怎么算出来的**（`phash.ts`）：
1. 用 sharp 把图缩成 32×32 灰度图
2. 做二维 DCT 变换（离散余弦变换，一种信号处理算法，JPEG 也用它）
3. 取左上角 8×8 共 64 个低频系数
4. 算这 64 个数的均值（排除第一个 DC 分量）
5. 每个数与均值比，大于等于得 `'1'`，小于得 `'0'`，得到 64 位 0/1 字符串
6. 这串就是这张图的「指纹」，两张图指纹不同的位数越少越像

**智能评分**（`duplicate-scoring.ts`，满分 100）：
- 分辨率 40 分（越大越好）
- 文件大小 30 分（越大越好）
- 拍摄时间 20 分（越新越好）
- 收藏加权 10 分（收藏过或评分>0 直接满分）
- 算出每组分数最高的就是「推荐保留」的那张

| **输入** | `findDuplicates()` 无参数；`findSimilar({ threshold })` 传阈值 |
| **输出** | `{ duplicates: 分组数组, bestItemIds: 推荐保留的 id 数组, ... }`；同时调用 `markDuplicates` 写入 `media_files.is_duplicate` 和 `original_id` 字段 |

## 附：日常开发怎么用这份文档

### 场景 1：要改某个功能，不知道代码在哪

1. **先看模块清单**（第三部分），5 大模块的代码路径都列了。
2. **再看目录结构**（第一部分），其他功能按 `services/`、`pages/`、`components/` 对号入座。
3. 找到文件后，在 VS Code 里 `Ctrl+点击` 跟着函数调用链深入。

### 场景 2：要新增一个 IPC 通道

参考已有模式：
1. 在 `src/main/preload.ts` 暴露 API（界面调用入口）
2. 在 `src/main/ipc/handlers/<域>.ts` 用 `ipcMain.handle('<域>:<动作>', ...)` 注册处理
3. 在 `src/renderer/` 调用 `window.electronAPI.<域>.<动作>(...)`

### 场景 3：要改某个调色算法

直接去 `src/renderer/utils/imageProcessor.ts`，所有调色算法都在这里：
- 亮度对比度：`processImageData` 主循环里
- HSL：`applyHSL`
- 曲线：`buildCurveMap` + 主循环里查表
- 色调分离：`applySplitTone`
- 清晰度/锐化/降噪：`applyClarity/applySharpen/applyDenoise`

### 场景 4：要改扫描行为

去 `src/main/scanner/index.ts`：
- 找游戏目录的逻辑：`findMediaDirectories`
- 签名文件夹列表：`MEDIA_FOLDER_SIGNATURES`（第 95 行）
- 递归扫描：`scanDirectory`
- 批量写入：`saveBatchToDatabase`

### 场景 5：要改数据库表结构

去 `src/main/database/connection.ts` 的 `runMigrations()`：
- 新表用 `CREATE TABLE IF NOT EXISTS`
- 新字段用 `ALTER TABLE ... ADD COLUMN`（注意要 `IF NOT EXISTS` 风格的兼容写法）
- 新索引在第 264-299 行的索引块统一加

### 场景 6：调试某个功能跑不通

1. **看日志**：日志在 `{userData}/logs/`，格式是 JSONL（每行一个 JSON 对象），可用 VS Code 打开看
2. **看崩溃报告**：`{userData}/crashes/`，保留最近 20 份
3. **看数据库**：用 DB Browser for SQLite 打开 `{userData}/database/wxnn_photo_manager.db` 直接查表
4. **看原图备份**：编辑过的原图在 `{userData}/editor-snapshots/`，最多 50 份

### 场景 7：要写新代码时遵守规范

- 跑 `npm run lint` 检查代码规范
- 跑 `npm run format` 自动格式化
- 跑 `npm run typecheck` 检查类型
- 跑 `npm test` 跑单元测试
- 提交前确保这四个命令都通过
